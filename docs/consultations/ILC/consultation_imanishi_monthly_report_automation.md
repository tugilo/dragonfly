## 相談内容

今西さんより、現在の月次集計表作成について以下の相談があった。

> 現在、このデータを使って集計表を作成して加盟店に毎月送信してます。  
> 弊社システム担当が自分で加工しているのを簡単に処理できないかと…  
> 見せ方などが変わることについては問題ありません。  
> 一度検証してみてもらえますか。

### 困っていること

- 店舗管理画面から出力した TSV を、毎月 Excel に貼り付け・変換・転記している。
- 集計表を加盟店ごとに作成し、毎月送信する作業が担当者依存になっている。
- 既存の見せ方に強い固定要件はなく、見せ方が変わってもよい。

### 現状の運用

確認できた資料から、少なくとも以下の2系統がある。

| 系統 | 入力 | 編集・集計用ファイル | 最終成果物 |
|------|------|----------------------|------------|
| チカラグループ | `チカラ_2026年4月分_TSV/*.tsv` | `分析基本共通_改_QR.xlsm` / `貼り付け用.xlsx` | `先方へ渡す最終のデータです。/該当店舗/*.xlsx`、ユーザー数集計 |
| ファボローネ | `静岡ファボローネ_tsv/*.tsv` | `分析基本共通_詳細集計対応版.xlsm` | `抜き出した数値をこちらのエクセルでまとめます。/*.xlsx` |

作業手順 PDF では、TSV を Excel で開き、明細シートへ貼り付け、マクロまたは更新処理で日別・項目別に変換し、最終 Excel の該当箇所へ転記する流れになっている。

## 初期検証メモ

### 確認したデータ構造

- TSV は `cp932` で読み取り可能。
- TSV の列は共通して `店名`, `日付`, `category`, `action`, `label`, `件数`, `時`, `分`, `秒`。
- チカラグループの TSV は 15 店舗分を確認。
- ファボローネの TSV は 36 店舗分を確認。
- 主な `category` は `501_Splash`, `502_Title`, `503_Subtitle`, `523_QrCodeReader`, `WebEvent`。
- Excel 側は、手作業転記・数式・マクロ・ピボット更新に依存している可能性が高い。

### 自動化できそうなこと

| 対象 | 自動化可能性 | メモ |
|------|--------------|------|
| TSV の読み込み | 高い | CP932 TSV として Python 等で安定して読み取れる。 |
| 日別・店舗別・項目別集計 | 高い | `日付`, `category`, `action`, `label`, `件数` の group by で再現できる可能性が高い。 |
| 欠損日付の補完 | 高い | 月初〜月末、または任意期間の日付を自動生成して 0 埋めできる。 |
| 店舗別 Excel 出力 | 中〜高 | 既存 Excel のレイアウト維持も可能だが、見せ方変更可なら新テンプレの方が安全。 |
| 加盟店向け PDF / Excel 出力 | 中 | 出力形式の希望確認が必要。 |
| メール送信までの自動化 | 中 | 宛先管理・送信前確認・誤送信防止の設計が必要。 |
| 既存マクロ完全再現 | 中〜低 | マクロ内部の仕様確認が必要。完全再現より、集計ロジックを作り直す方が現実的。 |

## 対応案

概算費用は、現時点の手元データ確認に基づくラフ見積。税別・初回構築費の目安であり、最終成果物の形式、対象店舗数、送信まで含めるかで変動する。

**費用感の確認:** フル機能の Web アプリとして継続運用する前提では、TSV 取込だけでなく、文字コード・例外データ・集計条件差・PDF/Excel出力・ログイン・履歴・保守性まで考える必要がある。一方で、今西さんへの初回提示としては高く見えすぎると検討に入りにくい。tugilo の AI駆動開発では、まず **MVP（最小実用版）** として、対象範囲を絞って安く・速く検証し、効果が見えた段階で拡張する提案が現実的。

前提として、**デスクトップアプリではなく Web アプリとして構築する方針**がよい。理由は、担当者のPC環境に依存しにくく、複数人で使いやすく、将来的に送信履歴・宛先管理・権限管理まで広げやすいため。

また、tugilo は **AI駆動開発** を前提にする。一般的なWebアプリ開発より、集計ロジックの試作、TSV解析、画面プロトタイプ、Excel/PDF出力の検証を短期間で進めやすい。一方で、業務要件の確認、現行Excelとの数値照合、例外データ対応、セキュリティ、運用設計は人が責任を持って確認する必要があるため、単純に「AIで安く作る」ではなく、**検証と改善サイクルを速く回して、投資判断をしやすくする** 位置づけにする。

### プラン別の具体イメージ

#### 1. AI検証プラン

- チカラグループ 1 店舗、ファボローネ 1 店舗など代表データを選ぶ。
- AI駆動で TSV の構造を解析し、Web アプリと同じ処理方式で日別の起動数、クーポン、スタンプ、QR 読み取りなど主要指標を試算する。
- 現行 Excel の数値と突き合わせ、どこまで自動化できるか判断する。
- 成果物は「検証結果メモ」と「サンプル集計ファイル」。

#### 2. AI駆動Web MVP

- ブラウザから月次 TSV をアップロードする。
- Web 画面上で対象月・対象グループを選ぶ。
- 全店舗分を読み込んで集計する。
- 日付が抜けている日は 0 として補完する。
- 店舗別・日別・項目別の Excel / CSV を出力する。
- 既存のマクロ Excel への貼り付け作業を大幅に減らす。
- ログイン、送信管理、高度な履歴管理は初期スコープから外し、費用を抑える。

#### 3. AI駆動レポート生成版

- 加盟店向けに「見やすいレポート形式」を新しく設計する。
- AIを使って、表中心・グラフ中心・要約コメント付きなど複数の見せ方を試す。
- Web 画面から対象月を選び、店舗別レポートを一括生成する。
- 店舗ごとに、月次合計、日別推移、主要アクション別件数を出す。
- 必要に応じて Excel と PDF の両方を出力する。
- 「今月の利用状況が一目で分かる」資料に寄せられる。

#### 4. AI駆動Web運用版

- ログイン画面を用意し、担当者だけが使えるようにする。
- 取込履歴、生成済みレポート、過去データの再ダウンロードを管理する。
- 店舗名やグループ、送付対象などをマスタとして管理する。
- 担当者が変わっても、同じWeb画面から運用できる。

#### 5. AI駆動送信管理まで含むWeb運用版

- 店舗ごとの送付先を管理する。
- レポート生成後、送信前に一覧で確認できるようにする。
- メール送信、送信履歴、再送管理まで含める。
- 誤送信防止のため、最初は「自動送信」ではなく「送信用ファイル一式を作る」段階から始めるのが安全。

> 進め方としては、デスクトップアプリではなく Web アプリ前提で考えるのがよいと思います。tugilo はAI駆動開発で進められるため、TSV解析・集計ロジック・画面プロトタイプを短期間で検証しやすく、担当者のPC環境にも依存しません。将来的に履歴管理や送信管理まで広げやすい点もあります。  
>  
> まず代表店舗で現行 Excel と数字が合うかを小さく検証する形なら、10〜20万円程度で確認できます。ここではAIを使って集計ロジックの試作を早く行い、投資判断できる材料を作ります。  
>  
> 数字が合うことを確認できたら、次に「ブラウザからTSVをアップロードすると店舗別の集計 Excel / CSV が出る」Web MVP が 80〜90万円程度、加盟店向けの見やすい Excel / PDF レポートまでWeb上で自動生成する形だと 160〜180万円程度が目安です。  
>  
> 自動化に関しては、対象を「TSVを出した後の加工〜集計表作成」に絞れば、かなり削減できるはずです。今回確認できたデータだけでも、チカラグループ15店舗・ファボローネ36店舗、合計51店舗分のTSVがあり、列構造も揃っています。ここは人がExcelに貼り付けて変換するより、プログラムで一括処理した方が向いています。  
>  
> 経営判断としては、金額だけでなく「毎月どれくらい作業時間が減るか」が重要です。例えば月40時間かかっている作業のうち、TSV取得後の加工が中心であれば、4〜8時間程度まで減らせる可能性があります。月32〜36時間削減、時給換算で月9.6〜18万円程度の削減効果が見込めるため、Web MVP なら数ヶ月〜1年弱で回収できる可能性があります。  
>  
> 見せ方が変わっても問題ないとのことなので、既存 Excel を完全再現するより、必要な指標を整理して見やすいレポートとして作り直す方が、今後の運用は楽になると思います。

## 推奨方針

まずは **プラン1: 代表店舗でWeb集計ロジックの検証** を行う。  
見せ方変更が許容されているため、既存マクロの完全再現ではなく、TSV を正として集計ロジックを作り直す方がよい。

初回検証では次を確認する。

1. チカラグループ 1 店舗分で、現行成果物と主要数値が一致するか。
2. ファボローネ 1 店舗分で、現行成果物と主要数値が一致するか。
3. 加盟店に毎月送る表として最低限必要な項目を洗い出す。
4. 既存 Excel レイアウトを維持する必要が本当にあるか確認する。

## 追加確認が必要なこと

- 加盟店に送っている最終成果物は Excel か PDF か、または両方か。
- 加盟店ごとの送信先、送信方法、送信前チェックの有無。
- 「必ず必要な指標」と「あればよい指標」の区分。
- チカラグループとファボローネで、集計期間の考え方が違う理由と今後も維持するか。
- `label` に含まれる位置情報など、加盟店向けに出してよい情報・出さない情報。
- 現行 Excel の数式・マクロで実施している変換ルールのうち、業務上必須のもの。
