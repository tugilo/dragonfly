# WordPress 会員向け掲示板（投稿型）構築・プラグイン整理 — 要件定義書

**文書の目的**: 提案・モック・見積にそのまま使うための要件定義です。技術的に実装可能な粒度（§2 以降）と、クライアントの判断材料（**§0・§1**）の両方を揃えています。未確定は主に **§7** に集め、断定できない箇所は**断定しません**。

**2本立ての案件**:
- **案件①** 会員向け掲示板（**投稿・案件共有型**。チャット型ではない）
- **案件②** 既存 WordPress のプラグイン整理（**影響確認**を主軸。削除一括を前提にしない）

# 0. 結論（先にここだけ読めばOK）

本書の前提と進め方を、非エンジニアの方にも分かる言葉で整理します。

- **掲示板の実装方針**: WordPress もともとの「**投稿**」機能を使い、**会員向け**の「**投稿・案件共有**」の場として整えます。いわゆる**チャット型**（リアルタイム雑談型）の掲示板ではありません。
- **会員向けの範囲**: **会員**だけが**見る・書く（権限のある人）**・**コメント**できる、という前提です（「誰を会員とみなすか」は既存サイトの仕組みに合わせます。未確定は §7）。
- **カテゴリ**: 追加・名称変更・削除は**管理者**だけが**管理画面**で行い、**会員**は**投稿のときにカテゴリを選ぶだけ**です。
- **プラグイン整理（案件②）**: **一括で消す**のではなく、**一覧・分類・影響の説明**をしたうえで、**止める／保留する／触らない**の候補を出します。本番で**大量に止める・消す**作業は、**別の工数**として扱います（§3-6）。
- **初期の作り方**: まずは**最小の範囲**（§1.3・§5）で動くものにし、足りない部分は**あとから足せる**前提です（§6）。
- **本書の位置づけ**: **モック**（画面案）と**見積**に進む**前**に、**方向性**と**決めのいくつか**を揃えるための整理です。ここに「前提としてOK」と書かれた方針で、次工程に進めます。

**前提（本案件で採用する方針）**

| 項目 | 採用方針 |
|------|----------|
| 掲示板の性質 | **投稿・案件共有型**（スレッド・非同期コメント。リアルタイムチャットは想定しない） |
| 基盤 | **WordPress コアの「投稿（デフォルトの post タイプ）」**＋**標準のカテゴリ・タグ**（タグの使否は §7 または 2-3 P7） |
| 拡張 | **新規プラグインの追加は原則しない**（会員制御等は**既存**の会員用実装に合わせ、必要最小限の**テーマ/子テーマ**内カスタマイズ） |
| 閲覧・投稿・コメント | **会員（ログイン済みかつ会員条件を満たすユーザー）のみ** ※会員の定義は **既存サイトの会員**に合わせる（**§7.1** で先に固める**前提**） |
| カテゴリ | **追加・編集・削除は管理者のみ**。**会員は投稿時の選択のみ**（WordPress 標準: `manage_categories` を持たない会員ロール想定） |
| 案件② | **棚卸し・分類・候補整理・影響の説明**まで。本番の**一括停止・一括削除**は**原則別見積** |

**用語**  
- **管理者**: サイト上で `manage_options` またはカテゴリ管理が可能な**運用担当ロール**（具体ロール名は既存に依存 → §7）  
- **会員**: 当該サイトの会員専用機能を利用するログインユーザー（既存会員機構の定義に従う）

# 1. 全体概要

## 1.1 本案件の目的

- **目的1**: 会員が、**会話ではなく**文章投稿として **コラム・案件・知見** を**蓄積・参照**できる領域を、**WordPress 標準の投稿**で提供する。  
- **目的2**: 多く存在するプラグインの**影響範囲**を**文書化**し、**停止/保留/維持**の**判断材料**を揃え、**不用意に壊す**操作を減らす。

## 1.2 今回の判断ポイント（打ち合わせ・返信用）

クライアント側で**先に「はい／いいえ／条件付き」**が欲しい項目です。メールや会議のチェックリストとして使えます。

| 論点 | 確認したいこと |
|------|----------------|
| 掲示板の方向性 | **投稿・案件共有型**（チャットではない）で進めてよいか |
| 投稿できる人 | **全会員**に投稿を許すか。**一部の役割**に限るか |
| 公開前の承認 | **そのまま公開**でよいか。**管理者の承認**が必要か |
| タグ | **最初の段階で**タグを**使う／使わない**（使う場合のルール含む） |
| プラグイン整理の範囲 | **調査・一覧・候補整理まで**を今回に含め、**本番での停止作業**は**別見積**でよいか |
| 次の一手 | 上記のうち**未決**が残る場合、**モック作成**に進んでよいか（仮置き前提） |

※見積の幅に**直結**するのは、多くの場合 **投稿者の範囲**・**承認の有無**・**タグ**・**過去データの移行**・**プラグイン整理の「実作業」まで含むか** です（§7.1 参照）。

## 1.3 初期構築でできること・今回はここまで（使う側の目線）

技術用語を極力避け、**利用イメージ**で整理します。実装の細目は §2・§5 と対応します。

**初期構築でできること（想定）**

- **会員が**、**会員専用の場所**から、**一覧**で投稿を**眺めたり**、**カテゴリ**で**絞り込んだり**、**キーワード**で**探したり**できる  
- **会員が**（権限のある範囲で）、**新しい投稿**を**書ける**（タイトル・本文・**カテゴリを選ぶ**。タグは §1.2・§7）  
- **投稿を開く**と**本文**と**コメント**が**見え**、**会員**が**コメント**を**書ける**（運用ルールは §7）  
- **管理者**が、**管理画面**だけで**カテゴリ**の**追加・名前の変更・削除**を**行える**。**会員**に**カテゴリの新設**は**させない**  
- **ログインしていない人**は、そのエリアを**見られない**（誘導は既存サイトに合わせる）

**今回はここまでを対象とする（初期）**

- まずは**使える最小**に寄せ、**足りないもの**は **§6（将来）** か**別見積**で足す前提です。  
- **技術寄りの境界**（テンプレート・権限 ID 等）は **§5** を参照してください。

## 1.4 2本立てに分ける理由

| 観点 | 案件① | 案件② |
|------|--------|--------|
| 主成果物 | 画面・**テンプレート**・会員**連携** | **一覧表**・**分類**・**推奨**・**リスク** |
| リスクの種類 | 仕様抜け・導線・権限不整合 | 無効化の**副次障害** |
| 依頼の切り方 | 機能としての**範囲**が合意しやすい | 調査と**本番作業**を**分け**やすい |

同一 WordPress 上のため**依存**はあるが、**契約上のスコープ**を分けないと、片方の変更要望が**他方の工数**を食い潰しやすい。

## 1.5 ゴール

- **G1 モック**: §1.7・§2-7 の内容が**図**として描ける。  
- **G2 見積**: §2-8 / §3-6 の**含む/含まない**で**行単位**の見積が組める。  
- **G3 合意**: クライアントが **初期スコープ（§1.3・§5）** と **今回やらないこと（§1.6）** を判断できる。

## 1.6 今回やらないこと（重要）

**初期の対象外**として、次を**はっきり**外します（打ち合わせで**繰り返し**確認しやすいよう、表の他に**短い箇条書き**も置きます）。

**特に今回の対象外として明確にしないもの**

- **チャット型**の掲示板（リアルタイムの会話が主目的のもの）  
- **大規模**な過去データの**一括**自動**移行**（手動・少数件は§5・§7で合意可能）  
- **多段階承認**・**複数承認者**が絡む**稟議**のようなフロー  
- **会員システム（会員プラグイン等）の全面**乗り換え・**大改修**  
- 案件②における、本番での**大量**プラグイン**停止・削除**（**まとめて**の実施は**原則**対象外／**別工数**）

| 範囲外（初期） | 補足 |
|----------------|------|
| **チャット型**の掲示板（**同期・ルーム**等） | 投稿**詳細＋コメント**の**非同期**で足りる想定。変更する場合は **別仕様** |
| **新規の**掲示板専用**大型プラグイン**の導入 | 原則 **行わない**（例外はクライアント合意＋**別見積**） |
| **会員制御プラグインの**乗り換え・大改修 | 案件①・②**いずれの初期**でも**行わない**（必要なら **別案件**） |
| **多段階稟議・複数承認者ルート** | 要望が出たら **将来（§6）** または **別見積**（§2-3 承認） |
| **大規模な過去投稿の一括**自動**移行** | **手動**または **件数上限付き**は §5 で**別途合意**可能 |
| 案件②の **本番一括**無効化・**一括**削除 | **別工数**（§3-6） |
| **一般公開向け**の**記事**と**会員掲示板**の**完全な情報設計**の**一本化** | カテゴリ**の分離**要否は §7。初期は**最小**で**切り**やすい |

## 1.7 画面イメージの概要（図がなくてもイメージできる説明）

手書きやワイヤーの**たたき台**と**同じ前提**で、画面の**流れ**を文章で示します（要素の**詳細表**は §2-4）。

**一覧（掲示板の「表紙」）**

- **左**に**カテゴリ一覧**（「すべて」＋各カテゴリ。**件数**が付くと分かりやすい）  
- **右上あたり**に**検索バー**（キーワードで投稿を探す）  
- **中央の広いエリア**に、**投稿のカード（または行）**が**新しい順**などで並ぶ。各カードに**タイトル**・**日付**・**抜粋**・**カテゴリ**・**投稿者**などが**載る**イメージ  
- **投稿をクリック**すると**詳細**へ

**詳細**

- **タイトル・日付・投稿者・カテゴリ・本文**が読める  
- **下**に**コメント**が**並び**、**会員**が**コメントを書き込める**（ルールは §7）

**会員向けの投稿フォーム**

- **タイトル**と**本文**、**カテゴリを選ぶ**（**新規カテゴリの作成欄は出さない**）  
- **タグ**を使うかは §1.2・§7。**承認**が必要な運用の場合は、**下書き**や**公開待ち**の**出し分け**が**出る**（§2-3）

**管理者向けのカテゴリ管理**

- 会員向けの画面ではなく、**管理画面**（いわゆる`wp-admin`）の**カテゴリ管理**で**十分**、という前提（専用画面の**新規開発**は**初期**では**必須にしない**）

**スマートフォン**

- 左のカテゴリは、**上に折りたたみ**にする**など**、**狭い幅**で**使いやすい**出し方に**する**（**モック**で**形**を**決める**）

# 2. 案件①：会員向け掲示板（投稿型）の構築

**補足**: 画面の**使う側**の文章イメージは **§1.7**。**実装用の**要素一覧は **§2-4 以降**。

## 2-1. 目的

- **事業/運用目的**: 会員が、**文章ベース**で **案件・コラム・知見** を**投稿**し、**分類**と**検索**で**あとから辿る**（**知の蓄積**が主目的。チャットの代替ではない）。  
- **システム目的**: **WordPress 標準の「投稿（post）」一種**に**集約**し、**テーマ側**で会員専用アーカイブ・**権限**を**実装**する。  
- **測定可能な到達点（初期）**: 会員が **一覧→詳細→（任意で）コメント**、**権限付き**会員は **投稿作成**、**管理者**は **wp-admin 上のカテゴリ**管理。

## 2-2. 利用者

### 管理者

| 行為 | 仕様上の扱い |
|------|--------------|
| カテゴリの **新規/編集/削除** | **必須**。WordPress 管理画面 **投稿 > カテゴリー**（または**同等UI**）で実施 |
| 投稿の**編集/削除/公開** | **要否は運用**。**トラブル時**の**全権**は**管理者**想定。通常は**本人編集**を会員に付与するか **§7 高** |
| 会員向け**投稿**の**承認** | **初版では「なし」または「下書き＋管理公開」**のいずれかに**絞る**想定。別フローは **§7 高** |

### 会員

| 行為 | 仕様上の扱い |
|------|--------------|
| **閲覧** 一覧・詳細 | **会員**のみ（非会員は **§2-3 権限**） |
| **投稿** 新規 | **会員**に付与する `publish_posts` 等の有無は **§7 高**。**初期**は**全会員** or **役割制限**のいずれか**明示**必須 |
| **コメント** | **有効**を初期想定。会員**のみ** or **全員**は **§7 中**（一般は**会員限定**推奨） |
| カテゴリ | **選択のみ**（`edit_posts` のみで **カテゴリ作成**は**不可**にする**実装**が初期方針） |
| タグ | **初期は「非表示」or「会員付与可」**の**どちらか**を §7/モックで**確定**（§2-3 参照） |

## 2-3. 機能一覧（実装粒度）

以下は**実装時にコード・設定で検証可能**な**単位**に分解したもの。

### 投稿機能

| ID | 要件 | 実装/検証の目安 |
|----|------|-----------------|
| P1 | **投稿作成** | 会員: **フロント**の投稿フォーム（`wp-admin` 直**回避**推奨）**または** `wp-admin` 投稿 screen の**制限**のいずれかを **クライアント合意**（§7 中） |
| P2 | **投稿編集** | **投稿者本人（スレッド主）のみ**を初期推奨。**他人の投稿の編集**は**管理者**のみ。**実装**は `user_can( 'edit_post', $id )` 等で**検証** |
| P3 | **投稿削除** | **投稿者本人（スレッド主）のみ削除可**とするか、**削除は管理者のみ**とするかを要合意。WordPress の投稿削除権限＋画面制御で実装可能。初期**最小**は**「削除は管理者のみ」** or **下書き化**（§5） |
| P4 | **投稿項目: タイトル** | **必須**、最大長は **WordPress デフォルト**＋**フロント**の**バリデーション**（表示**エラーメッセージ**） |
| P5 | **本文** | **必須**、**可視**エディタ。文字数**上限**は**未設定** or **要合意**（§7 低） |
| P6 | **カテゴリ** | **1つ以上**必須 or **任意**を合意。初期**推奨: 1つ必須**（未分類を**避ける**） |
| P7 | **タグ** | **初期A**: 画面に**出さない**（標準**タクソノミ**は**未使用**）。**初期B**: **表示**し**会員**が**選択**可→**新規自由タグ**は**禁止**し**候補のみ**にする、等を**要合意**（§7 中） |

### 表示機能

| ID | 要件 | 実装/検証の目安 |
|----|------|-----------------|
| V1 | **投稿一覧** | `archive` / カスタムクエリ。**新着**。**ページネーション**（件数/ページ **要合意、初期例: 10**） |
| V2 | **投稿詳細** | `single.php` 等。**日付/投稿者/カテゴリ/本文/コメント**（§2-3 コメント） |
| V3 | **一覧 左カラム** | **カテゴリ**一覧。選択中**ハイライト**（§2-4 一覧） |
| V4 | **0件** | カテゴリ・検索**結果**それぞれ**0件**メッセージ |

### コメント機能

| ID | 要件 | 実装/検証の目安 |
|----|------|-----------------|
| C1 | **コメント表示** | 標準**コメントループ**。**会員**のみ**閲覧**可能なテンプレート**内**（非会員**アクセス**は**ブロック**） |
| C2 | **コメント投稿** | ログイン**会員**。**匿名**は**不許可**（**初期**）。**承認**は WordPress 設定**または** 会員**のみ**即時 等、**要合意**（§7 中） |
| C3 | **スパム** | 既存の Akismet 等があれば利用。なければ reCAPTCHA 等の新規導入は**加算**（§2-8） |

### 通知機能

| ID | 要件 | 実装/検証の目安 |
|----|------|-----------------|
| N1 | **新規投稿の通知** | 新しい投稿が立ったとき、会員へ何らかの通知を送る要望。**メール通知**を第一候補として検討。 |
| N2 | **通知対象** | 全会員へ送るか、一部会員のみ送るかは要合意。初期は**全会員メール通知**が分かりやすい。 |
| N3 | **標準機能での可否** | WordPress **標準だけでは「新規投稿を全会員へ自動通知」は弱い**。実現する場合は**カスタム実装**または通知用プラグインが必要になりやすい。 |

### 検索・絞り込み

| ID | 要件 | 実装/検証の目安 |
|----|------|-----------------|
| S1 | **キーワード検索** | 検索フォーム。**`pre_get_posts`** で**投稿**に**限定**。**対象**: **タイトル+本文+抜粋**を**初期推奨**（**カスタムフィールド**は**将来**） |
| S2 | **カテゴリ絞り込み** | **カテゴリアーカイブ**リンク または クエリ `cat=`。**現在のカテゴリ**表示 |
| S3 | **タグ絞り込み** | **P7 で「タグ使用」の場合のみ**。**タグアーカイブ** |

### カテゴリ管理

| 要件 | 実装/検証の目安 |
|------|-----------------|
| 管理者のみ**追加/編集/削除** | `manage_categories` **を持たない**会員。管理者は**wp-admin** |
| 会員は**投稿時に選択**のみ | フォームは **`<select>`** 等。**新規**カテゴリ**入力欄は出さない** |

### 権限制御

| ID | 要件 | 実装/検証の目安 |
|----|------|-----------------|
| A1 | **会員限定閲覧** | **未ログイン** **または** 会員**でない**場合: **会員専用**テンプレート**へ**誘導、**404** または**ログイン**（**既存**の**会員判定**に**合わせる**。§7 高） |
| A2 | **会員限定投稿** | `current_user_can( 'publish_posts' )` **等**。**会員ロール**へ**一時**付与するか、**会員専用ロール**かは **§7 高**（既存**と**整合） |
| A3 | **非ログイン** | 掲示板**URL**に**直アクセス**→ **ログイン** or **会員**登録導線（**文言**は**既存**） |

**承認フロー（任意だが**見積**に**影響**）**

| 分岐 | 内容 | 含める/含めない |
|------|------|------------------|
| **なし** | 投稿＝**即**公開 | **案件① 初期**に**含めやすい** |
| **下書き + 管理公開** | 会員は `draft`、管理者が `publish` | **+工数**（**一覧/管理**導線） |
| **多段**承認 | **本案件 初期 対象外** | §6 / 別**見積** |

## 2-5. WordPress での実現方法

| 区分 | 方法 |
|------|------|
| **投稿** | コアの `post`。会員専用アーカイブは `pre_get_posts` とテンプレート階層（`archive` / `home` 等との衝突に注意）または固定ページ配下の仮想 URL（**URL 設計は実装前に確定**）。 |
| **カテゴリ** | 標準の `category` タクソノミ。会員掲示板専用の**子カテゴリ**で一般ブログと分離する案は初期で取りやすい（要否は §7）。 |
| **タグ** | 標準の `post_tag`。使わないならフロントフォームに出さない。 |
| **コメント** | コア。テーマで `comments_template()`。`comments_open` と会員条件の併用 |
| **権限** | ロール＋`map_meta_cap` 等で、会員に限定的に `edit_posts` / `publish_posts` を付与。既存会員プラグインとの衝突有無は §7 高。 |
| **会員**限定 | 会員専用の子固定ページ配下、または同ドメイン上で条件分岐。判定は既存の会員フラグ・ロールに合わせる（§7）。 |
| **検索** | コアの検索。`pre_get_posts` で `post_type=post` および会員掲示板用のカテゴリ等に制限。 |
| **投稿者本人の編集・削除** | **実現可能**。投稿者に `edit_post` / `delete_post` を許可し、他人の投稿には権限を持たせない形にする。フロント画面側のボタン表示制御が必要。 |
| **新規投稿の会員通知** | **WordPress標準だけでは弱い**。管理者向け通知やコメント通知はあるが、**新規投稿を全会員へ自動メール送信**する標準機能は前提にしづらい。実現する場合は**カスタムメール送信**または通知用プラグインが必要。 |

**カスタマイズが必要な箇所（初期で想定）**

- 会員専用テンプレート（一覧・詳細・投稿フォーム）  
- 権限（`map_meta_cap` 等、会員ロールへ `publish_posts` / `edit_posts` をどう付与するか）  
- 検索条件の制限（会員掲示板用の投稿に狭める等）  
- 会員専用導線（メニュー・URL）  
- 投稿者本人にだけ**編集/削除**を見せる画面制御  
- 新規投稿時の**会員通知**（行う場合）  

## 2-6. 制約・注意点

| 区分 | 内容 |
|------|------|
| WordPress 標準で足りる範囲 | 投稿（post）・カテゴリ（category）・（任意）タグ（post_tag）・コメント・コアの検索の骨格。管理画面のカテゴリ管理。投稿者本人の編集・削除も、権限設計を行えば実現可能。 |
| テーマ/コード必須の部分 | 会員以外の閲覧停止、会員専用レイアウト、フロント投稿フォーム、検索結果のスコープ絞り込み、投稿者本人だけに編集/削除を出す制御。 |

| 将来の限界 | 多段承認、会員グループ別の閲覧、大量の貼付ファイル、大規模な外部全文検索、通知条件の細分化は、標準投稿＋子テーマの範囲を超えやすい。別フェーズまたは別方式（§6）。 |

# 5. 初期構築スコープ（重要）

**非エンジニア向けの言い方**は **§1.3** を先に。ここでは、**実装・見積**で使う**定義**を**表**にまとめます。

**「使える」最小定義（初期）**  
会員が、導線上で**一覧 → 詳細 →（設定通り）コメント**を行える。会員の一部（または合意上の全員）が**新規投稿**できる。管理者が `wp-admin` の投稿カテゴリで**カテゴリを管理**できる。

| 含む | 具体 |
|------|------|
| 投稿一覧、詳細、カテゴリ絞り込み、キーワード検索（S1 の要領） | 子テーマ等でのテンプレート実装 |
| 会員限定閲覧、会員限定投稿（§2-2 および §7 の確定） | 権限とテンプレート |
| コメント表示・投稿 | C1〜C2。コメント承認の要否は合意（§7） |
| 会員向け新規投稿画面 | 1 案。フロントを優先するかは §7 中 |
| 管理者のカテゴリ管理 | `wp-admin` のみ（専用画面の新規開発は不要） |
| タグ | 初期は**使わない**（画面に出さない）を推奨可能（§7） |

| 今回の初期で行わない | §1.4 参照。大規模移行、会員制御の乗り換え、掲示板専用大型プラグイン、本番でのプラグイン大量停止。 |

- **新規投稿時の通知**を付けるか（特にメール通知）  
- **タグ**利用要否（**初期**で**出さない**のが**分かりやすい**／使う**なら**ルール）
