# 大涌谷黒たまご館 WordPress サイト：天気情報表示機能 要件整理

**文書の位置づけ**  
本稿は、WordPress 管理サイト「大涌谷黒たまご館」の「本日の大涌谷」ページ向けに、日本気象協会 Weather Data API を用いた天気表示を追加する案件について、要望・技術方針・未確定事項を整理したものです。提案資料および実装スコープ判断のたたき台として利用できます。

**前提（共有情報の整理）**

- 対象 CMS は **WordPress**
- 対象ページは **「本日の大涌谷」**
- 天気データは **日本気象協会 Weather Data API** を利用する
- 既存の汎用天気プラグインで本 API に特化したものは**ほぼ存在しない**前提

---

## 1. 案件概要

**何をする案件か**  
「本日の大涌谷」ページ上で、**気温・天気**などを**シンプルなカード型**の UI に表示し、**約 1 時間ごと**に表示内容（気温・天気マーク等）を更新する機能を、WordPress 上に実装する。

**対象ページ**  
- **本日の大涌谷**（当該ページ専用、または同ページ内のブロック・ショートコード等で埋め込む想定）

**利用する API**  
- **日本気象協会 Weather Data API**（気温・天気状況の取得。詳細は契約内容・提供エンドポイントに依存）

**WordPress での実装前提**  
- 実装の中心は **PHP によるサーバー側**での API 呼び出しと、取得結果の**キャッシュ・表示**。
- `functions.php` への一括直書きではなく、**小さな専用プラグイン**として切り出す方針を望ましい（保守・更新・有効/無効の切り分けのため）。

---

## 2. 先方要望

**表示したい内容**  
- **気温**、**天気**に関する情報（具体的なラベルや項目数は、API 仕様と合わせて確定予定）
- 共有いただいたイメージに近い、**分かりやすいカード型**のレイアウト

**更新頻度**  
- 表示上は **1 時間程度の粒度**で、気温・天気マーク等が**更新されたように見える**こと（＝毎回ページ表示のたびに API を叩き続ける、という挙動は望まない。詳細は「4. データ取得・更新方針」参照）

**見た目のイメージ**  
- **シンプルなカード型**。過度に装飾せず、テキスト・日付・天気アイコン・気温が一目で分かる構成

**公開サイトとしての要件**  
- **API キーはクライアント（ブラウザ）に出さない**こと
- ページを開くたびに API を多発呼び出ししない**キャッシュ**により、**API 利用回数を抑える**こと
- **API 取得失敗時**も、**極力レイアウトが崩れない／真っ白にならない**ような**フォールバック**を考慮すること

---

## 3. 実装方針

**PHP サーバー側で API を取得する理由**  
- ブラウザから直接 API を呼ぶ方式では、**API キーがクライアントに露出**するか、**プロキシ的な中継**が必要になる。本案件では、**WordPress 動作の PHP サーバー上**でリクエストを行い、**キーはサーバー側の設定のみ**に留める形が、公開サイトの一般的な安全策に合致する

**WordPress への組み込み**  
- 専用プラグイン内で、API 呼び出し、キャッシュ保存、**ショートコード**または**テンプレート用関数**で HTML を出力、といった**一連の流れ**をまとめる

**独自プラグイン化を推奨する理由**  
- テーマ更新で `functions.php` の差分が上書きされにくい  
- 機能の**単位で有効/無効**ができる  
- 後から「天気だけ別サーバー」などに分けにくい巨大な一ファイルより、**責務が明確**になる

**テンプレート呼び出しの想定**  
- テーマの「本日の大涌谷」用テンプレート、ブロック、または固定ページの本文中に、**ショートコード**（例：`[owakudani_weather]` のような仮名）を配置する、または**テンプレートから専用関数**を 1 行呼び出す、といった形を想定（最終的な挿入方法はテーマ構成と合わせて確定する）

---

## 4. データ取得・更新方針

**API を毎回叩かない理由**  
- 閲覧者の**ページビュー数**に比例して API 呼び出しが増え、**利用料金・レート制限**の面で不利になる。公開サイトでは、**一定間隔のバックエンド取得＋保存**の方が予測しやすい

**キャッシュ制御の考え方**  
- WordPress の **transients、オプション、カスタムテーブル、ファイルキャッシュ**等のいずれか、または併用で、**最終取得時刻**と**表示用データ**を保持する
- フロント表示時は、**原則としてキャッシュ済みデータ**を返し、**期限切れ時のみ**（または**WP-Cron 等**でバックグラウンドで）**再取得**する設計を想定

**1 時間ごと更新という要望との整合**  
- キャッシュ有効期間（TTL）を**約 60 分**に設定し、**再取得のトリガ**を「**期限切れ＋次の表示リクエスト**」または「**1 時間周期のスケジュール**」のいずれか（または併用）で満たす
- 閲覧が少ない時間帯は、実更新が少し遅れても**最大 1 時間程度**の遅延は許容範囲、という整理が取りやすい

**API 取得失敗時のフォールバック方針**  
- **最後に成功したデータ**を**一定期間**そのまま表示（「最終取得：○日○時」等の**注記**はオプション）  
- 初回起動で一度も成功していない場合は、**プレースホルダ**（「現在取得中」「ただいま表示できません」＋**カード枠の維持**）で、**高さゼロ・真っ白**を避ける

---

## 5. 表示仕様（現時点案）

**表示項目**  
- 日付または「本日」に相当する**日付表現**  
- **天気**（API が返す区分に応じたラベル）  
- **気温**（単位表記含む）  
- 上記に加え、**補足の一言**（例：大涌谷に関する注意や季節メッセージ）が必要かは**要確認**

**時刻表示**  
- データの**基準時刻**（何時点の予報/実況か）を、必要に応じて**小さく併記**する案（表示する/しないは確定前）

**天気アイコン表示**  
- API または**マッピングテーブル**で、**API の天気コード**から**用意したアイコン画像**または**Web フォント／SVG**に対応づけて表示

**気温表示**  
- 整数または小数、**摂氏**、必要なら**体感**や**風**は**スコープ外** unless 先方要望

**補足文言の扱い**  
- 管理画面の**固定フィールド**や**ACF 等**と連携するか、**プラグイン内の固定文言**かは、**更新頻度と運用**で決定

**デザイン**  
- **共有イメージに近い、シンプルなカード型**（枠・余白・アイコンと数値のバランスを優先。最終的な**マージン・色**はテーマに合わせる調整が発生しうる）

## 6. 技術上の考慮事項

**API キー秘匿**  
- **wp-config.php 定数**、**環境変数**、または**WordPress 管理画面の専用設定**（**権限付きユーザーのみ**）に保存し、**リポジトリにコミットしない**

**キャッシュ**  
- **TTL 約 1 時間**、**再取得失敗**時の**前回成功データの継続**、可能なら**再試行回数/間隔**の**バックオフ**を整理

**フォールバック**  
- 上記 4 章の通り。スタイルは**必ず**カード枠＋**最低限のメッセージ**が出る**最小 HTML**にする

**WordPress テーマ依存を減らす構成**  
- マークアップとクラス名は**プラグイン側で BEM 風**等に**名前空間**し、**テーマの CSS 変数**に合わせる**フック**や**上書可能なテンプレート**を**オプション**で用意すると、**テーマ変更**に強い

**将来的な保守性**  
- **バージョン管理されたプラグイン**、**短い README**（設定項目・API 制限の注意）を同梱し、**他ページへの再利用**（ショートコード引数等）を**拡張しやすい**構造にする

---

## 7. 見積もり前提

**簡易版と標準版の違い（イメージ）**  
- **簡易版**：**最小表示**、キャッシュは**素朴な**実装、**失敗時**は**テキストのみ**のシンプルなフォールバック、**デザイン**は**テーマに極力任せ**る  
- **標準版**：**カード型の安定レイアウト**、**ワンタイムのデザイン調整**、**失敗時の前回表示**、**管理画面でキー/一部文言**、**更新間隔の微調整**、**程度の**運用しやすさ

**今回の要望からみた妥当性**  
- **イメージに沿ったカード**、**約 1 時間更新**、**安定表示・キー非公開・キャッシュ**を満たすには、**簡易版より標準版**に近いスコープが**中心**になりやすい

**概算金額感**  
- **簡易版**：**約 10 万円前後**  
- **標準版**：**約 12 万円～15 万円前後**  
- 本案件の**共有イメージ・1 時間更新・安定表示**を重視する場合、**標準版**を**想定主軸**とする

**API 利用料**  
- **上記金額に API 料金は含まれない**想定。正式な従量・契約内容は**日本気象協会（または正規の契約窓口）**への確認が必要

## 8. 未確定事項・確認事項

| 区分 | 内容 |
|------|------|
| 表示項目 | 気温・天気のほか、**風**・**降水確率**・**注意報**等を**出すか**（API 契約と合わせて確定） |
| デザイン反映範囲 | **共有イメージ**との**差分**（色・角丸・フォント）を、**何まで含めるか**（テーマ既存アセット流用の可否含む） |
| 更新間隔 | **厳密に 60 分**か、**分単位の許容**、**営業時間外**の扱い |
| API 契約 | **プラン**、**コール数上限**、**対象地点**（大涌谷周辺の**緯度経度/地点 ID**） |
| 本番反映 | **FTP**、**Git デプロイ**、**マネージド WP** 等、**当サイトの運用**に合わせた**手順**と**担当** |

以上を踏まえ、**本ドキュメントをベース**に、**次回は表示項目**と**API 契約条件**、**挿入位置（ショートコード vs テンプレート）**の**3 点**を先に**確定**すると、実装工数のブレを抑えやすい。
