# DragonFly SPA 要件書 v1（SSOT）

**目的:** BNI DragonFly 機能を「Laravel API + React(ReactAdmin) + MUI の SPA」で構築するための単一ソースオブトゥルース（SSOT）要件書。  
**制約:** 本ドキュメントは要件のみを固定する。既存実装の良否評価は行わず、実装方針は次フェーズで決定する。  
**作成日:** 2026-03-03

---

## 0. 概要

### 0.1 目的

BNI 定例会において「その場で記録し、後で振り返る」ことを可能にする。  
ブレイクアウトの同室者・メモ・会全体での興味・1on1 希望などを記録し、振り返り時に一覧できるようにする。

### 0.2 対象ユーザー

- **自分＝記録者** を主たるユーザーとする。  
- 記録者は「ある定例会に参加した参加者（participant）」のうちの一人であり、その人が同室者を選び、メモ・フラグを付ける。

### 0.3 対象会議

- **meeting 単位** で運用する。  
- まずは **meeting 199** で検証する想定。

### 0.4 本方式

- **Laravel API**（既存・拡張）をバックエンドとする。  
- **React（ReactAdmin）+ MUI** の **SPA** をフロントエンドとして構築する。  
- React / Vite 等のフロント基盤は、本要件時点では未導入の想定。

## 1. ユースケース

以下を最低限含む。各ユースケースは箇条書きと成功条件で記述する。

### UC-01: 会議の参加者一覧を即表示

- 指定した meeting（例: 199）の参加者一覧を、画面表示時に即取得して表示する。
- **成功条件:** meeting number を指定すると、その回の participants（＋ member 情報）が一覧で表示され、検索・タップの対象になる。

### UC-02: Session1 の同室者を“タップで追加/削除”して保存

- ブレイクアウト Session1 で同室だった人を、**プルダウンではなくタップ**で「選抜ボード」に追加・削除し、保存する。
- **成功条件:** 参加者一覧からタップで同室者を追加し、チップ等で削除できる。保存後、Session1 の同室者として永続化される。

### UC-03: Session2 の同室者を“タップで追加/削除”して保存

- ブレイクアウト Session2 で同室だった人を、同様にタップで追加/削除して保存する。
- **成功条件:** Session1 と独立して Session2 の同室者を管理でき、保存後永続化される。

### UC-04: 同室者を選ぶ→即メモ（自動保存 or ワンタップ保存）

- 同室者（または参加者）を選択したら、その人に対するメモを **即入力** できる。自動保存（debounce）またはワンタップ保存で反映する。
- **成功条件:** 画面遷移なしでメモを入力でき、保存状態（保存中/保存済み）が分かる。プルダウン操作に依存せず、タップ＆即メモが可能であること。

### UC-05: 過去に同室だった履歴（回数/最終回/過去メモ）を表示

- ある人物（相手）を指定したとき、**過去の meeting で同室だった履歴**（同室回数、最終同室回、過去のメモ）を表示できる。
- **成功条件:** 相手の participant（または member）を指定すると、同室回数・最終回・直近のメモ（例: 最新3件）などが一覧できる。

### UC-06: 会全体で「気になる」「1on1したい」などを記録

- 定例会全体を通じて、特定の人物に対して「気になる」「1on1したい」などの **フラグ** を記録できる。
- **成功条件:** 会議中または振り返り時に、人物ごとにフラグを付与・解除でき、その状態が保存される。

### UC-07: 後日、振り返り（Session別同室者＋メモ＋フラグ）を一覧できる

- 後日、ある meeting について「Session1 の同室者」「Session2 の同室者」「各同室者へのメモ」「気になる/1on1 フラグ」を一覧で振り返れる。
- **成功条件:** meeting を指定すると、Session 別の同室者リスト・メモ・フラグが一画面（またはタブ/ペイン）で確認できる。

## 2. UI要件（スピード最優先）

BNI の進行テンポに合わせ、以下を **必須要件** として明文化する。

### 2.1 選択方式

- **プルダウンは使用しない。** メンバー選択は **「検索 → タップ」** で追加できること。
- 検索は氏名・カテゴリー等で絞り込み可能とし、候補をタップで即「選抜」に追加する。

### 2.2 画面構成

- **画面遷移を最小限** とする。**1 画面で完結** し、モーダルは必要最小限に留める。
- **3 ペイン構成（推奨）**
  - **左ペイン:** 検索 ＋ メンバー一覧（タップで追加）
  - **中央ペイン:** 選抜ボード（Session1 / Session2 / All 等のタブ、チップで削除）
  - **右ペイン:** 人物カード（メモ即入力、履歴、フラグ表示・編集）

### 2.3 キーボード操作（任意だが推奨）

- 検索入力にフォーカスを維持し、続けて操作できること。
- **Enter** で候補選択／追加ができるとよい（実装可否は次フェーズ）。

### 2.4 保存 UX

- **自動保存（debounce）** または **Ctrl+Enter 保存** のいずれか（または両方）をサポートする。
- **保存状態の表示** が必須: 「保存中」「保存済み」をユーザーが判別できること。

## 3. データ要件

### 3.1 既存データ（現状）

要件として参照する既存の概念・テーブルは以下の通り。実装の有無や詳細は問わない。

- **meetings** … 定例会（回数・日付など）
- **members** … 人物マスター（氏名・カテゴリー・区分など）
- **participants** … その回の参加者（meeting × member）
- **breakout_rooms** … その回のブレイクアウトルーム（Session は room_label 等で区別する想定）
- **participant_breakout** … 参加者とルームの紐付け（同室者情報）
- **breakout_memos** … 同室者へのメモ（meeting + 自分 participant + 相手 participant）

### 3.2 追加が必要な概念（候補）

以下の概念は **要件として必要** とする。テーブル追加・カラム設計は次フェーズで決定する。

- **会全体の「興味／1on1」フラグ**  
  - meeting に紐づくか、相手（member または participant）に対して恒久的に持つかは未決（未決事項に記載）。
- **相手に対する評価・次アクション**  
  - 例: 1on1 予定、紹介したい、など。  
  - フラグのみで足りるか、自由テキストや種別が必要かは次フェーズで検討。

## 4. API要件（SPA向け）

既存 API を踏まえつつ、SPA で必要な I/O を要件として定義する。URL は仮でよい。

### 4.1 参加者取得（meeting number）

- **入力:** meeting number（例: 199）
- **出力:** その回の参加者一覧（participant_id, member 情報など）。検索・タップの元データ。
- **エラー:** 404（meeting が存在しない）、422（number 不正など）

### 4.2 Session 別 同室者の取得／保存（session = 1 or 2）

- **取得**
  - **入力:** meeting number, participant_id（自分）, session（1 または 2）
  - **出力:** その Session の同室者一覧（participant_id, member 情報、既存メモなど）
  - **エラー:** 404（meeting / participant なし）、422（session 不正）
- **保存**
  - **入力:** meeting number, participant_id（自分）, session, 同室者 ID リスト（およびルーム名等、仕様による）
  - **出力:** 保存結果（成功・更新後の同室者一覧など）
  - **エラー:** 404, 422（バリデーションエラー）

### 4.3 メモの upsert（meeting + 自分 participant + 相手 participant）

- **入力:** meeting number, participant_id（自分）, target_participant_id（相手）, 本文（body）, 任意で breakout_room_id や session
- **出力:** 保存されたメモ（id, body, updated_at など）
- **エラー:** 404（meeting / participant なし）, 422（バリデーション）

### 4.4 履歴取得（相手に対して過去 meeting での同室／メモ）

- **入力:** participant_id（自分）または member_id（自分）、相手の participant_id または member_id（仕様による）
- **出力:** 過去の同室回数、最終同室 meeting、過去メモ一覧（例: 最新3件）
- **エラー:** 404（対象が存在しない）, 422（パラメータ不正）

### 4.5 フラグ取得／保存（気になる／1on1 など）

- **取得**
  - **入力:** meeting number（任意）, participant_id（自分）
  - **出力:** 人物ごとのフラグ一覧（例: 気になる, 1on1したい）
  - **エラー:** 404, 422
- **保存**
  - **入力:** meeting number（任意）, participant_id（自分）, 対象（相手）, フラグ種別・ON/OFF
  - **出力:** 更新後のフラグ状態
  - **エラー:** 404, 422

## 5. 非機能要件

- **速度:** 「10 秒で 5 人追加＋メモ」ができることを目標とする。  
  - プルダウンではなくタップ＆即メモで、この目標を満たし得る UI であること。
- **スマホ対応:** スマホでも操作できること。タップ領域が十分で、縦スクロールで一覧が扱えること。
- **冪等性:** 保存 API は何度呼んでもデータが壊れないこと（冪等に設計可能なものは冪等とする）。
- **ログ／監査:** updated_at が追えること。誰がいつ更新したかの履歴は将来拡張とする。

## 6. スコープ

### 6.1 今回（MVP SPA）

- meeting 指定で運用（まずは 199 で検証）。
- Session1 / Session2 の同室者管理。
- メモの入力・保存・表示。
- フラグ（気になる／1on1）の記録・表示。
- 履歴は最小限: 同室回数 ＋ 最新メモ 3 件程度を想定。

### 6.2 将来（本版のスコープ外）

- 認証／権限（ユーザー化、自分＝記録者の識別をログインで行う）。
- meeting 自動作成、参加者インポートの自動化。
- 共有（紹介者に渡す等）。
- タスク化（1on1 リマインド等）。

## 7. 未決事項（質問リスト）

以下の点は本要件書では決定せず、未決事項として列挙する。

1. **自分（記録者）の識別方法**  
   - 暫定: participant_id の手動選択でよいか。  
   - 将来: ログインして自分を特定するか。

2. **フラグのスコープ**  
   - 「気になる」「1on1」は **meeting 単位** か、**相手（member）に対して恒久** か。  
   - 会議ごとにつけるか、人物につけるか。

3. **履歴の定義**  
   - 「同室だった」は **participant_breakout の記録のみ** を指すか。  
   - 会全体での接点（名刺交換のみ等）も履歴に含めるか。

4. **ブレイクアウトの回数**  
   - 現在は Session1 / Session2 の 2 回を想定。  
   - 3 回以上に増える可能性があるか。ある場合、API・UI の拡張方針。
