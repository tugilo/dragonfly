## 2. 現在の業務の良いところ

現在の Excel 群は、これまでの業務を支えるために **非常によく作り込まれた仕組み** です【事実・評価】。単なる表計算ではなく、実際には次のような **業務システム** として機能しています【事実】。

| 良い点 | 説明 |
|--------|------|
| **Web 注文の入口がある** | 顧客向け **[注文フォーム](https://workerbros.jp/order.php)** があり、受注の入口がデジタル化されている【事実：URL・ドキュメント記載】|
| **見積〜帳票まで一連のマクロ** | `見積書作成システム_ver4.1.0.xlsm` で、入力・テンプレ転記・**見積／請求／納品／領収** の PDF 出力まで実装されている【事実】|
| **売上・請求の自動集計** | `売上台帳作成ツール`・`月次請求書作成ツール` が **フォルダ内の見積ブックを走査**し、台帳化・月次請求（Excel＋PDF）まで行う【事実】|
| **ボディーマスタが資産としてある** | `ボディーマスタ.xlsx` は **約 8,100 行規模** のSKU・価格情報を持ち、見積ブックへ取り込まれている【事実】|
| **版代・プリント・オプション・ボディー** | 見積ロジックは、加工区分や明細行のルールが VBA・シート上に体系化されている【事実】|

**まとめ:** ここまで作り込まれているからこそ、次の段階は **「別ツール・別ファイルに分散した力を、一本のデータの流れに統合する」** ことに価値があります。今回のシステム化は、これまでの投資を否定するものではなく、**積み上げを次の形に載せ替える取り組み** として位置づけます。

## 3. 現在の業務フロー

以下は、ヒアリング・運用説明と Excel 解析を組み合わせた **いまの流れ** です。枠外の補足は **【事実】／【推測】** を分けて記載します。

```text
Web注文フォーム
  ↓
メール受信 【推測：一般的な流れ。通知方法の詳細は要確認】
  ↓
PDF受付票確認 【推測。Web・メール連携の詳細は要確認】
  ↓
見積書作成システム.xlsm（入力・マスタ参照・見積書シート生成） 【事実】
  ↓
案件ごとの Excel ファイル保存（例：yyyymmdd_案件名.xlsm） 【事実：コードより】
  ↓
売上台帳作成ツール.xlsm（フォルダ内 *.xls* 走査・「見積書」シートから抽出） 【事実】
  ↓
月次請求書作成ツール.xlsm（同一フォルダから集約・請求ブック＋PDF） 【事実】
  ↓
請求書 PDF 出力 【事実：月次ツールおよび見積ブック側の PDF マクロ】
```

**ボディーマスタの位置づけ【事実】:**  
`ボディーマスタ.xlsx` は、見積ブック内の **`ボディーマスタ` シートへマクロで全件コピー**（値貼り付け）され、見積入力（ボディー行・UserForm 検索）の参照元になります。ファイルパスは **環境依存のコードが残っている** ため、運用上の注意点として調査レポートに記載済みです。

## 4. 現在の問題点

問題点は **「Excel が悪い」** のではなく、**データとルールの置き場所** に起因する **構造的な負荷** として整理します。

| 問題点 | 現在の状態 | 起きていること |
|--------|------------|----------------|
| **データ分散** | 見積は案件ごとブック、台帳・請求は別ブック、マスタは別 xlsx | 同じ顧客・案件でも **ファイルをまたいで** 情報が分断される |
| **二重入力・転記** | 見積入力後、集計ツールがブックを開いて値を読む | 入力と集計の間に **ブリッジ作業・確認** が挟まりやすい【推測】|
| **PDF／Excel 探索** | メール・フォルダにファイルが散らばりやすい【推測】 | 「どの版が正か」の判断が **探索コスト** になりやすい |
| **VBA のブラックボックス化** | 開発者と連絡が取れない状況がドラフトに記載 | 仕様変更・障害時に **読み解く人が限られる** |
| **ファイル名・セル位置依存** | 月次・売上はファイル名8桁日付や特定セル（例：BA1/BB1）を参照【事実】 | 軽微なレイアウト変更が **連鎖的に壊れる** リスク |
| **商品マスタ更新負荷** | マスタを更新後、見積ブックへ **都度取込**【事実】 | SKU 増・改定のたびに **手戻り・反映漏れ** が起きやすい【推測】|
| **売上・請求のフォルダ走査依存** | `Dir` でフォルダをループし各ブックを開く【事実】 | 件数増で **時間・安定性・権限管理** の負荷が増す |
| **インボイス対応の不明確さ** | VBA 文字列に「インボイス」「登録番号」等が見当たらない【事実】 | 帳票セル側の記載は **要確認**。制度対応を **データで担保しにくい** |
| **担当者依存** | 運用上、フォルダ運用・テンプレ版管理・マスタ更新が特定者に集中しやすい【推測】 | 休暇・異動で **再現性が落ちる** |

## 5. このまま続けた場合のリスク

経営判断に載せやすい言葉で整理します。

1. **件数が増えるほど、「探す・開く・突合する」作業が線形に増える**（フォルダ走査・ファイル探索の限界）。
2. **担当者が変わると、暗黙の運用ルールの継承が難しく、業務が一時停止するリスク**がある。
3. **Excel と VBA を安全に直せる人が社内・外部にいない場合、法令・税率・フロー変更への追従が遅れる**。
4. **転記・ファイル操作が増えるほど、請求漏れ・金額ミス・集計違いの確率が上がる**（完全防除は誰にも不可能だが、構造で下げられる）。
5. **メーカー・SKU が増えるほど、マスタ取込と現場確認の負荷が増える**。
6. **補助金や税務説明で「データがどこにあり、いつ誰が確定したか」を示す**とき、ファイル分散型は説明コストが高い。

## 6. 今回のシステム化の目的

**受注・見積・売上・請求を一つの流れで管理し、一回入力した情報が次の業務へ自然につながる状態を作る。**

それに伴い、次の考え方を **自然に** 実現します。

- **人を増やす前に構造を整える** … 採用や残業で押し流す前に、データの流れを一本化する。  
- **一回の入力でつながる** … 案件・明細を一度登録し、見積PDF・売上計上・請求に再利用する。  
- **誰がやっても回る** … 手順と権限が画面に落ち、特定個人の記憶に依存しにくくする。  
- **ブラックボックス化しない** … 仕様・マスタ・帳票ルールが文章と DB で説明できる状態にする【推測：運用ルール含め要合意】。  
- **将来拡張できる** … 在庫 API・会計連携・AI は **後から足せる境界** で設計する。

| 課題 | 優先度 | 理由 |
|------|--------|------|

| **金額・税率・税額** の計算ルールの明示化と実装 | **P0** | 現状はセル数式＋VBA の組合せ【事実】。監査・インボイス説明に必要。 |

## 8. 課題ごとの解決方針

| 課題 | 解決方針 | 解決後にどう変わるか |
|------|----------|----------------------|
| Excel ファイルが分散している | **案件・見積・請求を DB で一元管理**する | 顧客・案件単位で履歴と状態が追える。フォルダ探索が主役ではなくなる。 |
| フォルダ走査集計に依存 | **DB 上の日付・ステータスで集計**（月次請求・売上一覧） | 処理が再現可能・高速になり、権限管理もしやすい。 |
| セル位置・ブック構造に依存 | **スキーマ（項目定義）に立ち返る** | 変更の影響範囲がコードレビューで追える。**ブラックボックス化を減らす**。 |
| マスタ更新が重い | **ボディーマスタを DB 化**し、**CSV アップロード**等で一括更新 | 反映漏れ・バージョン違いを減らし、**価格改定に強く**なる。 |
| インボイス要件がコード上不明瞭【事実】 | **登録番号・交付条件をマスタ化**し帳票に反映 | 説明資料・監査・取引先対応がしやすい。**要確認**項目はヒアリングで埋める。 |
| 担当者依存 | **画面・権限・手順**で標準化 | **誰がやっても回る**オペレーションに近づく。 |
| Web 注文と見積のギャップ | **API／取込／手入力のいずれかで接続**【要確認】 | 入口から案件が途切れにくい。 |

## 9. システム化後の業務イメージ

```text
顧客が Web から注文 【既存フォーム or 将来の統合画面：要確認】
  ↓
管理画面で案件として受付（ステータス：受付／見積中／確定 等）
  ↓
商品マスタ（ボディー・料金表相当）を参照して見積明細を編集
  ↓
見積 PDF を発行（社内・顧客に共有）
  ↓
受注確定
  ↓
売上として計上（一覧・月次で見える）
  ↓
月次請求に集計
  ↓
インボイス対応の請求書 PDF を発行
```

### Before / After（業務の見え方）

| 観点 | Before（いま） | After（目指す姿） |
|------|----------------|-------------------|
| データの置き場 | 複数ブック＋フォルダ | **一本の業務 DB** |
| 入力 | 見積用・（暗黙の）再入力・集計用 | **原則一回**（修正・承認は別） |
| 集計 | マクロがフォルダを開く | **クエリ・締め処理** |
| マスタ | xlsx をコピー取込 | **DB＋CSV 等** |
| 帳票 | Excel 印刷 PDF | **同一データから PDF 生成** |
| インボイス | **帳票・VBAから要確認** | **項目をデータで持つ**【方針】 |

## 10. 必須要件

以下は **業務上守りたい能力** を分類したものです（画面名・API 名には落とさない）。

### 顧客管理

- **顧客情報**（名称、住所、連絡先など）
- **担当者**（複数可になってもよい【要確認】）
- **請求先**（納品先と分離するか **要確認**）
- **注文・見積・請求の履歴**

### 受注管理

- **Web 注文の取込**（連携方式は **要確認**：既存フォーム維持／置換／手入力併用）
- **案件管理**（案件番号・案件名・紐付け）
- **ステータス管理**（受付→見積→確定→請求 等）

### 見積管理

- **見積明細**（版代・プリント・オプション・ボディー等の行種別を保持）
- **税率・送料・手数料** 等、既存Excelで扱っている項目【事実】
- **見積 PDF 出力**

### 売上管理

- **売上一覧**
- **月次集計**
- **顧客別集計**（少なくとも CSV 等で出せること）

### 請求管理

- **月次請求書**（複数案件の集約イメージは現行ツールに準拠）
- **インボイス対応**（必要記載・登録番号の扱いは **要確認**）
- **請求 PDF 出力**

### 商品管理

- **ボディーマスタ**（検索用キー、メーカー、商品コード、商品名、色・サイズ、上代・下代・販売価格 等【事実】）
- **メーカー／色／サイズ／商品コード** の管理
- **料金表マスタ相当**（DTF・インクジェット等の_grid【事実】）をどう DB 化するかは基本設計で詳細化
- **CSV 取込**（マスタ更新の実務を軽くする）

### ファイル管理

- **PDF 保管**（見積・請求・【要確認】納品・領収の扱い）
- **デザインデータ**（容量・形式・版管理は **要確認**）
- **案件への紐付け**（検索できること）

## 11. 初期開発スコープ

**120 万円規模の提案** で説明しやすいよう、初期は **「Excel 分散を解消する核」** にフォーカスします。  
「すべての理想機能を一度に」とは書かず、**運用を止めずに載せ替える最小十分セット**を定義します。

初期開発に含める想定（**P0 中心**）：

| 領域 | スコープの狙い |
|------|----------------|
| **顧客管理** | 顧客マスタと履歴の参照ができる |
| **案件管理** | 案件単位で状態が追える |
| **見積管理** | 現行に近い明細構造と PDF |
| **請求管理** | 月次締め・請求書・PDF・**インボイス項目の枠** |
| **売上管理** | 一覧と月次・顧客別の基礎集計 |
| **商品マスタ** | ボディーマスタ DB 化、**初期一括取込**【事実：8千行前後】 |
| **PDF 出力** | 見積・請求を優先。**納品・領収**は **要確認**（既存にある【事実】） |
| **インボイス対応項目** | データ項目と帳票への出力（要件は **要確認**） |
| **管理画面** | 社内オペレーションのメイン入口 |

**複雑性の内訳（提案・概算の説明用）:**  
現行 VBA に見られる **明細種別・税率・マスタ・月次集約・PDF** を Web で置き換えるため、**業務システム新築に近い工数**になり得る——一方で **AI 完全自動や在庫 API は含めない**（§12）ことで、スコープと費用のバランスを取る。

## 12. 初期開発ではやらないこと

次は **初期スコープ外** とし、**将来フェーズまたはオプション**として扱います。

| やらないこと（初期） | 理由／将来 |
|----------------------|------------|
| **メーカー在庫 API 連携** | 要件整理ドラフトでも「現時点困難」と整理済み。受注後確認で回す。 |
| **AI による完全自動見積** | 誤価格リスク。将来は **下書き補助** として検討（P3）。 |
| **会計ソフト自動連携** | 会計側仕様・契約が必要。エクスポート先行。 |
| **既存 Excel との完全双方向同期** | 競合・整合性維持が極難。必要なら **インポートのみ** を個別設計。 |
| **帳票のピクセル完全一致** | 体裁は既存 PDF と並べて **Fit & Gap** で調整。完全一致は保証対象外とする。 |
| **すべての例外業務の完全自動化** | 例外は画面・ワークフローで吸収し、自動化は段階的。 |

## 14. 補助金活用を見据えた整理

**補助金・税制の制度要件は、申請時点の公募要領・専門家の確認が必須**です。ここでは **説明のしやすさ** の観点のみ整理します。

| 観点 | システム化との対応 |
|------|-------------------|
| **受発注の電子化** | Web 入口〜案件管理の一本化で説明しやすい。 |
| **請求の電子化** | 請求書 PDF とデータ管理のセット。 |
| **インボイス対応** | 登録番号・記載項目を **データ管理**する方針。 |
| **業務効率化** | フォルダ走査・転記削減の Before/After。 |
| **データ管理の改善** | 履歴・検索・エクスポート。 |

## 15. 提案書へ反映すべき要点

- 現在の Excel／VBA は **業務システムとしてよく作り込まれている**（見積・PDF・売上・請求・マスタ・フォルダ集計まで【事実】）。
- ただし **データが分散**し、**ファイル名・セル位置・VBA** に依存するため、**継続的な保守・制度対応・スケールに限界**が出やすい。
- **Web 統合**により、**受注から請求まで一気通貫**のデータの流れを作れる。
- **一回の入力で次工程につながる**ため、ミスと作業時間を構造的に減らせる。
- **管理者も現場も同じ情報**（案件・金額・状態）を参照しやすくなる。
- **120 万円規模**は、既存ツールの置換＋**商品 DB 化（大規模マスタ）**＋**帳票・請求・インボイス項目**を含むため、**筋の通った初期レンジ**として説明できる。
- **補助金**の対象・率は要領次第だが、**説明しやすい題材**になりやすい。

## 17. まとめ

現在の Excel／VBA は、これまでの業務を支えてきた **重要な仕組み** です【評価】。見積から PDF、売上台帳、月次請求、ボディーマスタ取込まで、**現場に必要なことは既にある程度そろっている**【事実】。

一方で、**業務量の増加**、**インボイス等の制度対応**、**担当者の入れ替わり**、**商品マスタの肥大化**を考えると、**Excel とファイル管理だけで支え続けるには限界**が見え始めます。問題の本質はツールの善恶ではなく、**情報が分散し、ルールがファイルやマクロに埋もれやすい構造**にあります。

今回のシステム化は、現在の業務を **否定するものではありません**。これまで積み上げてきた仕組みを、**より安全で、誰でも使いやすく、将来拡張できる形**に移す取り組みです。**人を増やす前に構造を整え**、**一回の入力でつながる** 流れを作ることが、明るい未来への近道になります。
