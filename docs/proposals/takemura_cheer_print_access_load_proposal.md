## 1. ご相談の背景

大規模イベント（アリーナ級＝数千人規模の告知・来場を想定）で Cheer Print を案内する予定があり、アクセス集中時の **遅延・停止** が心配、とのご相談をいただきました。

現時点のヒアリングでは、次のような構成です。

- 公開サイト中心（ログイン機能なし）
- WordPress で運用（テーマは完全自社制作）
- AWS EC2 **t3.micro** 相当（メモリ約 1GB）上で稼働
- サーバー本体の実装・運用は **制作会社** 側

**本提案のゴール:**  
イベント当日に、サービス停止や著しいレスポンス低下を起こしにくい状態へ近づけるため、**必要な確認・対策・測定**を段階的に進めることです。

## 2. 現時点で分かっていること

| 項目 | 内容 |
|------|------|
| サイト | https://www.cheer-print.com/ |
| 用途 | コンビニプリント（Cheer Print）の一般公開ページ |
| 平常時 | WordPress にしては **表示が軽快**（体感・単発確認とも） |
| テーマ | 市販テーマではなく **完全自社制作** |
| CDN・ページキャッシュ | **有無は未確認**（公開ヘッダだけでは断定できない） |
| ピーク耐性 | **未確認**（今軽いことと、当日も大丈夫かは別問題） |

平常時に軽いのは好材料です。  
一方で、イベント告知・QR・MC 一斉誘導などが重なると、同じサーバーでも一気に負荷が上がります。  
そのため、「感覚」ではなく **構成確認 → 必要なら測定** の順で進めるのが安全です。

## 3. まず確認すべきこと

本格的な負荷試験の前に、制作会社へ次を確認します。回答内容によって、**簡易診断で足りるか／負荷試験まで必要か**を判断します。

### イベント側

- 開催日時・ピークになりそうな時間帯
- 告知方法（MC「今すぐアクセス」、会場スクリーンの QR、SNS 同時告知、販売開始時刻の有無など）
- 会場規模・来場見込み（分かる範囲で可）

### サーバー・WordPress 側（制作会社）

- CloudFront 等 CDN の有無
- ページキャッシュの有無と設定方針
- DB 構成（同一サーバー内か、RDS 等か）
- Apache／PHP／監視（CloudWatch 等）で見られる項目
- **本番サイトへの試験許可**（時間帯・強度・連絡体制）

### ページキャッシュについての注意

ページキャッシュは有効な対策になり得ますが、自社テーマでも次のような機能がある場合、設定を誤ると表示や検索がおかしくなることがあります。

- 検索・絞り込み
- ランダム表示
- Cookie や URL パラメータによる表示切替
- 外部サービス連携
- 公開直後の頻繁な更新

そのため、キャッシュ導入は **竹村様が単独で本番へ入れる作業にはせず**、制作会社へ可否と設定内容を確認・実施いただく方針とします。

## 4. 推奨する進め方

価格を先に決めるのではなく、**確認 → 判断 → 必要な段階だけ実施**します。

```text
① 制作会社へ構成・イベント導線を確認
        ↓
② 簡易診断（公開情報＋ヒアリング整理＋改善項目）
        ↓
③ 必要なら対策（キャッシュ／CDN／一時スペック等 ※制作会社）
        ↓
④ 必要なら段階的な負荷試験（次廣サーバーから夜間などに実施）
        ↓
⑤ 結果報告 → 改善判断（必要なら再試験）
```

| 順番 | 内容 | 主担当 |
|------|------|--------|
| 1 | 制作会社へ構成確認・試験許可取り | 竹村様（窓口）／制作会社（回答） |
| 2 | 現状の簡易診断・改善項目の整理 | 次廣 |
| 3 | 必要な対策の実施 | **制作会社** |
| 4 | 段階的な負荷試験（許可時間帯） | 次廣（実行）／制作会社（監視） |
| 5 | 結果報告と改善判断 | 次廣（報告）／制作会社（対応） |

**スペックについて:**  
キャッシュや CDN とあわせて、イベント期間中だけ t3.small／t3.medium などへ **一時的にスペックアップ**する方法も有力です。実装リスクが比較的小さく、料金差も限定的になりやすいため、必要に応じて制作会社と検討します。

## 5. 次廣の対応範囲

### 含むもの

- 試験シナリオの作成（対象 URL・段階的な負荷の考え方）
- 負荷試験プログラムの作成（**次廣側サーバー**に用意）
- 許可された時間帯での実測（原則、夜間など低利用帯）
- 応答時間・エラー率などの分析
- 竹村様／制作会社向けの結果報告書作成
- 改善の優先順位に関する提案（実装は制作会社）

### 含まないもの（制作会社側）

- AWS・CDN・Apache／PHP 等の設定変更
- WordPress 本体・キャッシュプラグインの本番設定
- イベント当日の常駐監視・無停止の保証

| 役割 | 担当 |
|------|------|
| 調整窓口・イベント情報の共有 | **竹村様** |
| キャッシュ／CDN／AWS 設定・本番改修 | **制作会社** |
| 負荷試験の設計・実行・結果分析・報告 | **次廣** |

## 6. 概算費用（税別）

現状構成と試験条件を確認したうえで、必要な範囲だけを選び、**正式見積**をご提示します。  
最初から最大金額を確定するものではありません。

| 段階 | 内容 | 概算（税別） |
|------|------|--------------|
| **① 簡易診断** | 公開情報の確認、構成ヒアリング整理、改善項目の整理、制作会社向け確認リスト | **3〜5 万円** |
| **② 負荷試験** | シナリオ作成、試験プログラム作成、許可時間帯での実測、結果分析・報告 | **8〜15 万円** |
| **③ 再試験** | 改善後の Before／After 確認（1回想定） | **3〜5 万円** |

### 進め方の例

1. まず **① 簡易診断** で、制作会社回答を踏まえ「これだけで足りそうか／②が必要か」を判断する  
2. ピークリスクが高い、または数字で確認したい場合に **② 負荷試験**  
3. 対策を入れたあと、効果を見るなら **③ 再試験**

### 費用に含まないもの

- AWS 利用料（CDN、スペック変更、転送量など）
- 制作会社の作業費・保守費
- 本番障害に対する無制限の保証
