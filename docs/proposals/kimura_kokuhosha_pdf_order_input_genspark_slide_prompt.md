## Genspark用プロンプト（以下をそのまま貼り付け）

あなたは、製造業・老舗企業の業務改善、基幹システム連携、BtoB提案資料に強い、経営者・現場管理者向けプレゼンのプロフェッショナルです。

以下の提案書をもとに、**株式会社国宝社（BPS木村）・木村秀継さん**向けの、分かりやすい提案スライドを作成してください。

---

### スライドの目的

国宝社様の製本販売管理において、**出版社から届くPDF注文書の手入力負担**を減らすため、既存の VB + Oracle 基幹システムを温存しながら、**社内ローカルサーバー上の Web 入力支援システム**を段階的に導入する提案を説明する。

第2回121（2026-05-29）で合意した方向性に沿い、**実現イメージと進め方の判断材料**を提示する。  
今回は **無償提案・簡易モック共有** の段階であり、実装確約や最終見積の提示ではない。

---

### 想定読者

- 木村秀継 様（経営者）
- 国宝社の事務・受注担当者
- システム・IT担当（いる場合）

読者は Oracle や VB の詳細に詳しくない可能性がある。  
**「基幹を壊さない」「社内で完結」「人が確認してから登録」** が伝わる構成にしてください。

---

### トーン

- 売り込みすぎず、現場の安心感を大切にする
- 「全部作り替え」ではなく **「入口だけ改善」**
- 「AIで全部自動化」ではなく **「解析＋人の確認」**
- 創業160年・V字回復の強みを尊重する
- Google Workspace / GAS は **不採用**（121合意）。社内 Web が本命であることを明確に
- 老舗企業の **継続性・セキュリティ・段階的DX** を前面に

### 必ず入れる重要メッセージ

1. 国宝社は創業160年の製本会社。製本基幹は **VB + Oracle** で長年稼働しており、**これを温存する**。
2. いちばんの負担は **PDF注文書の手入力**。大口顧客だけで **月300〜400件**、全体ではその倍近い可能性。
3. 提案の核は **「PDF注文書入力支援 Web」**。基幹の **周辺システム** として追加する。
4. **Google Workspace / GAS 案は不採用**。第2回121で **社内ローカルサーバー上の Web** に合意。
5. インフラは **Hyper-V 上 Linux Web**、社内 Wi-Fi / VPN、**外部公開なし**。
6. 流れは **PDFアップロード → 解析 → 人が確認・修正 → Oracle登録**。**100%自動化は目指さない**。
7. 出版社ごとに PDF フォーマットが異なるため、最初は **PC作成PDF・数パターン** から始める。
8. 手書き注文書・基幹全面刷新・得意先直接入力は **Phase 1 では含めない**。
9. Phase 1 で入口改善 → Phase 2 で日報等の Web 化 → Phase 3 で外部連携・将来展開。
10. 次のステップは **簡易モック共有 → 追加PDFサンプル → 詳細設計 → PoC**。
11. **ROI（人件費ベース・試算）:** 大口350件/月・削減率30〜50%で **年210〜700時間** 相当。パッケージB（180万）の回収 **約1.0〜3.4年**（1件時間・単価要確認）。**保証しない**。
12. **概算（税別）:** 検証パック **50万前後** / 初期導入（推奨）**180万前後** / 拡張 **230万前後**（詳細は概算見積参照）。50万の検証で ROI 前提を実測確認できる。

#### 3. 国宝社様の強みと、いまの課題

**2カラム構成**

左（強み）:
- 創業160年
- 製本業界でV字回復
- ジャンプ・マガジン等の実績
- 家系図・ルーツ調査も展開

#### 4. 現状の流れ（As-Is）

**フロー図にする:**

出版社・大口得意先  
↓ PDF注文書  
事務員が目視確認  
↓ **手入力**  
既存「製本販売管理システム」（VB + Oracle）  
↓  
生産・納品へ  
↓  
Excel で日報・分析

**強調:** ボトルネックは **PDF → 手入力** の部分

---

#### 5. 表面の要望と、裏にあるお困りごと

**上段: 要望（5つ）**
1. PDF手入力を減らしたい
2. 既存システムは温存したい
3. Oracleへ安全に登録したい
4. 社内ローカルWebで運用したい
5. 段階的に広げたい

**下段: 裏のお困りごと（3〜4点に要約）**
- 件数が増えると **人が足りなくなる**
- 転記ミス・見落としのリスク
- フォーマット差が **担当者の記憶に依存**
- 基幹を触る全面改修は **リスクが大きい**
- 機密データを外部クラウドに出したくない

---

#### 6. なぜ「全部作り替え」ではないか

**比較表**

| 選択肢 | 判断 |
|--------|------|
| 基幹システム全面刷新 | リスク大・コスト大・現場停止の恐れ |
| Google Workspace / GAS | **不採用**（121合意） |
| 既存VBに直接改修 | 保守・ソース・テストの制約 |
| **周辺Web + Oracle連携** | **今回の本命** |

**メッセージ:** 160年続く基幹を守り、**入口だけ**改善する

---

#### 7. 提案するシステム像（To-Be Phase 1）

**3層の図解:**

```
【新規】PDF入力支援 Web（社内）
  PDFアップロード / 解析 / 確認・修正 / Oracle登録
        ↓
【既存】VB + Oracle 製本販売管理システム（温存）
        ↓
【既存】Excel 日報・分析 等
```

**操作の流れ（6ステップ）:**
1. PDFをドラッグ&ドrop
2. 基本情報・明細を自動抽出
3. 画面上で確認・修正
4. 不明項目を目立たせる
5. 登録前に最終確認
6. Oracle登録 → 管理No表示

---

#### 8. なぜ「100%自動化」ではないか

**4つの理由（アイコン付き箇条書き）:**
- 出版社ごとに PDF レイアウトが違う
- 備考欄に現場判断（紙種・加工・納期）が必要
- Oracle 登録には既存ルール（採番等）の順守が必要
- **人の確認** があるから、現場が安心して使える

**メッセージ:** 自動化は **入力支援**。最終判断は **人**

---

#### 9. インフラとセキュリティ（121合意）

**図解または箇条書き:**

- 既存 Windows サーバーの **Hyper-V** 上に Linux Web
- 社内 Wi-Fi / 有線 LAN からアクセス
- 社外からは **既存 VPN**
- **外部インターネット公開なし**
- Google Workspace / GAS は **使わない**

**強調:** データは **社内完結**

**Phase 1（最優先）** — PDF注文書入力支援  
- PC作成PDF・初期数パターン

#### 11. Phase 1 で含めること / 含めないこと

**2カラム**

含める:
- PDFアップロード・解析

#### 13. 投資対効果（ROI）— 人件費ベース試算

**メッセージ:** 180万円だけでなく、**何年で元が取れるか** を試算する（断定・保証はしない）

**試算の式（小さく表示）:**

年間削減効果 ＝ 削減時間 × 人件費単価  
投資回収期間 ＝ 初期投資 ÷ 年間削減効果

**前提（121ベース）:**

- 対象: 大口顧客 PDF **月350件**
- 削減対象: **手入力時間**（確認・修正時間は別。ミス低減はプラスα）
- 1件入力時間・人件費単価: **貴社ヒアリング後に確定**（現時点は試算）

**3シナリオ表（大きく見せる）:**

| シナリオ | 1件時間 | 削減率 | 年間削減 | 年間効果※ | 回収期間※ |
|----------|---------|--------|----------|-----------|-----------|
| 保守 | 10分 | 30% | 210時間 | 約52万円 | 約3.4年 |
| **標準** | 15分 | 40% | 420時間 | **約105万円** | **約1.7年** |
| 楽観 | 20分 | 50% | 700時間 | 約175万円 | 約1.0年 |

※人件費単価 **¥2,500/時間** の参考試算。パッケージ B（180万円）想定。

**判断の目安（箇条書き）:**

- 回収 **〜2年** → 導入検討しやすい
- **2〜3年** → 検証パックA（50万）から段階導入
- **3年超** → 1件時間・件数の再確認

**脚注（必須）:** 削減率・回収期間の **保証は含まない**。正式見積ではなく **概算試算**。
