## 1分で分かる今回の相談

> **対象読者: 全員（ここだけ読めば概要が分かります）**

Cheer Print は、普段は問題なく速く表示されています。  
ただし、現在は一般的な小規模サイト向けに使われることの多い構成です。普段の閲覧には問題がなくても、ライブ会場や SNS から多くの人が一斉にアクセスした場合は、ページが遅くなったり、表示できなくなる可能性があるため、別途確認が必要です。

1. 一度作ったページを使い回す「キャッシュ」

3. イベント期間中だけサーバーを大きくする対策

を検討します。  
さらに必要な場合は、実際に複数人が同時に見ている状態を再現し、どこまで安定して表示できるかを測ります。

**竹村さん**は制作会社との調整を担当し、**サーバー設定は制作会社**、**測定と報告は次廣**が担当します。

#### 「今はめちゃくちゃ軽い」ことの読み方

体感・単発測定とも、**平常時は軽快**です。WordPress でも次のような理由で軽く感じることがあります。

| 要因（想定） | 本件との関係 |
|--------------|--------------|
| **完全自社テーマ** | 汎用テーマ＋大量デモ機能より、必要な処理だけに寄せやすい |
| **前面に出るプラグインが少なめ** | HTML 上は search-filter／wp-data-access 等（＋AIOSEO）。重いページビルダー連打型ではなさそう |
| **ログインなしの閲覧中心** | 会員セッションや重い管理系処理が乗らない |
| **今は混んでいない** | 1人〜数人のアクセスでは t3.micro でも余裕が出やすい |

**ただし「今軽い」＝「アリーナ級（数千人規模）の同時アクセスでも大丈夫」ではない。**

```text
平常時（今）     … 厨房に注文がほぼない → すぐ出てくる → 軽い
ピーク時（当日） … 同じ厨房に注文が一斉に入り、処理が追いつかなくなる可能性がある
```

だから対策の優先順位は変わりません。

1. まず制作会社へ **キャッシュ／CDN／AWS 構成**を確認（竹村さんが単独で本番へキャッシュを入れない）  
2. 必要なら制作会社が **キャッシュ・CDN・一時スペック**等を実施  
3. 必要なら **夜間の負荷実測**で、仮説の同時アクセス帯でも軽さが残るかを数字で確認  

むしろ現状が軽いのは好材料で、**対策を足せばピーク耐性を伸ばしやすい土台**、と伝えられます。

## 0.2 Webサイトの仕組み（静的／動的／WordPress）

> **対象読者: 全員**（たとえ中心。技術名は補足）

竹村さんの「PHP を毎回実行せず HTML を返せないか」は、技術的には正しい方向です。  
ここでは、負荷対策の前提として **サーバーが何をしているか** を短く共有します。

### 全体像（スマホ → 画面が出るまで）

```text
[利用者のスマホ／PC]
        │  「このページ見せて」と依頼（リクエスト）
        ▼
[インターネット]
        │
        ▼
[Webサーバー]  ← Cheer Print では Apache（Amazon Linux）を確認
        │  必要ならプログラムやDBも動かす
        ▼
[HTML・画像・CSS などを返す]（レスポンス）
        │
        ▼
[ブラウザが画面を描画]
```

**Webサーバー**は、レストランの「受付＋厨房の入口」のような役割です。  
依頼を受け、用意できた料理（ページ）を返します。人が増えると、受付も厨房も混みます。

本件のサーバー実体は **AWS 上の EC2（仮想マシン）1台**に、Apache と WordPress が入っているイメージです（RDS 分離の有無は要確認）。

---

### 静的サイト（Static）

あらかじめできあがったファイル（HTML／CSS／画像）を、そのまま配る方式です。

```text
リクエスト → Webサーバーがファイルを探す → そのまま返す
（プログラムも DB もほぼ動かない）
```

| 特徴 | 内容 |
|------|------|
| 強み | 速い・同時アクセスに強い・落ちにくい |
| 弱み | お知らせ更新・商品追加のたびにファイルを作り直す運用になりやすい |
| たとえ | **印刷済みのチラシを棚から渡す**だけ |

イベント告知だけで中身がほぼ変わらないページなら、静的が最強です。  
Cheer Print のように日々ニュースや商品が増えるサイトは、全部を静的だけで回すのは運用が重いです。

---

### 動的サイト（Dynamic）

アクセスのたびに、プログラムがページを**その場で組み立てる**方式です。

```text
リクエスト
  → プログラム実行（WordPress なら主に PHP）
  → 必要ならデータベース（MySQL 等）に問い合わせ
  → HTML を生成して返す
```

| 特徴 | 内容 |
|------|------|
| 強み | 管理画面から記事追加・デザイン変更がしやすい |
| 弱み | **人が来るたび厨房（PHP＋DB）が動く** → 同時アクセスで負荷が乗りやすい |
| たとえ | **注文が入るたびに厨房で一品ずつ作る** |

「動的＝悪い」ではありません。更新しやすい代わりに、ピーク対策が必要、というだけです。

---

### WordPress の場合（Cheer Print）

WordPress は典型的な **動的 CMS** です。

```text
① ブラウザが https://www.cheer-print.com/ を開く
② Apache（Webサーバー）が受け取る
③ PHP で WordPress が起動する
④ **自社制作テーマ**（見た目・テンプレート）やプラグインが動く
⑤ 記事・商品情報を DB から読む（ことが多い）
⑥ HTML を組み立てて返す
⑦ 続けて画像・CSS・JS も取りに来る（ページ1枚で通信が複数回）
```

| 部品 | 役割（たとえ） |
|------|----------------|
| **EC2（サーバー本体）** | 店舗そのもの（狭い店＝ t3.micro） |
| **Apache** | 受付 |
| **PHP / WordPress** | 厨房のシェフとレシピ |
| **テーマ（完全自社制作）** | 盛り付け・見た目のルール。市販テーマではなく制作会社がスクラッチで作成（パス上の名称は `my-theme`） |
| **プラグイン（AIOSEO 等）** | 追加の調理工程（多いほど重くなりやすい） |
| **MySQL（DB）** | 食材倉庫（在庫照会） |
| **画像・CSS・JS** | お皿や盛り付け用具（これも毎回運ぶと重い） |

ログインなしの一般公開でも、**「見るだけ」でも上記①〜⑦は走り得ます。**  
だからイベントで人が集中すると、PHP 実行・DB 照会・メモリ不足が同時に起きやすい、というのが第2回で共有した不安の正体です。

---

### 「HTML を返すだけ」に近づける＝ページキャッシュ

竹村さんの希望（毎回 PHP を走らせない）に一番近いのが **ページキャッシュ** です。

```text
【キャッシュなし＝毎回フル調理】
リクエスト → PHP → DB → HTML 生成 → 返す

【ページキャッシュあり＝一度作った料理のコピーを出す】
最初の1回: PHP → DB → HTML を生成し、コピーを保存
2回目以降: 保存済み HTML をそのまま返す（厨房ほぼ休み）
```

| 方式 | 厨房（PHP/DB） | 同時アクセス耐性 |
|------|----------------|------------------|
| 動的のまま（キャッシュなし） | アクセスごとに稼働 | 弱い（t3.micro は特に） |
| **ページキャッシュ** | ほぼ初回だけ | **大きく改善** |
| 完全静的ファイル | ほぼゼロ | 最も強い |

公開ページ中心の Cheer Print は、ページキャッシュとの相性が良いタイプです。  
（管理画面や「今ログインしている自分専用ページ」はキャッシュしにくい。本件はログインなしなので有利。）

---

### CDN（CloudFront 等）は何をするか

CDN は、画像や CSS などを **利用者の近くの倉庫** から配る仕組みです。

```text
【CDN なし】
全員のスマホ → 遠い1店舗（EC2）に殺到

【CDN あり】
画像・CSS など → 各地の倉庫（エッジ）から配布
HTML も設定次第で倉庫から返すこともある
EC2 は「本店」として残るが、行列が減る
```

ページキャッシュが「厨房を休ませる」なら、CDN は「本店への来店自体を減らす」対策です。  
両方あると、イベントピークに強くなります。

---

### 本件への当てはめ（一文で）

| 現状のイメージ | Cheer Print |
|----------------|-------------|
| サイト種別 | **WordPress＝動的** |
| サーバー | EC2 t3.micro ＋ **Apache**（小さな1店舗） |
| 不安 | 人が集中すると厨房（PHP/DB）がパンク |
| まずやること | **ページキャッシュ**で「印刷済みチラシを渡す」状態に近づける |
| あわせて | **CDN** で画像などを本店から外す |
| そのあと | 必要なら負荷試験で「想定した混雑でも大丈夫か」を数字で確認（§2.0／§2.1） |

「フロント側だけで全部解決」は難しく、**キャッシュ（サーバー／WP側）＋ CDN（配信側）** が本筋です。  
ブラウザだけの工夫（画像を小さくする等）は補助にはなりますが、ピーク耐性の主役にはなりません。

## 2.0 「同時アクセス」とは何か（よくある誤解）

> **対象読者: 全員**

### 結論（竹村さん向け一言）

**「同時 40」＝“そのサイトには1日40人までしか入れない”ではありません。**  
**「ちょうど同じ瞬間に、だいたい40人がサイトを開いている／読み込んでいる」**という意味です。

1日に数百〜数千人が訪れても、タイミングがずれていれば同時は小さくなります。  
逆に、MCで「今すぐ QR！」と誘導すると、短時間だけ同時が一気に跳ねます。

### たとえ

| たとえ | 意味 |
|--------|------|
| **エレベーター** | 「同時 40」は、かごに今乗っている人数。1日の乗客総数ではない |
| **窓口** | 列に並んだのべ人数ではなく、**今カウンターの前に立っている人数** |
| **アリーナ** | 入場者 8,000 人でも、トイレに並ぶのはそのうちの一部が「同時」 |

### のべアクセス・同時アクセス・VU（分けて使う）

| 指標 | 意味 | 例（シナリオ A） |
|------|------|------------------|
| **のべ訪問** | 当日サイトを開いた人の合計イメージ | 約 **400 人**がどこかのタイミングで訪問 |
| **同時アクセス** | 同じ時間帯にサイトを利用している人数のイメージ | ピーク時に約 **40 人**が重なって利用 |
| **VU（Virtual User）** | 負荷試験上で、決められた閲覧行動を同時進行する仮想利用者。実利用者数と1対1ではない | k6 で 40 VU を走らせる |
| **リクエスト／秒（RPS）** | サーバーが1秒あたり処理する通信回数 | 同じ VU 数でも `sleep` 次第で RPS は大きく変わる |

```text
【NGな理解】
同時40 → 40人までしか使えない／41人目から入れない
VU 180 = 実ユーザー180人が画面を眺めている

【正しい理解】
同時アクセス40 → いちばん混む時間帯に、だいたい40人が利用しているイメージ
VU 180 → 定義したページ遷移・待機時間を、180の仮想利用者が同時進行している試験条件
のべ400 → 1日（またはイベント中）にサイトを見た人の合計イメージ
```

### 負荷試験の「180 VU で合格」の意味

| 言い方 | 意味 |
|--------|------|
| **180 VU で合格** | 定義した閲覧間隔・ページ遷移を **180 の仮想利用者が同時進行しても、合否基準を満たした** |
| **180 VU で不合格** | その試験条件だと遅い／エラーが出る → キャッシュ・CDN・スペック等の検討材料 |
| **VU ≠ 入場制限** | WordPress が「181人目を拒否する」わけではない |

Cheer Print はログイン不要の公開サイトなので、実際には「入れない」より **極端に遅い・タイムアウト・サーバーダウン** が問題になります。

### なぜ同時帯の負荷を見るのか

サーバーが苦しいのは「1日の合計人数」より、**同じ時間帯に PHP／DB／回線が混むとき**です。  
負荷試験では、ビジネス上の「同時アクセスイメージ」を手がかりに **VU と閲覧間隔（sleep）・RPS** を設計します。

## 2.1 人数未定のときの推計（TOYOTA アリーナ → 同時アクセス）

> **対象読者: 全員**（表の「試験条件」列だけ技術寄り）

竹村さん側で「何人想定か分からない」場合、**会場キャパを上限母数**にして割り出すのが実務的です。  
対象イベントが [TOYOTA ARENA TOKYO](https://www.toyota-arena-tokyo.jp/organizer/)（トヨタアリーナ東京）想定なら、公式の収容目安は次のとおりです。

| 利用形態 | 収容の目安（公式・公表ベース） | 出典 |
|----------|-------------------------------|------|
| スポーツ等 | **約 10,000 人** | [施設概要](https://www.toyota-arena-tokyo.jp/organizer/)／[利用プラン](https://www.toyota-arena-tokyo.jp/organizer/plan/) |
| コンサート（ステージ設営） | **約 8,000 席** | [利用プラン](https://www.toyota-arena-tokyo.jp/organizer/plan/) |
| 可動席収納など | **約 4,700 席**（レイアウト依存） | 同上 |

※「満席＝その人数がサイトに来る」ではない。  
※会場外（SNS・配信・後日アクセス）が乗ると、**会場収容を超える**こともある。

### 割り出し式（シンプル版）

```text
当日サイト訪問ののべ人数 ≒ 入場者数 × 認知・誘導率
ピーク同時アクセス（イメージ） ≒ のべ訪問 × 同時率
試験目標 VU（仮） ≒ 上記を手がかりに置き、sleep 等とセットで定義
```

| 係数 | 意味 | 仮置きの目安 |
|------|------|----------------|
| **入場者数** | 当日チケット／キャパ | 未確定ならコンサート **8,000**、上限確認なら **10,000** |
| **認知・誘導率** | QR・スクリーン・MC・物販でサイトを知る／開く割合 | 弱い 5%／普通 15%／強い（一斉誘導） 30〜50% |
| **同時率** | そのうち「ほぼ同じ数十秒〜数分」に重なる割合 | 分散 5〜10%／開始直後や QR 一斉 20〜40% |

### 早見表（会場 8,000 人のとき・仮説）

| シナリオ | 推定されるピーク利用人数（仮説） | 試験条件の考え方（参考） |
|----------|----------------------------------|--------------------------|
| **A. 控えめ** | 約 **40 人** | 軽い確認試験の参考 |
| **B. 通常告知** | 約 **180 人** | 試験条件を決めるための参考（現時点の仮説） |
| **C. 強い一斉誘導** | 約 **480 人** | 余裕確認の参考 |
| **D. 最悪寄り** | 約 **1,200 人** | 強いストレステストの参考 |

会場 **10,000 人**なら、上表のおおよそ **×1.25**。

> **この人数と、負荷試験の VU（仮想ユーザー）は、同じ数字になるとは限りません。**  
> VU は試験上の仮想利用者で、閲覧の間隔の置き方によって実際の負荷の強さが変わります。

**重要:** 約 180／約 480 は **「この人数まで大丈夫」という保証値ではない。**  
MC・QR・販売開始時刻・SNS 同時告知などでピークは大きく変わる。  
→ **現時点の仮説として利用人数イメージを置き、イベント導線確認後に試験条件を確定する。**

### 竹村さんへの伝え方（人数が分からないとき）

0. 先に §2.0 を共有し、**「同時＝1日の上限人数ではない」**を揃える。  
1. **まず会場形態・告知方法を確認**する。  
2. 分からなければ、「通常告知ならピーク利用 **約180人イメージ**、強い誘導なら **約480人イメージ**」を参考に試験条件を仮置きし、導線が分かり次第見直す、と伝える。  
3. 約1,200人イメージは最悪寄りの参考。キャッシュ／CDN／一時スペックとセットで制作会社と検討。

### 注意（割り出しの落とし穴）

| 落とし穴 | 対策 |
|----------|------|
| 入場者≠サイト利用者 | 必ず「誘導率」を掛ける |
| 同時≠のべ | 1人が数ページ見ても、同時利用人数は「重なり」だけ |
| 会場外トラフィック | SNS バズ時は会場キャパを母数にしない（別途上振れ） |
| 検索・カート・外部決済 | Cheer Print は閲覧中心なら HTML キャッシュが効きやすい。検索連打は別枠 |
| t3.micro の現実 | **B 前後でもキャッシュなしは厳しい可能性大**。試験前にキャッシュ／CDN を優先 |

### 人数未定時の試験ターゲット（本件の仮置き案・技術者向け）

| 段階 | 試験 VU（仮） | 意味 |
|------|---------------|------|
| Stage A | 20 | ごく軽い確認 |
| Stage B | 50 → 100 → **180（仮）** | 徐々に増やす。利用人数仮説 B を手がかりにした試験条件 |
| Stage C | **180（仮）** を 5〜10 分維持 | 導線確定前の仮判定 |
| （許可時）余裕 | **480（仮）** まで短時間 | 利用人数仮説 C を手がかりにした余裕確認 |

→ 人数未定でも試験は進められるが、**利用人数仮説と VU は別物。数値は導線確認後に確定する。**

## 3. 推奨アプローチ（順序を統一）

> **対象読者: 全員**（Stage の細部は技術者向け）

外向け提案・簡易診断と同じく、次の順で進める。

```text
[構成確認・試験許可]
        ↓
[対策前の現状確認]（軽い確認。明らかに危険なら省略可）
        ↓
[制作会社による対策]（キャッシュ／CDN／一時スペック等 ※必要な場合）
        ↓
[Stage A] ごく軽い確認（10〜20 VU・1〜3分）
        ↓ 問題なければ
[Stage B] 人数を徐々に増やす（5〜10分）
        ↓ 目標に到達できたら
[Stage C] ピーク相当をしばらく維持（5〜10分）
        ↓
[必要なら再試験・対策前後の比較]
```

**注:** 現状が明らかに危険（すでに遅い・落ちる）と判断される場合は、対策前の現状確認を省略して先に対策してよい。

### Stage A — ごく軽い確認（スモーク）

- 目的: 試験の負荷が想定以上になっていないか、エラーが出ていないか確認  
- 規模: 10〜20 VU、1〜3 分  
- 合格: エラーほぼなし、主要ページが開ける

### Stage B — 徐々に増やす（ランプアップ）

- 目的: **何人程度から急に遅くなり始めるか**を確認する  
- やり方: 1〜2 分ごとに VU を増やす（例: 20→50→100→…）  
- 見るもの: 応答の遅れ、エラー、CPU／クレジット関連・メモリ（取得できる場合）

### Stage C — ピーク相当を維持

- 目的: イベント開始直後のような短時間の混雑を再現  
- やり方: 目標 VU を 5〜10 分維持（閲覧間隔は合意した待機時間）  
- 合格基準は §5

### 対策前後の比較

| 条件 | 狙い |
|------|------|
| 対策前の現状確認（安全な範囲） | いまの限界感 |
| 対策後の本試験 | キャッシュ／CDN／スペック変更の効果 |
| 再試験 | 対策前後の比較用 |

**本番でキャッシュを故意に切っての比較は危険**なので、原則は「対策前の短い現状確認 → 対策後の本試験」の時系列比較とする。

## 5. 合否基準（案）

> **対象読者: 冒頭の一般向け判定は全員／数値表は制作会社・技術者**

### 一般向け判定（先にこれ）

| 判定 | 意味 |
|------|------|
| **問題なし（合格）** | 想定したアクセスでも、ほぼ問題なく表示できる |
| **要改善** | 表示はできるが、遅くなる時間帯がある |
| **危険（不合格）** | エラーや表示不能が起きる可能性が高い |

技術者は、成功率・応答時間・サーバー使用率を使ってこの判定を行います。

### 技術者向けの数値たたき台

イベント用途の公開サイト向け。数値は確定前のたたき台。  
CPU 単独で機械的に合否を決めず、**応答・エラーとセット**で見る。

| 指標 | 合格（案） | 要改善 | 不合格 |
|------|------------|--------|--------|
| **HTTP 成功率** | ≥ 99% | 95〜99% | < 95% |
| **応答 p95（HTML）** | ≤ 2.0 秒 | 2〜5 秒 | > 5 秒 またはタイムアウト多発 |
| **5xx / 接続失敗** | ほぼ 0 | 散発 | 継続発生 |
| **EC2 CPU** | 高負荷終了後に速やかに低下し、応答基準を維持 | 80〜90%以上が継続するが応答は維持 | 90〜100%付近で応答悪化・エラー発生 |
| **メモリ** | OOM なし（取得できる場合） | swap 多発 | プロセスキル／ダウン |
| **CPUクレジット（t3）** | 残高と動作モードを確認 | 継続的に減少／余剰利用が発生 | Standard で枯渇し性能低下、または Unlimited で追加課金が継続し問題視される水準 |

**t3 と CPUクレジット（重要）**

- 通常の T3 は **Unlimited が初期設定**のことが多い。  
- **Unlimited:** クレジット残高がゼロでも直ちに性能制限されるとは限らず、余剰クレジットでバースト継続 → 条件により **追加料金**。  
- **Standard:** クレジット枯渇時に **性能がベースラインへ制限**され、遅くなる。  
- CloudWatch 確認候補: `CPUCreditBalance`／`CPUSurplusCreditBalance`／`CPUSurplusCreditsCharged`、およびクレジット設定が **standard / unlimited** のどちらか。

**メモリについて**

- EC2 標準の CloudWatch メトリクスだけではメモリ使用率が取れないことが多い。  
- **CloudWatch Agent 等が未導入の場合は取得準備が必要**（なければ OS ログや制作会社の既存監視で代替）。

**その他**

- CDN がある場合、利用者側の表示は速くても **本店サーバー（オリジン）は別指標**で見る。  
- 埋め込み動画（YouTube 等）は外部側の話が多く、WordPress サーバー本体の試験と切り分ける。

## 6. 試験シナリオ（Cheer Print 向け）

> **対象読者: 制作会社・技術者**（URL 一覧は全員が見てよい）

ログインなし前提。代表的な読み取りのみ。  
検索クエリ付き URL（`?search_word=`）は DB／動的処理が重くなりやすいので、**初期試験では除外**し、必要なら別枠で少量だけ見る。

### 6.1 推奨ターゲット URL

| 優先 | URL | 理由 |
|------|-----|------|
| 必須 | https://www.cheer-print.com/ | ランディング・画像多い |
| 必須 | https://www.cheer-print.com/news/ | キャンペーン導線 |
| 必須 | https://www.cheer-print.com/how-to-purchase/ | 購入方法（コンバージョン寄り） |
| 高 | イベント連動のお知らせ詳細（例: `/news/avexroyalbrats_01/`） | SNS／ライブからの直リンク想定 |
| 中 | https://www.cheer-print.com/bl/ 等ジャンル一覧 | 回遊で踏まれやすい |
| 低（別枠） | `/?s=` や `?search_word=` 付き | 動的検索。本番試験では慎重に |

### 6.2 仮想ユーザーの行動割合（案・本試験設計時）

| 割合（案） | 行動 |
|------------|------|
| 40% | トップ `/` を GET |
| 25% | お知らせ一覧 `/news/` または連動記事詳細 |
| 20% | ジャンル一覧（`/bl/` `/sports/` `/entertainment/` 等をローテ） |
| 10% | 購入方法 `/how-to-purchase/` |
| 5% | 静的アセット（CSS/JS/フォント）※オリジン試験では任意。CDN 後は外してよい |

※ §8.2 のサンプルコードは **均等ランダム**であり、上記割合は未反映。実試験時は重み付けする。

### 6.3 試験の分離（HTTP 主試験とブラウザ体感）

`k6` の `http.get(HTML)` だけでは、ブラウザのように **画像・CSS・JS を自動取得しない**。  
そのため §6.2 の「静的アセット 5%」は、明示取得するか別シナリオにしないと再現できない。

| 試験種 | 目的 | やり方 |
|--------|------|--------|
| **WordPress サーバー本体の負荷試験（主・オリジン負荷試験）** | PHP／DB の処理能力 | k6 HTTP で HTML 等を直接取得。大量 VU はこちら |
| **ブラウザ体感確認（副）** | 画像・CSS・JS を含む表示の体感 | 少数の k6 browser または実ブラウザ。大量 browser VU は負荷発生側の消費も大きいため主試験と分ける |

**やってはいけないこと（原則）**

- `wp-admin` やログイン試行を負荷対象にしない  
- `/contact/` へのフォーム連打、検索爆撃、`/wp-json/`・XML-RPC 連打をしない  
- 許可なくピークの数倍を長時間かけない  
- 本番ピーク時間帯に無断で実施しない

`loadtest-cheer-print.js` 例（ピーク VU・sleep は合意後に変更）:

```javascript

export const options = {
  stages: [
    { duration: '1m', target: 20 },   // Stage A
    { duration: '2m', target: 50 },   // Stage B
    { duration: '3m', target: 100 },  // 仮ピーク（合意値に変更）

  ],
  thresholds: {
    http_req_failed: ['rate<0.01'],          // エラー率 < 1%
    http_req_duration: ['p(95)<2000'],       // p95 < 2s
  },
};

const BASE = __ENV.BASE_URL || 'https://www.cheer-print.com';

// 簡易サンプル: 均等ランダム（§6.2 の割合は未反映）
const PAGES = [
  `${BASE}/`,
  `${BASE}/news/`,
  `${BASE}/how-to-purchase/`,
  `${BASE}/news/avexroyalbrats_01/`,
  `${BASE}/bl/`,
  `${BASE}/sports/`,
];

export default function () {
  const url = PAGES[Math.floor(Math.random() * PAGES.length)];
  const res = http.get(url, {
    headers: { 'User-Agent': 'cheer-print-loadtest/1.0 (authorized)' },
  });
  check(res, {
    'status is 200': (r) => r.status === 200,
  });
  sleep(1); // 閲覧間隔。同じ VU 数でも sleep で RPS が変わる
}
```

実行例:

```bash
k6 run -e BASE_URL=https://www.cheer-print.com loadtest-cheer-print.js
```

## 9. 結果が出たあとの判断フロー

> **対象読者: 全員**

```text
ピークで「問題なし」
  → 当日は監視を中心に運用

ピーク前に遅延・エラー（要改善／危険）
  → まず原因を確認する
  → ページキャッシュを検討
  → CDN（画像などを別拠点から配る仕組み）を検討
  → 必要ならサーバーを一時的に大きくする（例: t3.small / t3.medium）
  → DB（データの置き場）が原因なら分離を検討

対策前は危険・対策後は問題なし
  → 「作り置き（キャッシュ）が効いた」など、効いた対策を説明できる
```

上記は **必ず全部をこの順で実施するリストではありません。**  
原因に応じて、制作会社と必要な対策を選びます。

**サーバー増強は「最後の手段」と決めつけない。**  
キャッシュや CDN とあわせて、イベント期間だけ一時的に大きくする方法は、実装リスクが比較的小さく料金差も限定的になりやすいです。制作会社と並行検討します。

## 付録. 用語の最短解説

> **対象読者: 必要時のみ**

| 用語 | 意味 |
|------|------|
| **Webサーバー** | ブラウザからの「ページを見せて」を受け、HTML 等を返す受付役（本件は Apache）。詳細は §0.2 |
| **静的サイト** | できあがったファイルをそのまま配る方式。速いが更新運用が重い |
| **動的サイト** | アクセスのたびにプログラム＋DB でページを組み立てる方式。更新しやすいがピークに弱い |
| **WordPress** | 動的 CMS。PHP で動く。Cheer Print がこれ |
| **同時アクセス** | 同じ時間帯にサイトを利用している人数のイメージ。1日の上限人数ではない（§2.0） |
| **のべアクセス／のべ訪問** | 期間中にサイトを見た合計イメージ。同時より大きくなるのが普通 |
| **VU（Virtual User）** | 負荷試験上で、決められた閲覧行動を同時進行する仮想利用者。実利用者数と1対1ではない（`sleep` で RPS が変わる） |
| **RPS** | 1 秒あたりのリクエスト数 |
| **p95** | 遅い側から見て上位 5% を除いた応答時間の目安 |
| **ページキャッシュ** | 一度作った HTML のコピーを配り、PHP/DB をほぼ休ませる仕組み（§0.2） |
| **CDN** | 画像などを近い拠点から配り、本店（EC2）への来店を減らす（§0.2） |
| **CPUクレジット** | t3 系が基準性能を超えて CPU を使う仕組み。Standard では枯渇すると性能制限、Unlimited では余剰利用により追加料金が発生する場合がある |
