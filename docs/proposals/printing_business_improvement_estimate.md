## 1. はじめに

いまお預かりしている現場は、**かなり整理されています**。受注の入口から、見積・売上台帳・月次請求、PDF、商品一覧（マスタ）までが、長年かけて **業務システムとして** 組み込まれている状態です。**現在のExcel／VBAは、これまでの業務を支えてきた重要な業務システム** であり、その評価を前提にしています。

一方、案件やファイルが増えるほど、「探す・転記・確認・集計」が増えやすい構造でもあります。今後を見据えると、**情報が自然につながる一本の仕組み** に載せ替えるフェーズが、現実的な選択肢になります。

本資料は、その **第一段階（核をつくる部分）に投資規模がどのくらいか** を、過不足なく伝えるための **概算（案・税別）** です。価格の押し付けではなく、**業務改善のために何を優先して、何をこの段階では含めないか** とセットで読むことを意図しています。

**補足（「ゼロから」とはどういうことか）:** 今回想定しているのは、**ブラウザで動く業務システムを、新しく一からつくる**ことです（いまお使いの表計算ファイルの中へ、少しずつ機能を足していくタイプではありません）。だから概算にも、要件の整理や画面・帳票の新規準備に相当する工数が含まれます。**ただし、いままでの運用や金額の考え方・帳票の役割は設計へ引きつぎとして活かす**ので、「仕事のやり方そのもの」を白紙から考え直す、という意味でのゼロスタートではありません。

## 2. 今回目指すこと

大きく、次の方向です。

**「人が頑張ってつなぐ仕組み」から、「情報が自然につながる仕組み」へ**

より具体的には、以下を自然なゴールとして置きます。

- **人を増やす前に構造を整える**（人手だけで増やそうとする前に、流れと情報の置き場をそろえる）  
- **一回の入力でつながる**（同じ内容を見積・売上・請求で繰り返し扱わない）  
- **誰がやっても基本的に回る**（手順と権限が、画面とルールで見える）  
- **探す時間を減らす**（案件や顧客から、状態と書類のたどりがちが短くなる）  
- **受注〜請求を一つの流れとして見る**（経営と現場が同じタイムラインを共有できる）

## 3. 初期開発の考え方

最初からすべてを「入りきりで」つくらないことを、はっきりお約束します。

- **やること:** いまバラバラになっている情報を、**一つの流れでつなぐ「核」** をつくる。  
- **やらないこと（このフェーズでは）:** 「完全自動」「すべての例外を機械が判断」「すべての帳票を現物と完全一致」までを、初期で抱え込まない。  

そのうえで、次もセットです。

| 前提 | 内容 |
|------|------|
| **現場を止めない** | いきなり全置き換えではなく、運用と並走しながら段階的に載せ替える発想です。 |
| **段階導入** | 最初は「つなぐ核」と「運用が回る下限」に絞り、あとから足せるようにします。 |
| **将来拡張前提** | あとから、再注文・顧客向け画面・データの書き出し・分析などを足しやすい土台だけ先に決めます。 |

## 4. 初期開発範囲

以下は **「分散した業務をつなぐ核」** のための機能イメージです。文言は現場語に寄せています。

### 顧客管理

- **顧客情報**（名称・連絡先など、取引に必要な土台）
- **履歴**（案件・見積・請求とつながる動きやすさ）
- **請求先**（納品先と分ける場合はヒアリングで整理）

### 案件管理

- **Web注文からの情報**（既存フォームとどう繋ぐかは確定ヒアリングが必要です）
- **ステータス管理**（受付・見積中・確定・請求 等、運用に合わせて定義）
- **案件の履歴**（いつ何があったかが追いやすい）

### 見積管理

- **見積の作成**（現在の見積の考え方に沿った明細の持ち方）
- **見積のPDF**（外に渡す体裁の作成）
- **明細の管理**（項目が増えても追える形）

### 売上管理

- **売上の一覧**（いつ・誰・いくら、が並ぶ）
- **月次でのまとめ**（いまフォルダを集めてやっているイメージの代替）
- **顧客別のまとまり**（顧単位で確認しやすい）

### 請求管理

- **月次請求の流れ**（締めの単位は運用確認が必要です）
- **インボイス（適格請求書）に向けた記載の土台**（番号や必要項目を「データとして持つ」）
- **請求書のPDF**

### 商品マスタ管理（ボディー等）

- **いまのボディーマスタ相当を、画面上の一覧へ**（量が多い前提で、検索しやすさを優先）
- **表ファイル（CSV等）でのまとめ取り込み**（価格改定・追加分の反映が楽になる）
- **メーカー・サイズ・色・商品コード** など、現場で使っている区分の整理

### 管理画面（社内向け）

- **社内向けの作業画面**
- **誰が何をできるかの整理**（閲覧と操作の線引き）
- **基本設定**（税率や会社側の共通情報など、運用ルールと合わせて定義）

> **ひとことで:** 「全部入り」の完成品ではなく、**まずは分散した情報が一本の線でつながる状態** が初期のゴールです。

## 5. 今回含めないもの（将来の拡張候補）

以下は **「やらない」ではなく、この段階では抱え込まず、後から広げられる候補** です。

| 将来の拡張候補 | イメージ |
|----------------|-----------|
| **メーカーの在庫とシステムの自動連携** | 連携できることが分かったら。当面は確認フローを残す運用でも十分検討可能です。 |
| **見積の完全自動のつもりでのAI運用** | 補助や下書き程度は将来あり得ますが、現場と税務を踏まえると **人の確認** が前提です。 |
| **会計ソフトとの完全自動のつもりでの連携** | データの書き出しから始め、信頼が積み上がってから詳細連携が現実的です。 |
| **すべての例外ルールを機械任せにする自動化** | 例外は画面と運用で受け、そのうち繰り返しが増えたものから自動化を検討します。 |
| **帳票の見た目の完全一致保証** | 既存PDFと並べながら近づけますが、「ピクセルまで同一」の保証は別コストになります。 |
| **経営会議にそのまま出せるような高度な分析だけの画面** | まずは一覧・集計・エクスポートを土台に、分析は段階的にです。 |

## 6. 概算費用

### 規模感の説明（なぜこのレンジか）

現在お預かりしている仕組みは、すでに **「見積～PDF～売上台帳～月次請求～商品一覧」まで含む業務システム** でした。その役割を、**Web上の新しいシステムとして一から用意し、機能を載せ替えていく**イメージです（表計算ツール側を段階的に厚くしていく改修とは別レイヤーの投資になります）。  
単純な「伝票入力アプリ」のつもりではなく、**複数ファイルにまたがっていた処理を一本化したうえでの置き換え** と捉えると、次のような塊になります。

- **運用や帳票の押さえ込み（要件・設計）** … 税や請求書の項目、運用例外のヒアリングに時間がかかりやすい。  
- **見積〜PDF** … 明細種類が複数あり、画面上と紙出力の両方が必要。  
- **売上・月次請求** … 「集める・まとめる」がいま自動化されていても、その代わりに **ロジックの再現と確認** が必要。  
- **商品マスタ** … 一覧の行数が多いタイプであり、検索・取り込みの両方が必要。  
- **管理画面・初期移行・調整** … 運用開始までの細かな手直しや、最初の並走期間での調整。  

それらを **一度にすべてを豪華に完璧にではなく、「核」を優先したうえでおよそ総額というイメージ** にすると、だいたい下表の桁になります。**税別・概算・前後することを前提としたレンジ** です。

### 内訳（案・税別）

| 項目 | 概算（税別） |
|------|----------------|
| 要件整理・設計（ヒアリング含む運用の決めごとを固める時間） | ¥160,000 |
| 顧客・案件管理 | ¥200,000 |
| 見積・PDF（明細の複雑さを含む） | ¥280,000 |
| 売上・請求（月次のまとまりまで） | ¥220,000 |
| 商品マスタ（一覧化・検索・表ファイルでの取込） | ¥180,000 |
| 管理画面（社内操作・誰が何をできるか・基本設定） | ¥100,000 |
| データ移行・初期調整（スタートまでの並走での手直しを含む） | ¥60,000 |

**合計（案）: ¥1,200,000（税別）前後**

- インボイスや帳票の細部、フォームとのつなぎ方などで増減します。増える場合、その理由は **機能の追加** だけでなく、**運用確認や例外の拾い込み** になることがあります（押し売りにならないよう、増減は都度明示します）。
- 「安そうに見える／高そうに見える」両方になりにくいよう、上記は **現在の業務システムの置き換え・統合** という視点での **中身付きの目安** です。

## 10. 補足・考え方

今回のシステム化は、**現在のExcel／VBAや、そこまで積み上げてきた業務運用を否定するものではありません**。むしろ、それらがあってこそ、「次は情報をどう繋ぐか」という話に進める段階に来ている、という認識です。

投資の趣旨も同じです。**安売り広告でも、高額サービスの煽りでもなく**、現場が毎回消耗している「探す・転記・確認・集計」を構造側で軽くし、**より安全で、説明しやすく、将来も手が入れやすい形** に整えるための規模感です。

まずやるべきことはひとことでいうと、

**分散した情報をつなぐ「核」をつくる**

ことです。その核があると、その後に再注文や顧客向け機能、データの活用などを、現場の様子を見ながら足していけます。
