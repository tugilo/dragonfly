## 1. はじめに

国宝社様は、創業160年の製本会社として、ジャンプ・マガジン等の製本を手がけながら、家系図・ルーツ調査という新規事業も展開されています。25年で89.2%が廃業する厳しい製本業界の中でV字回復を達成されており、**現場の運用力と経営の継続性**は大きな強みです。

一方、製本販売管理の基幹システム（VB + Oracle）は、長年の業務の中心として機能していますが、**出版社から届くPDF注文書の手入力**が事務負担の大きな部分を占めています。

本提案は、基幹システムをいきなり作り替える話でも、Google Workspace / GAS で小さく始める話でもありません。  
第2回121（2026-05-29）で合意したとおり、**社内ローカルサーバー上の Web システム**として、いまの業務と Oracle を活かしながら、最も負担の大きい入口だけを改善します。

## 2. 提案の結論（先に要点）

| 項目 | 提案 |
|------|------|
| **適切なシステム像** | 既存 VB + Oracle を温存した **「PDF注文書入力支援 Web」**（周辺システム） |
| **第1段階の目的** | 大口顧客の PC 作成 PDF 注文書について、**転記・手入力を減らす** |
| **運用の原則** | PDF解析 → **人が確認・修正** → Oracle 登録。100%自動化は目指さない |
| **インフラ** | **社内ローカルサーバー**（Hyper-V 上 Linux Web）。社内 Wi-Fi / VPN。外部公開なし |
| **方式合意（第2回121）** | Google Workspace / GAS 案 **不採用** → 社内 Web システムで合意 |
| **作り替えないもの** | 既存 VB 画面、Oracle テーブル構造、現場の業務フロー全体 |
| **投資判断の目安** | 人件費ベース ROI 試算で回収 **約1.0〜3.4年**（§9・要確認） |

## 3. 現状（As-Is）

### 3.1 業務の流れ（受注まわり）

```text
出版社・大口得意先
  ↓ PDF注文書（フォーマットは出版社ごとに異なる）
事務員が内容を確認
  ↓ 手入力
既存「製本販売管理システム」（VB + Oracle）
  ├ 受注入力（基本情報・詳細情報）
  ├ 入荷状況・見返入力・受注台帳 等
  └ 生産・その後工程へ
  ↓
ODBC 経由で Oracle データを Excel へ
  ↓
日報作成・分析・伝票出力 等
```

### 3.2 既存システムの位置づけ

- **開発:** 三菱関連会社製の VB + Oracle 基幹システム
- **役割:** 受注・生産・販売管理の **業務の心臓部**
- **データ:** 受注ヘッダー **TTJUCUHD** と明細 **TTJUCUDT（Excel上 TTJUCHDT）** の 1:N 構造
- **強み:** 現場に根ざした画面・項目・工程管理が、長年の運用で定着している

### 3.3 受領資料から確認できた構造

2026-07-05 受領の DB 定義・Excel サンプル・注文書 PDF より、以下が確認できています。

**ヘッダー（TTJUCUHD）の主な項目例**

- 得意先・管理No（`TOKUICOD`, `KANRISEQ`）
- 製品情報（`SEIHINCD`, `SEISIKINM`, `SEIHINNM`, `ISBN`, `CHUBUSU` 等）
- 判型・サイズ（`HANKEICD`, `TENCSIZE`, `SAYUSIZE` 等）
- 各種日付（発売・見本・配本・揃い・製本：`HATYMD`, `MIHYMD`, `HAIYMD`, `SORYMD`, `NOUYMD` 等）
- 丁数・平台・輪転関連（`HONH32DS`, `HONR16DS` 等多数）
- 備考（`JUHDBIK1`, `JUHDBIK2`）

**明細（TTJUCUDT）の主な項目例**

- 種別（`TUKIMCOD`：表紙・本文・見返・カバー等）
- 印刷所・加工所（`INSATUCD`, `KAKOUJCD`）
- 加工内容・備考（`KAKOUNCD`, `JUDTBIKO`）
- 入荷関連（`NYUYTYMD`, `NYUKNYMD`, `NYUKASUG` 等）

**サンプルデータの特徴**

- 講談社系書籍（例: 深い河、午後の脅迫者 等）の実データ形式
- 備考欄に「FP4/15納品」「カバー→DCより」「定価改定あり」等、現場判断が必要な情報が混在

## 4. 表面の要望と、裏にあるお困りごと

### 4.1 表面の要望（121で明示されたこと）

| # | 要望 |
|---|------|
| 1 | PDF注文書の手入力を減らしたい |
| 2 | 既存 VB + Oracle は温存したい |
| 3 | Oracle DB へ安全に登録したい |
| 4 | **社内ローカルサーバー上の Web システム**で展開（Google Workspace / GAS 案は不採用） |
| 5 | 段階的に拡張したい |

### 4.2 裏にあるお困りごと（121・資料・現場構造から読み取れること）

| お困りごと | 説明 |
|------------|------|
| **入力が増えるほど、人が足りなくなる** | 大口顧客だけで月300〜400件、全体ではその倍近い。件数増に比例して事務負担が増える構造 |
| **同じ情報を、PDFと画面の間で二度扱う** | 注文書 → 目視 → VB画面入力。転記ミス・見落としの余地が常にある |
| **フォーマットが出版社ごとに違う** | 1社1様式。ルールを覚えている担当者への依存が強い |
| **項目が多く、判断も必要** | ヘッダー100項目超・明細も複数行。備考に現場知識（紙種・加工・納期調整）が入る |
| **基幹システムを触れない不安** | 三菱関連会社製・保守体制・Oracle の採番/トリガー。全面改修はリスクが大きい |
| **IT刷新とセキュリティの両立** | クラウドやGASに機密データを出したくない。一方で社員個人のChatGPT利用（シャドーAI）への懸念も |
| **老舗企業としての継続性** | 160年の事業を守りつつ、新規事業（家系図）も伸ばしたい。製本の基盤が止まると全体に波及 |
| **将来の人材・承継** | 属人化した入力ノウハウは、担当変更・繁忙期・後継者育成のボトルネックになりうる |

### 4.3 要望の奥にある経営テーマ

- **「止まらない製本基盤」** … 入力の壁を下げ、受注処理のスループットを上げる
- **「触らないで改善する」** … 現場の慣れを壊さず、周辺から効率化する
- **「段階的なDX」** … いきなり全刷新ではなく、効果の大きい入口から積み上げる
- **「社内で完結する安心」** … データ主権を社内に保ちつつ、Webの使いやすさを得る

## 5. 現在の問題点（整理）

### 5.1 業務上の問題

| 問題 | 影響 |
|------|------|
| PDF → 手入力の繰り返し | 時間コスト大。月数百件規模で積み上がる |
| 出版社ごとのフォーマット差 | 担当者の経験値に依存。教育・引き継ぎコスト |
| ヘッダー・明細の複雑さ | 入力漏れ・誤入力のリスク。確認・修正に時間 |
| 手書き注文書の存在 | PC作成PDFと別ルート。初期自動化の対象外にすべき |
| 繁忙期の集中 | 納期前後に入力が集中し、ボトルネック化 |

### 5.2 システム上の問題

| 問題 | 影響 |
|------|------|
| VB スタンドアローン UI | モダンな Web UX・多端末・外部連携がしにくい |
| Oracle 直接改修のハードル | 保守会社・トリガー・採番との整合が必要 |
| Excel ODBC 依存の二次加工 | 日報・分析は可能だが、入口（入力）の手間は残る |
| 入力と基幹が一体 | 改善のたびに「基幹を触る」判断が必要になり、動きにくい |

### 5.3 このまま続けた場合のリスク

- 受注件数が増えるほど、**事務入力が成長の天井**になる
- 担当者依存が残り、**休暇・退職・繁忙期に詰まる**
- 入力ミスから生じる **工程遅延・差し戻し** のコスト
- 「いつかシステムを…」が先延ばしになり、**改善のタイミングを逃す**
- シャドーAIで個人が試行錯誤 → **セキュリティと品質の両面リスク**

## 6. 適切なシステムの方向性

### 6.1 推奨するアプローチ：**入力支援レイヤー**

```text
┌─────────────────────────────────────────┐
│  新規: PDF注文書入力支援 Web（社内）      │
│  ・PDFアップロード / 解析 / 確認・修正    │
│  ・既存画面に近い項目で表示              │
│  ・確認後、Oracle TTJUCUHD/DT へ登録     │
└─────────────────┬───────────────────────┘
                  │ 書き込み（既存ルール順守）
                  ▼
┌─────────────────────────────────────────┐
│  既存: VB + Oracle 製本販売管理システム   │
│  （温存・直接改修しない）                 │
└─────────────────┬───────────────────────┘
                  │ ODBC 等（現状維持）
                  ▼
┌─────────────────────────────────────────┐
│  Excel 日報・分析・伝票出力 等            │
└─────────────────────────────────────────┘
```

### 6.2 なぜ「全面リプレイス」ではないか

| 選択肢 | 判断 |
|--------|------|
| 基幹システム全面刷新 | リスク大・コスト大・現場停止リスク。今回の優先課題（PDF入力）からは距離がある |
| Google Workspace / GAS 案 | **不採用**（第2回121合意）。外部アクセス・セキュリティの懸念。当日は話題提示のみで、結論は社内 Web |
| 既存 VB に機能追加 | 保守会社・ソース・テスト環境の制約。改修コストが読みにくい |
| **周辺 Web + Oracle 連携（社内ローカル展開）** | **第2回121で合意。Hyper-V 上 Linux Web、社内ネットワーク完結** |

### 6.3 なぜ「100%自動化」ではないか

- 出版社ごとに PDF レイアウトが異なる
- 備考欄に現場判断（紙種・加工・納期調整）が入る
- Oracle 登録は採番・必須項目・トリガーとの整合が必要
- **人の確認を挟む**ことで、品質と安心を両立する

## 7. 改善案（段階的ロードマップ）

### Phase 1 — PDF注文書入力支援（最優先・本提案の核）

**対象:** 大口得意先の PC 作成 PDF 注文書（初期は1〜数パターンのフォーマット）

| 機能 | 内容 |
|------|------|
| PDF アップロード | ドラッグ&ドrop、複数ファイル対応 |
| PDF 解析 | ヘッダー・明細項目の抽出（出版社フォーマット別ルール） |
| 確認・修正画面 | 既存 VB 画面に近い項目名で表示。低信頼項目をハイライト |
| Oracle 登録 | TTJUCUHD / TTJUCUDT へ、採番・必須項目を守って書き込み |
| 結果表示 | 管理No・登録結果・エラー内容 |
| フォールバック | 解析失敗時は従来どおり VB 画面で入力可能 |

**期待効果**

- 1件あたりの入力作業時間の短縮（**§9 ROI 試算:** 年210〜700時間削減見込み、要確認）
- 転記ミスの削減
- 入力ノウハウの一部をシステム化（フォーマット別ルール）

### Phase 2 — 社内業務の Web 化・可視化

| 機能 | 内容 |
|------|------|
| 日報・集計の Web 参照 | ODBC/Oracle 読み取りベースのダッシュボード |
| Excel 帳票の Web 化 | 手作業 Excel 加工の一部を Web 画面へ |
| 多端末対応 | 社内 Wi-Fi / VPN 経由で PC・タブレットから利用 |

### Phase 3 — 外部連携・将来展開

| 機能 | 内容 |
|------|------|
| 得意先直接入力 | 出版社・大口顧客が Web フォームで注文（初期スコープ外） |
| 追加フォーマット | 手書き OCR、新出版社フォーマット |
| 家系図事業との横展開 | 調査案件管理・顧客資料管理（別テーマとして検討） |
| AI ガバナンス | 社内利用ガイドライン整備、シャドーAI抑制 |

## 9. 投資対効果（ROI）— 人件費ベース試算

Phase 1 の投資判断には、**初期費用だけでなく、人件費換算の削減効果と回収期間** を示すのが有効です。  
本案件は **事務員の PDF → 手入力時間** が主コストのため、次の式で試算します。

```text
年間削減効果（円）＝ 削減時間（時間/年）× 人件費単価（円/時間）
投資回収期間（年）  ＝ 初期投資（円）÷ 年間削減効果（円）
```

**試算の主軸:** 直接の **手入力時間** の削減。転記ミス低減・繁忙期の残業抑制は **定性のプラスα** として補足する。

### 9.1 試算前提（121・受領資料ベース）

| 項目 | 値 | 備考 |
|------|-----|------|
| 対象件数 | 月350件 | 大口顧客（121より）。全体件数は要確認 |
| 1件あたり入力時間 | **要確認** | 試算では 10 / 15 / 20 分の3段階 |
| 削減率 | **30〜50%（試算）** | 解析＋確認運用。保証しない |
| 人件費単価 | **要確認** | 不明時は ¥2,000〜3,000/h の参考レンジ |
| 初期投資（比較対象） | パッケージ B **約180万円** | [概算見積](kimura_kokuhosha_pdf_order_input_estimate.md) 参照 |

### 9.2 3シナリオ試算（削減時間）

| シナリオ | 1件時間 | 削減率 | 月間削減 | 年間削減 |
|----------|---------|--------|----------|----------|
| **保守** | 10分 | 30% | 17.5時間 | **210時間** |
| **標準** | 15分 | 40% | 35時間 | **420時間** |
| **楽観** | 20分 | 50% | 58時間 | **700時間** |

### 9.3 年間効果・回収期間（@¥2,500/h・パッケージ B）

| シナリオ | 年間効果 | 回収期間 |

**投資回収の目安:** 約 **1.0〜3.4年**（1件時間・削減率・単価次第）。  
**〜2年** であれば導入検討しやすく、**3年超** の場合は Step 0 検証で前提を実測確認するのがよい。

## 10. 提案するシステム構成（Phase 1）

### 10.1 インフラ

| 要素 | 方針 |
|------|------|
| Web サーバー | 既存 Windows サーバーの **Hyper-V** 上に Linux |
| アクセス | 社内 Wi-Fi / 有線 LAN。社外は **既存 VPN** |
| 外部公開 | **しない**（新規外部回線不要） |
| DB 接続 | Oracle へ直接 or ODBC 書き込み（要確認） |

### 10.2 アプリケーション構成（案）

```text
[ブラウザ]
    ↓ HTTPS（社内）
[入力支援 Web アプリ]
    ├ PDF 解析エンジン（フォーマット別）
    ├ 確認・修正 UI（ヘッダー / 明細）
    ├ 登録前バリデーション
    └ Oracle 書き込みモジュール
    ↓
[Oracle DB]
    TTJUCUHD（ヘッダー）
    TTJUCUDT（明細）
```

### 10.3 画面イメージ（簡易モックで見せる流れ）

1. PDF をドラッグ&ドrop
2. 解析結果を **基本情報タブ / 詳細情報タブ** に分けて表示
3. 信頼度の低い項目・未マッピング項目を色分け
4. 修正後、「登録確認」画面で最終チェック
5. Oracle 登録 → 管理No 表示
6. エラー時は理由を表示し、VB 画面への誘導

### 含めない（Phase 1）

- 既存 VB システムの改修・リプレイス
- Oracle テーブル構造の変更
- 手書き注文書の OCR
- 得意先向け直接入力ポータル
- 外部クラウド SaaS / **Google Workspace / GAS**（第2回121で不採用）

---

## 12. 進め方（案）

| ステップ | 内容 | 期間目安 |
|----------|------|----------|
| **0. 現状整理** | 受領資料の精査、追加 PDF サンプル取得、書き込み権限確認 | 1〜2週 |
| **1. 簡易モック** | 画面イメージ・操作フローの共有（無償） | 合意後 数日 |
| **2. 詳細設計** | PDF→Oracle マッピング、採番・必須項目の確定 | 2〜4週 |
| **3. PoC / パイロット** | 1フォーマット・限定件数で試運用 | 1〜2ヶ月 |
| **4. 本番展開** | 対象出版社の拡大、運用マニュアル・教育 | 段階的 |

**重要:** Phase 1 完了後に効果測定（入力時間・件数・エラー率）を行い、Phase 2 以降の判断材料とする。

## 15. おわりに

国宝社様の要望は「新しいシステムが欲しい」ではなく、**160年続く製本事業の入口を、無理なく軽くしたい**——その意図だと理解しています。

全面刷新ではなく、**いま動いている基幹を守りながら、最も痛いところから改善する**。  
それが、121での合意・受領資料の構造・現場の安心のすべてに沿った、最も適切な第一歩だと考えます。

簡易モックを通じて、実際の操作感と実現イメージを共有させてください。
