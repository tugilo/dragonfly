- そのターゲットについて、**LINE で実際に来るメッセージのトップ3** は何だと思いますか。（「料金」「空き」「場所」「キャンセル」など、思いつくものを3つ）
- **絶対に AI に任せたくない内容** は、軍司さんの経験だと何ですか。1〜2例で構いません。

### ③ STARTスコープの具体化

#### 目的

**START で「作る」と言ったときの中身** を、AI・人・ログの三本で言語化し、見積りと実装のブレを減らす。

#### 話す内容

- START は「止血＋検証」であり、GROW / SHIFT は後続であること。
- 三本柱：**AIが対応する範囲**／**人へ渡す条件（エスカレーション）**／**ログ取得の粒度**。

#### 3-1. AIが対応する範囲

##### 目的

「AI に任せてよい業務境界」を一度は言葉にし、後から「あれもこれも」になりにくくする。

##### 話す内容

- 一次応答（案内・FAQ 的説明・次の一手への誘導）と、**クロージングや契約まで** は分ける。
- ナレッジの出所（軍司さん・クライアント・tugilo での整理分担のイメージだけでも）。

##### 具体質問

- START では、AI に主に任せたいのは次のどれを中心にしますか。優先順位を2つまで。  
  **A)** 営業時間外・初回の「反応が返ること」  
  **B)** よくある質問の説明（言い換えにも弱くないこと）  
  **C)** 予約・問い合わせフォームや担当への受け渡し前の整理  
  **D)** まずは対話の雰囲気・ブランドトーンの統一  
  **E)** その他（業務に即して）
- **AI の返答に必ず含めたい一文や禁止したいこと**（例：断り方、免責、特定の言い回し）はありますか。なければ「なしで進める」でも確定です。

#### 3-2. 人へ渡す条件（エスカレーション）

##### 目的

クレーム・法務・濃厚な商談を AI が抱え込まない線を引く。

##### 話す内容

- キーワード・感情が強い文・「予約変更」「返金」「契約」など、**ルールで人に渡す**考え方。
- 有人へ渡すときに **会話の要約を添える** 案（負担軽減）の可否。

##### 具体質問

- 人に必ず渡したい状況を、**具体的な言い方** で2つ挙げてください。（例：「キャンセル」「クレーム」「個人情報の変更」）
- 有人へ渡すとき、**理想はクライアントの誰が受ける** 想定ですか。  
  **A)** 代表・オーナー本人  
  **B)** 受付担当  
  **C)** 軍司さん側の運用サポート  
  **D)** 未定だが「この役割」と決めたい
- AI が迷ったときの一言は、次のどのトーンが合いますか。  
  **A)** 「担当が確認して折り返します」  
  **B)** 「本日○時までにご連絡します」（時間の約束は別途ルール）  
  **C)** 「詳細はこちら（フォーム/電話）」への誘導のみ

#### 3-3. ログ取得の粒度

##### 目的

改善と再現性のため、**どこまで記録するか** を START で決め、後の GROW につなぐ。

##### 話す内容

- 記録するもの：生の会話／要約／タグ（例：「料金」「予約」「クレーム」）／有人に渡したフラグなど。
- 個人情報・保管期間は別途必要なら「次の論点」とする。

##### 具体質問

- START 時点で、**クライアントまたは軍司さんが後から見たい集計** は何ですか。次から1〜2個。  
  **A)** どんな相談が多かったか（大分類でよい）  
  **B)** AI が答えた／人に渡したの件数比  
  **C)** 時間帯別・曜日別の混み具合  
  **D)** まずは会話の保存だけでよい  
  **E)** その他
- ログは **誰がどの画面や形で触る** 想定に近いですか。  
  **A)** tugilo が export / 簡易レポート  
  **B)** スプレッドシート連携を今後見据える  
  **C)** クライアントには要約だけ見せる  
  **D)** まだ決めなくてよい

### ④ やらないことの明確化

#### 目的

スコープ漏れと期待値ズレを減らし、**初事例を無事終える**ための「線」を引く。

#### 話す内容

- START で「やらない」と決めると、工数・リスク・説明が楽になる。
- 後から GROW / SHIFT で足せることは、あえて最初に入れない。

#### 具体質問

- START の段階では、次のうち **意図的にやらない** ものを選んでください（複数可）。  
  **A)** 予約システム・カレンダーとの自動連携  
  **B)** 顧客DB・会員システムとの突合  
  **C)** 音声・電話連携  
  **D)** 完全自動で成約まで  
  **E)** 多言語対応  
  **F)** 全部入れたい（その場合、優先を1つに絞る話に切り替える）
- 軍司さんからクライアントへ説明するとき、**「やらない」と言うと安心する項目** はありますか。
- tugilo 側で想定している「やらない」例（例：違法・過度な医療助言など）に、**業種特有の禁止領域** はありますか。

### ⑤ 横展開を前提とした設計

#### 目的

1件目で捨てず、**2件目以降が楽になる「型」** を二人で言語化する。

#### 話す内容

- **共通化する部分**：Webhook の受け方、AI の出し方、エスカレーションの型、ログの枠、運用チェックリストの素案。
- **個別化する部分**：Lステップのシナリオ、業種ごとのトーンと禁止事項、ナレッジ、有人の受け皿。
- 軍司さんのビジネスとして「同型で何社回せるか」の感覚すり合わせ（数値必須ではない）。

#### 具体質問

- 横に広げるとき、**すべての案件で同じにしたいもの** を3つまで挙げてください。（技術・運用・営業のどれでも可）
- **案件ごとに必ず変えるもの** は何ですか。例：業種別マニュアル、挨拶文、エスカレーション先、など具体で2つ以上。
- 2件目以降、**軍司さんが毎回やる作業** と **tugilo が毎回やる作業** の分け方で、理想に近いのはどれですか。  
  **A)** 軍司さん：導線・シナリオ／tugilo：裏側一式  
  **B)** 軍司さん：ヒアリング＋一次文面／tugilo：実装＋調整  
  **C)** クライアントによって変える  
  **D)** 今日はラフでよいので方向だけ
- 「型」にするとき、**プロダクト名やパッケージ名を付ける** 必要はありますか。  
  **A)** 今は不要  
  **B)** 社外説明用に欲しい  
  **C)** 内部コードだけでよい

### ⑥ 価格と進め方の方向性

#### 目的

提案書のレンジ（START / GROW / SHIFT）と照らし、**初事例の見え方と利益配分の素地** を話す。ここで確定しなくてよいが、**ズレがあれば早期に認識する**。

#### 話す内容

- START は「検証＋止血」、成果が見えたら GROW の話に自然に接続する。
- 軍司さん・tugilo・クライアントの **提案の持ち方**（共同提案／紹介／再販など）はケースによる。

#### 具体質問

- 第1事例は、**クライアントへの提示価格の主軸** はどれに近いですか。  
  **A)** 軍司さん見積に AI 部分をのせる  
  **B)** tugilo 見積を分けて提示  
  **C)** パッケージ一括  
  **D)** まだ決めないが、今日の時点の仮説を共有したい
- START の **成果の見方** を1つ決めるなら何にしますか。  
  **A)** 返信までの時間短縮  
  **B)** 有人対応件数の減少  
  **C)** 問い合わせ取りこぼしの減少  
  **D)** 顧客満足の声  
  **E)** その他
- 着手までに、**軍司さん側で必須になる準備**（アカウント・シナリオ案・クライアント合意など）を、手順で2〜3ステップに分けると何になりますか。

### ⑦ クロージング（次アクション）

#### 目的

**誰が・いつまでに・何をするか** を少なくとも1本、必ず残す。

#### 話す内容

- 今日決まったこと／宿題／次回の議題を30秒で要約して口頭確認する。
- 次回の有無・候補日は軽く触れる。

#### 具体質問

- 今日の時点で **確定でいいこと** を、箇条書きで一緒に言い換えます。軍司さん側から1つ足りないものはありますか。
- **次に軍司さんがやる1番目の作業** は何ですか。期限の目安は「○日まで」「○週以内」のどちらで置きますか。
- **次に tugilo（次廣）がやる1番目の作業** は何にしますか。（例：START 条件メモの共有／ヒアリング項目の清書／架空シナリオのたたき台、など具体を一緒に置く）
- **次回打ち合わせ** は必要ですか。  
  **A)** 要る（候補日程を今決める）  
  **B)** 不要、まずは非同期で十分  
  **C)** クライアント候補が動き次第
