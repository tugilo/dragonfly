## 結論

木村さんからの中心要望は、既存の **VB + Oracle 製本販売管理システム**をすぐに作り替えることではなく、現場で負担が大きい **PDF注文書の手入力を減らすこと**。

最初の提案は、既存システムを温存しながら、社内ネットワーク上の Web 画面で PDF を読み取り、確認・修正後に Oracle へ登録する **入力補助システム** として設計するのがよい。

## 木村さん側の要望

### 1. PDF注文書の手入力を減らしたい

- 大口顧客から届く PDF 注文書を、事務員が既存システムへ手入力している。
- 対象件数は大口顧客だけで月間 **300-400件**。全体ではその倍近い件数がある可能性がある。
- まずは PC 作成の PDF 注文書を対象にし、手書き注文書は初期対象から外すか後回しにする。
- PDF をアップロードし、解析結果を人が確認・修正してから登録できる画面がほしい。

### 2. 既存 VB + Oracle システムは温存したい

- 現在の基幹システムは、三菱関連会社が開発した **VB + Oracle** の製本販売管理システム。
- 受注入力、入荷状況、見返入力、受注台帳作成入力、製品問合せなど、現場業務の中心になっている。
- 既存システムに直接手を入れるよりも、周辺に Web インターフェースを追加して補助する方が現実的。
- 既存画面・既存DB・既存業務フローを壊さないことが重要。

### 3. Oracle DBへ安全に登録したい

- PDF 解析結果は、最終的に Oracle DB に登録する必要がある。
- テーブル定義、キー項目、英字項目名、画面項目との対応関係を確認したうえで設計する。
- シーケンス番号、トリガー、既存 VB 側の前提を壊さない書き込み方法を確認する。
- ODBC 経由で書き込みできるか、別接続方式が必要かは確認待ち。

### 4. 社内完結型で運用したい

- Google Workspace / GAS は、外部アクセスやセキュリティ面の懸念があり、今回は主軸にしない。
- 既存 Windows サーバー上の **Hyper-V** に Linux Web サーバーを立てる構成を検討する。
- PC やスマートフォンから社内 Wi-Fi 経由でアクセスできる構成が望ましい。
- 外部公開はせず、社外からは既存 VPN を利用する想定。

### 5. 段階的に拡張したい

- 第1段階は、最も負担が大きい PDF注文書の入力自動化。
- 第2段階以降で、日報などの Excel 帳票の Web化、多端末対応を検討する。
- 将来的には、得意先が直接入力できる仕組みも可能性としてあるが、初期スコープには入れない。

## 初期提案スコープ

### 含める

- PC 作成 PDF 注文書のアップロード
- PDF から注文情報を抽出する処理
- 抽出結果の確認・修正画面
- 既存 Oracle 項目へのマッピング設計
- 人が確認した後の Oracle 登録
- 社内ネットワーク限定の Web 画面
- Hyper-V 上 Linux Web サーバー構成案
- エラー時に既存業務へ戻れる運用

### 含めない

- 既存 VB システムの全面リプレイス
- Oracle DB の構造変更
- 手書き注文書の完全自動認識
- 得意先直接入力システム
- 外部公開されたクラウドサービス
- Google Workspace / GAS を主軸にした実装

## 簡易モックで見せたい流れ

1. PDF注文書をドラッグ&ドロップする
2. PDF 解析結果を注文ヘッダーと明細に分けて表示する
3. 既存画面に近い項目名で確認・修正する
4. 不明項目・低信頼項目を目立たせる
5. 登録前に確認画面を表示する
6. Oracle 登録後、管理No または登録結果を表示する
