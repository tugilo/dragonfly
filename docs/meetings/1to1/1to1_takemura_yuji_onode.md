- **平常時の体感:** WordPress にしてはかなり軽い（単発 TTFB おおよそ 0.24〜0.45 秒）。**今軽い≠ピーク耐性あり**。まず制作会社へ構成確認→簡易診断で要否判断。

#### 話した内容（重要）

##### 1. 相談の背景

##### 3. 懸念（WordPress の動的実行）

アクセスのたびに PHP が実行される前提での心配事項:

- PHP 実行数の増加
- MySQL へのアクセス増加
- メモリ不足
- レスポンス低下
- サーバーダウン

##### 4. 竹村さんが考えていた案

- PHP を毎回実行しない
- HTML を返すような仕組みにできないか
- フロント側だけで対策できないか

→ 方向性としては **ページキャッシュ（＋CDN）** が最も近い。完全に「フロントだけ」で EC2 原点の動的負荷をゼロにするのは難しく、キャッシュ／CDN／オリジン設定の組み合わせが現実的、という整理。

##### 5. 対策候補（優先度つき）

**優先度: 高**

| 対策 | 狙い |
|------|------|
| **ページキャッシュ導入** | 一度生成した HTML を配信し、PHP 実行回数を削減 |
| **CloudFront 等の CDN** | 画像・CSS・JavaScript をエッジ配信し、EC2 へのアクセスを軽減 |

**状況次第**

- Web サーバーキャッシュ（Nginx／Apache 側）
- PHP-FPM 設定
- OPcache
- WordPress プラグイン見直し（重い・重複キャッシュ等）

**必要なら**

- EC2 スペックアップ
- 負荷分散（ALB 等）
- DB 分離（RDS 等）
