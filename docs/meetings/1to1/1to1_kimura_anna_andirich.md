## ■ 事業理解（株式会社Andirich）

### 何を提供しているか

木村さんの仕事は、単なるツール制作ではなく、**現場の事務作業・売上管理・顧客管理・現場管理の流れを聞き取り、ムリ・ムダが少ない業務オペレーションに作り直すこと**。

特に建設業では、元請け・下請けとの契約、見積、請求書、現場スケジュール、人員配置など、管理対象が多く、既存ツールでは業界独自の運用に合わないことが多い。そのため、Google Workspace や Kintone を使って、顧客ごとにフルオーダーメイドで管理ツールを作る。

### 強みの理解

| 強み | 読み取り |
|------|----------|
| **かゆいトコロに手が届く** | 現場の小さな困りごとや、PCが苦手な人のつまずきまで拾える |
| **フルオーダーメイド** | 既存ツール導入ではなく、現場のやり方に合わせて業務設計する |
| **納品後サポート** | 作って終わりではなく、使い始めてからの「やっぱりこうしたい」に対応する |
| **Google / GAS / Kintone** | Excel・スプレッドシート限界層に対して、開発より軽い改善を提供できる |
| **事務代行・オンライン秘書** | ツールだけでなく、運用に入り込んで実務を支えられる |

### 顧客像

- 建設業の経営者、管理者、事務担当。
- 元請け・下請けとの契約、見積、請求、現場スケジュール、人員配置を整理したい会社。
- Excelやスプレッドシートで管理しているが、関数・手入力・転記・ファイル乱立が限界になっている会社。
- 既存の施工管理ツールや汎用SaaSでは、自社独自の運用に合わない会社。
- PCやITが得意ではなく、使い始めの伴走を必要としている会社。

#### 出発点として疑問を持つもの：補助金ありき

**「IT導入補助金で利用できるようにするため」** を主目的にWeb化を進めるのは、出発点として間違っているのではないか、という見方。

- 補助金により導入コストを下げられる点は、**お客様にとってのメリット** としてはある。
- ただしそれは **Web化そのものの必然性** ではなく、**費用面の追い風** にすぎない。
- 期限があるから進める、と、プロダクト・業務上の限界から進める、は別問題。
- 次廣は、補助金ありきで簡単なシステム開発だけを求められる案件は、これまで基本的に断ってきた。杏那さん案件も同じ基準で判断する。

#### 出発点として賛同できるもの：スプレッドシートの限界

一方、次のような **業務・プロダクト上の理由** からWeb化するなら、次廣は前向きに検討したい。

- スプレッドシートでの案件・売上・在庫・書類・日報管理に **限界が出てきている**
- **UI上の制限が多すぎる**（入力・閲覧・権限・現場での使い勝手）
- パッケージの提供力・拡張性を上げるために、クラウド化が **自然な次の一手** である

この出発点なら、桜井シーリング様向けシステムと同等（部分的にはそれより簡単）な内容のWeb化は、技術的にも事業的にも筋が通る。

#### 杏那さんの本業との関係

杏那さんの強みは、エンジニアとして大規模開発することではなく、

- 建設業の現場に寄り添った業務整理
- Google Workspace / Kintone / スプレッドシートでの管理ツール制作
- PCが苦手な人への伴走
- 事務代行・オンライン秘書

にある。121でも、自社はエンジニア会社ではないと説明していた。

Web化してクラウドサービス化すると、

- 初期開発費（120万〜200万円規模の可能性）
- 保守・運用（月額30,000円前提のたたき台）
- 商品設計・価格設計・サポート体制

が必要になり、**事業モデルが「業務改善パートナー」から「SaaS / プロダクト事業」に近づく**。

補助金起点か、スプレッドシート限界起点かを確認しないまま進めるのは避けたい。現時点では、121で出た文脈が **どちらに寄っているか** は未確認。
