# Lステップ Webhook を活用した AI チャットボット構築のご提案

**提出：** tugilo（次廣）  
**想定読者：** 非エンジニアの経営者・担当者の方  
**伴走のスタンス：** 仕様を押し付けるのではなく、**現場と経営の両方に効く形まで一緒に落とし込みます。** 営業トークではなく、実装可能性と次の一手の整理が目的です。

---

## 1. 背景と目的

**LINE で問い合わせが来るのに、担当が手が回らず返信が遅れる。** 同じ説明を何度も打っている。**夜間・休日の一言に営業機会が流れている**——こうした「現場のしわ寄せ」は、少人数経営ほどリアルです。

その先に置きたいのは、**「売りの導線は軍司さん側で活きて、そのあとの受付・記録・フォローまでが社内で途切れない状態」** です。  
本稿は、その **入口となる仕組み** を、技術的に **実現できる** ことと、**どう広がるか** を一枚にまとめたものです。

### 今回の相談内容

1to1 の場で、軍司さんから「Lステップの **Webhook** を使って、AI で会話に応答する仕組みはつくれるのか？」というテーマが上がりました。

### 本提案の目的（単なるチャットボットではない）

「LINE 上で便利な返答ができるアプリを1つ足す」のがゴールではありません。

- **今回：** Webhook と AI を組み合わせたとき、**何ができて、どこまでが現実的か** をはっきりさせる。
- **今後：** 予約・顧客情報・社内記録につなげ、**売る仕組みと回る仕組みが両方そろう** 道筋を共有する。

進め方は **小さく作って育てる** こと。**いきなり業務を丸ごと変えずに、仕組みを足していく** こと。ここは tugilo の前提として変わりません。

---

## 2. 結論（できるかどうか）

### Webhook で AI チャットは実現可能

**結論：実現可能です。** すでに類似の積み上げ方でプロダクト・業務支援を進めており、この組み合わせは **設計次第で十分に現実的** と判断しています。

Lステップの **Webhook** は「こんなメッセージが来た」と外部へ知らせる窓口です。  
ここにつなぎ、tugilo 側の **API サーバー** で AI に渡して返答文をつくる。必要なら **DB** を参照・記録して LINE に返す——この流れを設計します。

### ただし設計が重要

「つなげば動く」だけでは足りません。

- 誤案内や、有人が守るべき案件の取りこぼし
- ログが残らず、改善の判断ができない

最初に決めるのは、**どこまで AI が答え、どこから人に渡すか**。**何を記録し、経営・現場のどちらの判断に効かせるか**。ここが分かれ目です。

---

## 3. 実現構成（シンプルに）

専門用語は最小限にしています。

```
お客様のスマホ（LINE）
        │
        ▼
LINE 公式 ＋ Lステップ（受付・シナリオの土台）
        │
        │  Webhook「こんな会話が来た」
        ▼
tugilo の API サーバー（中継・ルール・ログ）
        │
        ├── AI（応答の草案）
        └── 自社 DB（必要なら参照・記録）
        │
        ▼
Lステップ／LINE 経由でお客様へ返信
```

お客様から見れば「LINE で話している」だけ。裏で Lステップ → tugilo → AI（＋DB）が動きます。  
要件の細部は、**導入後も伴走しながら** 詰めれば十分です。

## 4. 今回の提案内容

### まずお伝えしたいこと（誤解を防ぐために）

これは **ただの自動返信ツールではありません。**

- **業務改善・売上に効く入口** です。問い合わせの初動が速くなり、取りこぼしと説明の重複を減らすことは、**機会損失と人件費の両面で経営に効きます。**
- **人を減らすための仕組みではありません。** **判断と返信の負荷を下げ、担当者が「考えるべき案件」に時間を残す** ための設計です。
- ルールの外は **人に渡す（エスカレーション）** 前提です。**「AI で楽にしつつ、大事なところは人が守る」** 形にします。

### AI チャットボットの基本機能（イメージ）

- 内容を読み取り、**自然な文章で返す**
- トーン・禁止表現・会社の約束ごとに合わせる
- **営業時間外** も、ルールに沿った一次対応ができる

### FAQ だけにしない理由

固定 FAQ は「言い換え」に弱いです。AI は **意図を汲んで答えやすく**、会話の途中から深掘りにもつなげられます（※すべてを AI に任せる設計にはしません）。

### エスカレーション設計（人に渡す線を引く）

- 「予約変更」「料金・契約」「クレーム」など、**条件が揃えば有人へ**
- 自信がないときは **「担当が確認します」** へ
- 有人へ渡すとき、**直前の会話の要約** を添える——引き継ぎコストを落とします

## 5. 特徴・強み

### シナリオだけではない「AI の判断」の置きどころ

枝分かれシナリオだけだと、想定外の一言で止まります。AI は **言い換えと意図読取** に強い。**正確性が命の領域はルールで縛る**——この両輪が、tugilo では強みです。

### データが残り、改善の材料になる

会話ログ・要約・タグが残ると、**何に時間が消えているか・どこで迷わせているか** が見えます。**一回きりの施策で終わらず、小さく作って育てる** 材料になります。

### 業務への接続（業務を変えずに、仕組みを足す）

既存のやり方をいきなり捨てる必要はありません。**今の LINE 運用の外側に API と DB を置き、徐々に橋をかける。** これが **業務を変えずに仕組みを入れる** イメージです。

## 6. プラン（START / GROW / SHIFT）

**想定レンジ** です。要件により前後します。正式見積りはヒアリング後にご提示します。

**価格の考え方：** **¥600,000 程度から** 着手できます。**DB・外部 API 連携** が増えるほど設計・運用の複雑さが上がり、**価格帯も上がる** 構造です。

各プランの違いは **「足し物の一覧」ではなく「何が変わるか」** で捉えてください。

---

### STARTプラン：**まず止血する。** **¥600,000〜**

**変わること：** 「LINE が来るたびに全て人が即レスしなければならない」状態から、**決まった範囲は一次対応が回り、手が止まる時間が減る** 状態へ。

| 項目 | 内容 |
|------|------|
| **内容** | Webhook 経由の AI 応答の基本実装。エスカレーションルール（初版）。会話ログの取得・保存（最小）。運用マニュアル（概要）。 |
| **ベネフィット（経営視点）** | **時間削減：** 同一内容の説明・夜間の初回アウトなど、**担当者の「すぐ返さなきゃ」の件数が減る。** **売上・機会：** 返信遅れによる離脱・温度低下を抑えやすい。**人依存：** 「この人しか返せない」ゾーンを一部 **仕組み側に寄せる** ところから始められる。**小さく作って育てる** 第一歩。 |

---

### GROWプラン：**回る状態にする。** **¥800,000〜**

**変わること：** 「とりあえず動いている」から、**精度・ログ・改善の話が同じテーブルで回る** 状態へ。属人の勘より **見える事実** で手を入れられるようになります。

| 項目 | 内容 |
|------|------|
| **内容** | START の内容に加え、**ナレッジ整備**（社内文書・Q&A の取り込み）、**分析しやすいログ設計**（タグ・要約等）、エスカレーションの **細かめチューニング**、簡易ダッシュボードまたは定期レポート（いずれか）。必要に応じ **軽いデータ連携**（範囲は要定義）。 |
| **ベネフィット（経営視点）** | **時間削減：** 手戻り説明・社内確認の **往復が減りやすい**（精度とナレッジの整理）。**売上・機会：** 関心の高い相談・見込みの **パターンが見え、打ち手を優先できる。** **人依存：** 調整・改善が **特定個人の記憶頼み** から抜け、**誰が見ても同じログから判断** しやすくなる。 |

---

### SHIFTプラン：**仕組みに置き換える。** **¥1,000,000〜**

**変わること：** **「人が頑張る業務」が「手順・データ・連携で回る業務」に置き換わる。** 経営者の目線では、**オペレーションそのものの設計変更** に近い変化です。

| 項目 | 内容 |
|------|------|
| **内容** | GROW の内容に加え、**自社 DB・外部 API 連携**（予約・会員・スプレッドシート等、要相談）、**権限・個人情報** を踏まえた設計、連携エラー時の **監視・通知** の土台。定型業務の **自動化に耐える** 処理フローの整理。 |
| **ベネフィット（経営視点）** | **時間削減：** 転記・二重入力・「誰が最新か」確認などの **裏方コストが落ちやすい。** **売上・機会：** LINE から予約・CRM・フォローまで **一本の流れ** に近づき、**成約に効く動きをデザイン** しやすい。**人依存：** 忙しい人ほど詰まっていたボトルネックを **仕組みに寄せ**、担当交代・増員にも強い。**売る仕組み × 回る仕組み** が一段そろう。 |

---

**進め方：** **START → GROW → SHIFT** の順で、効き目と負担を見ながら階段を上げる前提です。

## 7. 今後の展開（重要）

今回の仕組みは **入口** です。一覧ではなく、**こんな1日が近づくイメージ** で書きます。

**朝——** LINE で「今日まだ空いてますか？」が来る。一次返答と空きの目安が即時に返り、**担当の朝イチがブラウザのコピペ地獄にならない。**  
**昼——** 興味が強い発言はログにタグがつき、**週次で「フォロー優先」が数字で見える。**  
**夕方——** 予約が決まればカレンダーや台帳に **手入力せず反映**（連携の段階で）。引き継ぎは **要約付き** で、**その日抱えたまま持ち越す案件が減る。**

最初から全部は不要です。**Webhook ＋ AI ＋ ログ** まで決めておくと、この **「1日の流れ」の足場** が固まります。**今回が入口である** ことはここで明確にしておきます。

## 8. 協業価値（なぜこの2人で強いか）

**軍司さん単独だと、** 導線・シナリオ・Lステップ運用は最適化できても、**Webhook 先の AI・DB・社内システム連携は別の設計領域** です。「見せ方は完璧だが、裏でデータがつながらず、結局手作業」のギャップが残りやすい。

**tugilo 単独だと、** API や業務設計は進めても、**LINE 上で「売れる流れ」を作る勘所・現場運用** は軍司さんの領域です。導線が弱いまま裏だけ整えても、**問い合わせの量・質自体が伸びにくい** ことがあります。

**2人で組むと、** 同じクライアントについて **「導線」と「回る仕組み」を一枚の設計図で話せる。** 依頼主にとっては **窓口が分断されず、説明の二度手間と責任のなすりつけ合いが減る。** 軍司さんが **売る仕組み**、tugilo が **回る仕組み** を担い、**掛け算で意味が出る** のがこの組み合わせです。

| 役割 | 担うこと |
|------|----------|
| **軍司さん** | Lステップ・LINE の **売れる導線**、シナリオと運用の勘所 |
| **tugilo（次廣）** | AI・API・DB・業務設計による **回る仕組み** |

---

## 9. まとめ

- **チャットボット一点** ではなく、**定義・ログ・将来の拡張まで含めた仕組み** として捉えています。
- **Webhook ＋ AI は実現可能** で、差は **エスカレーションとデータ設計** に出ます。**設計まで含めて伴走する** 前提で組み立てます。
- 進め方は **START で止血し、現場で使いながら GROW で回る状態へ、必要に応じて SHIFT で仕組みへ置き換える** のが現実的です。

**まず START で動かし、運用しながら育てる。** それが当事者にとって無理のない一歩だと考えています。次の打ち合わせで、社内の決裁ラインや優先したい痛みを一緒に線引きできれば幸いです。
