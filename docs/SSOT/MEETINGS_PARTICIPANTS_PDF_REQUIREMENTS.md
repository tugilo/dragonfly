# Meetings 参加者PDF取込 — 要件整理たたき台

**作成日:** 2026-03-17  
**ステータス:** たたき台（実装前の要件整理）  
**関連 Phase:** PHASE_MEETINGS_PARTICIPANTS_PDF_REQUIREMENTS

---

## 1. 機能概要

毎回例会前日に送られてくる「メンバー表PDF」を Meetings に登録し、その回の参加者情報を Religo 上で管理できるようにする。可能な範囲で PDF を解析し参加者候補を抽出し、人の確認・修正を経て既存の `participants` に反映する。

**実現したいこと**

- 例会ごとに「参加者PDF」を Meeting に紐づけて保存できること
- （理想）PDF から参加者候補を自動抽出し、確認・修正のうえで participants に登録できること
- 既存の Meetings 一覧・詳細・メモ・BO・統計の流れを壊さず、参加者情報の入口を追加すること

**前提として置かないこと**

- PDF 解析で 100% 正確に取り込めること
- 完全自動で人的確認なしに登録すること

---

## 2. 背景と目的

### 背景

- 毎回、例会前日に PDF でメンバー表が送られてくる
- 現状はその PDF を人が見て、参加者情報を把握している
- Meetings 画面には一覧・詳細 Drawer・メモ編集・BO 編集・統計カードなどがすでにある
- 今後は「その回に誰が参加するか」も Meeting に紐づけて扱いたい
- 可能なら PDF を解析し、参加者一覧を自動抽出して登録したい

### 目的

- **主目的:** 例会参加者PDFを起点に、その回の参加者情報を登録・管理できるようにするための要件を整理する
- **副目的:** 業務フロー・画面・データ・Phase 分割・リスクを「たたき台」として固め、実装 Phase に進むかどうかの判断材料をそろえる

---

## 3. 想定ユースケース

| ユースケース | 概要 |
|-------------|------|
| UC1: PDF を保存するだけ | 例会前に届いた PDF を「この回の資料」として Meeting に添付し、後から参照・ダウンロードできる |
| UC2: PDF を見ながら手で登録 | PDF は保存しつつ、人が画面で参加者を手入力（または既存 CSV 取込と同様の手順で登録） |
| UC3: PDF から候補を抽出して確認して登録 | PDF をアップロードすると参加者候補が抽出され、一覧で確認・修正したうえで「登録」で participants に反映する |
| UC4: 既存メンバーと照合 | 抽出された名前を既存 `members` と照合し、マッチした場合は member_id を自動で紐づけ、未マッチは要確認として表示する |

---

## 4. 業務フロー案（A / B / C）

### A案: PDF を添付保存だけ

- **流れ:** 例会前に届いた PDF を Meeting にアップロードして保存。参加者登録は人が PDF を見ながら手で行う（既存の CSV 取込や手入力に頼る）。
- **メリット:** 実装が軽い。PDF 形式に依存しない。原本が残る。
- **デメリット:** 参加者登録の手間は減らない。PDF と participants の対応は人が担保する必要がある。

### B案: PDF を解析して参加者候補を抽出 → 人が確認して登録確定（推奨のたたき台）

- **流れ:**  
  1. Meeting に PDF をアップロード  
  2. サーバで PDF からテキスト抽出（＋必要なら区分の推定）  
  3. 抽出結果を「参加者候補」として画面に表示  
  4. ユーザーが候補を確認・追加・削除・区分修正・既存 member とのマッチング確認  
  5. 「登録」で participants（および必要なら members）に反映  
- **メリット:** 手入力の負荷を減らしつつ、最終的な確定は人が行うため誤登録を抑えられる。  
- **デメリット:** 抽出ロジック・画面・データ設計が必要。PDF 形式の揺れにある程度対応する必要がある。

### C案: PDF 形式が安定している前提でかなり自動登録に寄せる

- **流れ:** B案と同様だが、「登録」をほぼ自動で実行し、エラー・未マッチのみ人が修正する。
- **メリット:** 運用が楽になる可能性がある。
- **デメリット:** 形式崩れ・名前の揺れで誤登録が入るリスクが高い。BNI の参加者区分（メンバー・ビジター・代理・ゲスト）の自動判定も難しい。**第一歩としては推奨しない。**

### 比較まとめ

| 観点 | A案 | B案 | C案 |
|------|-----|-----|-----|
| 実装コスト | 小 | 中 | 大 |
| 運用負荷 | 高（手登録） | 中（確認して確定） | 低（ただし誤りリスク大） |
| 誤登録リスク | 人依存 | 低 | 高 |
| PDF 形式への依存 | なし | 中 | 高 |
| 推奨度（第一歩） | あり | **推奨** | 将来検討 |

## 5. 画面要件案

- **Meetings 一覧からの導線**  
  - 行アクションまたは「参加者取込」などで、その回の「参加者PDF登録 / 参加者取込」へ進めるようにする。
- **Meeting 詳細 Drawer 内の導線**  
  - Drawer 内に「参加者PDF登録」または「参加者取込」のボタン／リンクを設け、同じ回の PDF アップロード・取込フローに遷移する。
- **PDF アップロード画面／モーダル**  
  - 対象 Meeting を明示したうえで、PDF ファイルを選択・アップロードする。  
  - 保存のみ（A案）の場合は「保存」で完了。B案の場合は「アップロードして抽出」などで次へ。
- **抽出結果確認画面**  
  - 抽出された参加者候補を一覧表示。行ごとに種別（メンバー／ビジター／ゲスト／代理出席）、名前、既存 member とのマッチ有無などを表示。  
  - 追加・削除・種別変更・名前修正・マッチングの手動指定ができるとよい。
- **登録確定／一括保存**  
  - 「登録」で participants に反映。既存 CSV 取込と同様、member が未登録の場合は members に作成するか、既存 member に紐づけるかを方針に合わせて実装。
- **原本 PDF の表示・ダウンロード**  
  - Meeting 詳細や「参加者取込」画面から、紐づけた PDF を閲覧・ダウンロードできるようにする要否を検討。保存するなら「要」とするのが自然。

## 6. データ要件案

### meeting に紐づく「参加者取込データ」とは

- **参加者PDFの登録単位:** 1 Meeting に対し「1 回の取込」を 1 レコードで管理するか、単に「この Meeting に紐づく PDF が 1 つ」とするか。  
  - 再取込・上書き・履歴を考えると、「取込ジョブ」単位で `meeting_participant_imports` のような中間テーブルを用意し、そこに「原本PDF」「抽出テキスト」「抽出結果（JSON 等）」を紐づける案が扱いやすい。

### 保存すべきもの（案）

| 対象 | 説明 |
|------|------|
| 原本 PDF | バイナリまたはストレージパス。Meeting または import に 1:1 で紐づける。 |
| 抽出テキスト | PDF から取り出した生テキスト（デバッグ・再解析用）。 |
| 抽出結果 | 参加者候補の構造化データ（種別・名前・行番号など）。JSON で保存する想定。 |
| 登録済み参加者 | 既存の `participants` テーブル。取込確定時にここに insert/update。 |

### 既存構造との関係

- **participants:** `meeting_id`, `member_id`, `type`（regular / visitor / guest / proxy）, `introducer_member_id`, `attendant_member_id` はそのまま利用する。  
- **members:** 既存 CSV 取込と同様、未登録の名前は member を新規作成するか、既存 member にマッチさせるかは方針次第。  
- **中間テーブル・import_logs:**  
  - 取込履歴・抽出結果・原本PDF参照を残すなら、`meeting_participant_imports`（または類似名）のようなテーブルを新設する案が現実的。  
  - 列例: id, meeting_id, original_file_path, extracted_text, extracted_result (JSON), status (draft / confirmed / failed), created_at, updated_at。

### 冪等性・再取込

- **同じ PDF を再取込したらどうするか**  
  - 新規「取込」として 1 件追加するか、同じ Meeting の「未確定」取込を 1 件に限定して上書きするかは仕様で決める。  
- **上書き・差分反映**  
  - 「登録」時に、既存 participants を全削除してから今回の結果で作り直すか、差分だけ更新するかは運用と相談。全置換の方がシンプルで、既存 CSV 取込のイメージに近い。

## 7. 実装フェーズ案

いきなり全部やらず、段階的に Phase に分ける。

| Phase | 内容 |
|-------|------|
| **P1** | PDF を Meeting に紐づけて保存できる（アップロード・保存・一覧での表示または Drawer からの参照・ダウンロード）。参加者登録は行わない。 |
| **P2** | PDF から参加者候補を抽出し、画面に表示できる（テキスト抽出＋簡易パース。区分は仮でよい）。 |
| **P3** | 抽出結果を確認・修正し、participants に反映できる（追加・削除・種別変更・登録ボタンで一括反映）。 |
| **P4** | 既存 members と照合して自動マッチングし、候補行に member_id を表示／未マッチを要確認表示できる。 |
| **P5** | 再取込・上書き・差分確認など運用改善（取込履歴の表示、同じ Meeting への再取込時の挙動の整理）。 |

- P1 のみでも「PDF を Meeting に紐づけて残す」価値はある。P2〜P4 で「抽出 → 確認 → 登録」を実現する。

## 8. リスク・論点

| 論点 | 整理 |
|------|------|
| PDF 形式が崩れた場合 | レイアウト変更・表組み崩れで抽出が失敗したり誤る。フォールバックとして「抽出テキストだけ表示し、手で編集」できるようにするか、A案に戻る運用を許容する。 |
| OCR が必要か | テキストが埋め込まれた PDF ならテキスト抽出のみで足りる。スキャン PDF の場合は OCR が必要になるが、第一歩では「テキスト抽出のみ」を前提とする。 |
| 名前の揺れ | 表記ゆれ・同姓同名で 1 人にマッチしない／別の人にマッチするリスク。自動マッチングは「候補」に過ぎず、人の確認を必須とする。 |
| メンバー／ビジター／代理／ゲストの判定 | PDF 上の区分が表組み・見出しで分かれている想定。パターンが安定していれば自動推定可能だが、揺れがあるため B案では「人が確認・修正」を前提とする。 |
| 完全自動化の危険性 | 誤登録がそのまま participants に入ると、BO や紹介者・アテンドの整合が崩れる。人の確認をどこかで必ず入れる方針を推奨。 |
| 人の確認をどこまで残すか | 最低限「登録確定」の 1 回は人がボタンを押す形にし、自動で participants を上書きしないようにする。 |

---

## 9. 推奨方針（現実的な第一歩）

- **第一歩:**  
  - **P1（PDF を Meeting に紐づけて保存）** を最初の実装 Phase とする。  
  - そのうえで、**B案（PDF 解析 → 候補表示 → 人が確認して登録）** を P2〜P4 で段階的に実装することを推奨する。
- **前提にしないこと:**  
  - PDF 解析の 100% 正確性。  
  - 完全自動登録（C案を第一歩にしない）。
- **運用:**  
  - 抽出結果は「候補」とし、確定は人が行う。既存の `dragonfly:import-participants-csv` と同様、members / participants の更新は「登録」操作の結果としてのみ行う。

## 10. 今後の確認事項

- 実際に届く PDF のサンプル（レイアウト・表組み・区分の書き方）を確認し、テキスト抽出・パースの可否と難易度を評価する。
- 原本 PDF をストレージに保存する場合の保存先（local / S3 等）と、権限・保持期間のポリシーを決める。
- 「参加者取込」を Religo 管理画面のどのメニュー・どの画面から開くか（一覧行アクション／Drawer 内ボタン／別タブなど）を確定する。
- 同一 Meeting に複数回 PDF を取込んだ場合の挙動（常に新規取込とするか、未確定を上書きするか、履歴を残すか）を product 側で決める。
- P1 実装時に、既存 Meetings 一覧・Drawer・メモ・BO との導線を壊さないことを DoD に含める。
