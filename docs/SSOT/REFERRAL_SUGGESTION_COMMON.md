## 0. 製品理念（North Star）

> **あるメンバー（主役）について、本人の議事録に加え、章内の他メンバーの 121・定例会議事録など全記録を横断し、「この人に会うべき相手」の紹介候補を出す。**
>
> **Religo の利用者が増え、記録が蓄積されるほど、全員にとって相互のネットワーキング補助が強くなる（コミュニティ・フライホイール）。**
>
> **BNI はリファーラル・マーケットである。Religo はリファーラルを「出す」きっかけを作る — 特に「A さんのネットワークの B さん」を見つけ、A に紹介をお願いし、実績は A が B を自分（依頼者）に紹介したリファーラルになる（§0.8）。**

入口は「いま開いている 121」や「いま見ている定例会」でもよいが、生成の中心は **主役メンバー（subject）** である。主役のニーズ・供給・紹介希望を **本人の接触記録** から把握し、**他者の 121 notes・他回の定例会議事録** からマッチする相手・機会を見つけ、**会う／つなぐ候補**として提案する。

**個人の履歴だけのツールではない。** 各メンバーが Religo に 121・議事録を載せるたび、**同意した範囲で** workspace（章）の関係地図が厚くなり、**他のメンバー主役の提案**にも効く（§0.7）。**全員の記録を無条件に横断しない。**

| 原則 | 説明 |
|------|------|
| **主役（subject）** | 121 なら `target_member_id`（相手）または文脈で明示したメンバー。定例会なら MP 登壇者等（`subject_member_id`）。 |
| **横断コーパス** | **章（workspace）単位**のコーパス。**横断共有に同意した利用者**の 121 等 ＋ 章として扱う定例会議事録（§0.7）。**利用者が増え、同意者が増えるほど**コーパスが増える。 |
| **コミュニティ・フライホイール** | 同意したメンバー A の 121 → B の主役提案に効く。**貢献は任意・受益は主役側の操作**（§0.5・§0.7）。 |
| **横断共有の同意** | **他者の横断マッチングに自分のネットワーキング記録を材料にするか**は利用者ごとに選択。**未同意の記録は他者向け生成に使わない**（§0.7）。 |
| **提案の意味** | ① **会うべきメンバー候補** ② **つなぎ手経由リファーラル**（A のネットワークの B → A に紹介依頼 → **A のリファーラル**）③ 議事録明示の外部紹介。理念の主出力は **①②**（§0.8）。 |
| **きっかけ作り** | Religo は **紹介そのものを自動で成立させない**。提案 → 人が A にお願い → A が紹介 → **`introductions` で A の実績**（§0.8）。 |
| **二経路** | **自分の履歴**由来と **他メンバーのネットワーク**由来でフローが異なる。後者は **必ずつなぎ手仲介**（§0.8.6）。 |
| **Givers Gain** | 与えるのは **つなぎ手 A**。C の B 直接コンタクト・横取りは非目的（§0.8.7）。 |
| **根拠の明示** | 各候補に **どの他者のどの記録**（例: 「A 氏 121#38 の紹介希望」「第208回 MP」）に基づくかを `rationale` に必須。 |
| **記録の正** | 確定した紹介は **`introductions`（SPEC-009）**。提案は候補のみ。 |
| **人の確認必須** | 横断提案でも **自動で introductions は作らない**。 |
| **時間とともに変化** | 章の 121・議事録が増えるほどコーパスが厚くなり、**同じ主役でも再生成で候補が変わる**。 |
| **AI** | BYO API 有効時は主経路。コーパスは要約 Pack ＋必要箇所の抜粋（全文毎回は送らない）。 |

### 0.1 理想のデータフロー（概念）

```text
【主役】メンバー X（例: 121 の相手・MP 登壇者）

  (A) X についての記録
      · 当該 121 notes / 当該・過去定例会 MP
      · X との過去 121 すべて

  (B) 章内の「他者」記録（横断・同意済みのみ）
      · Y,Z,… の 121 notes — **Y/Z が横断共有を ON にした分のみ**
      · 他回定例会議事録（章共有ポリシーに従う — §0.7.3）

  (C) 既知のつなぎ
      · introductions（X が関与する from/to）
      · 既に会った／提案済みの抑制

        ↓ AI / ルール（マッチング）

  【出力】「X に会うべき人」候補
      · 候補メンバー M（member_id）
      · なぜ（Y の 121 に M のニーズがあった、等）
      · owner が 121 または introductions でつなぐ
```

**例（意図のイメージ）:** 主役が動物病院経営。他メンバーの 121 に「ペット保険を探している顧客」が書いてあれば、主役 X とそのメンバーをつなぐ候補を出す——**X の議事録だけを読んでも書かれていないマッチ**を、**他者の記録**から発見する。

### 0.2 記録ソース一覧（主役 Pack ＋ 横断コーパス）

| 区分 | 記録 | DB / 由来 | 役割 |
|------|------|-----------|------|
| 主役 | 当該 121 議事録 | `one_to_ones.notes` | きっかけ・直近合意 |
| 主役 | 当該・過去 121 | 同一 `target_member_id` | X の文脈の蓄積 |
| 主役 | X 関連の定例会 | `meeting_minutes` × `participants` / MP | 紹介希望先・供給 |
| **横断** | **他メンバー（同意済み）の 121** | `one_to_ones.notes`（§0.7 フィルタ後） | **他者のニーズ・紹介希望の源泉** |
| **横断** | **他回・当回の定例会議事録** | `meeting_minutes` | **章全体のニーズ・MP** |
| 抑制 | introductions | owner スコープ | 重複・済みの除外 |
| 補助 | contact_memos、添付、SPEC-013 | 各テーブル | 温度感・プロフィール |
| 参考 | Introduction Hint | Board | 同席ベースの**補完**（議事録横断とは別軸） |

Phase F では **メンバー別ニーズサマリ**（オフライン or 増分更新）を索引にし、全全文を毎回プロンプトに載せない。

### 0.5 コミュニティ・フライホイール（利用者が増えるほど全員に効く）

Religo は **個人の CRM** ではなく、**章が Religo で記録を共有するほど強くなるネットワーキング補助** を目指す。

```text
        メンバー A の 121 記録 ──┐
        メンバー B の 121 記録 ──┼──► 章（workspace）の横断コーパス
        定例会議事録（共有）     ──┘           │
                                              ▼
        メンバー C が「主役」の提案 ◄── マッチング（AI / ルール）
        メンバー D が「主役」の提案 ◄── 同じコーパスを参照
```

| 観点 | 製品既定 |
|------|----------|
| **貢献** | **横断共有 ON** のときのみ、自分の completed 121 等が他者主役の提案の材料になる（§0.7）。OFF なら **自分用の抽出・主役 Pack のみ**で利用可。 |
| **受益** | 主役の提案生成時、**同意済み他者の 121** と章議事録から候補を探す。同意者が少ないほど候補も少ない（トレードオフを UI で説明）。 |
| **スコープ** | **同一 workspace（章）** が基本。他章横断は P2 以降（[WORKSPACE_RESOLUTION_POLICY.md](WORKSPACE_RESOLUTION_POLICY.md)）。 |
| **プライバシー** | **利用者ごとの秘匿選択**が横断の前提（§0.7）。送信・表示は章メンバー向け業務記録の範囲。**根拠に出すのは同意済みソースのみ**（未同意メンバー名を根拠に載せない）。 |
| **操作の正** | 提案の採用・`introductions` 登録は **ログインユーザー（owner）** が行う。他者の代わりに自動登録しない。 |
| **成熟度** | 記録が少ない初期は提案も少ない。**利用普及とセット**で価値が立ち上がる — UI でコーパス充実度を示すことも Phase F 以降で検討。 |

**あなた（owner）個人の履歴だけ**ではなく、**横断共有に同意したメンバーの記録の合算**が効く。同意者が増えるほど、全員の主役提案に効くコーパスが厚くなる。

### 0.7 利用者ごとの秘匿と横断共有の同意

フライホイール（§0.5）と **利用者ごとの秘匿**は両立する。**「他の方に自分のネットワーキング（121 議事録等）を紹介候補の材料として使ってよいか」**を、利用者が選択できることは **Phase F（195）の必須要件**。

#### 0.7.1 設定の意味（2 軸・製品既定）

| 軸 | 利用者の問い | 理念上の既定 |
|----|--------------|--------------|
| **貢献（corpus contribution）** | 自分の 121 等を、**他メンバーが主役のときの横断マッチング**に使ってよいか | **ON/OFF を必ず提供**。OFF 時は他者向け生成のコーパスから **自分が owner の行を除外**。 |
| **受益（corpus consumption）** | 自分が主役のとき、**他者（同意済み）の 121** を材料に含めてよいか | P1 は **貢献軸を必須**とし、受益は **貢献 ON の他者分を読む**で足りる。受益のみ OFF は P2 検討。 |

**121 の相手（target）側:** 相手の 121 行は通常 **相手 owner の貢献設定**に従う。自分との 121 について「相手に書かれた自分の話」が横断に載るかは、**その 121 行の owner（記録者）の貢献設定**で決まる（相手が notes に書いた内容の扱い）。

#### 0.7.2 実装イメージ（Phase 195 — 概念）

| 項目 | 製品既定 |
|------|----------|
| **保存** | workspace 内の **member（または user）単位**の設定。例: `allow_cross_corpus_contribution`（bool）。テーブル名は PLAN で確定。 |
| **デフォルト** | **`allow_cross_corpus_contribution = false`（オプトイン）** — [Phase 195 PLAN](../process/phases/PHASE_195_REFERRAL_SUGGESTION_CROSS_MATCH_PLAN.md) §2 で確定。 |
| **UI** | 設定画面（プロフィール／Religo 設定）＋横断機能初回利用時の説明。**現在の貢献 ON/OFF**と「同意者 N 名分がコーパスに入っている」概要。 |
| **生成パイプライン** | 横断コーパス組み立て時に **貢献 ON の owner_member_id** に限定。AI プロンプト・索引・`rationale` 生成も同フィルタ。 |
| **監査** | run に **同意済み owner 数・除外数**のメタを任意で残す（デバッグ・透明性）。 |

**P2 拡張:** 121 行単位の「この回だけ非公開」フラグ、introductions の横断参照可否の分離。

#### 0.7.3 定例会議事録との扱い

| 種別 | 横断での扱い（検討既定） |
|------|--------------------------|
| **定例会 `meeting_minutes`** | 章の **共有資産**。**貢献設定の対象外** — 直近 N 回は横断コーパスに常時含める（Phase 195 PLAN §2）。 |
| **個人 121 notes** | **貢献設定の主対象**。秘匿ニーズが最も高い。 |

貢献デフォルト・定例会扱いは Phase 195 PLAN §2 で確定。残り Open Questions は PLAN §9。

#### 0.7.4 MVP（190–192）との関係

現行 MVP は **当該議事録のみ**参照のため、**他者への貢献同意は未実装**。利用者の 121 は **owner スコープの管理・提案**に留まり、他者の横断マッチングは行わない。

### 0.8 BNI リファーラル・マーケットとしての「きっかけ作り」（つなぎ手経由）

BNI の本質は **リファーラル・マーケット**（紹介によってビジネスが生まれるネットワーク）である。Religo のリファーラル提案は **議事録の抜き出し**にとどめず、**メンバーが実際にリファーラルを「出す」きっかけ**を増やす。

**非目的（製品の線引き）:** 依頼者 C が **B に直接コンタクトする**ための導線ではない。起点は常に **A さんのリファーラル発生**（C → A に紹介依頼 → A が B を C に紹介）。Religo はその **つなぎ手経由の紹介プロセス**を補助する。

#### 0.8.1 典型フロー（製品が支援する物語）

利用者 **C**（owner・依頼者）が提案を見る。

```text
1. Religo が示す
   「A さんのネットワークにいる B さん（根拠: A の 121 / 定例会 MP 等）が、
    あなた（C）のニーズと合いそうです」

2. C の行動（アプリ外も含む）
   A さんに「B さんを紹介してください」とお願いする
   （121・電話・例会後 — Religo は依頼のメモ・リマインドを支援しうる）

3. A の行動
   A が B を C に紹介する（BNI としてのリファーラル実行）

4. 記録の意味（BNI 実績）
   **A にとって**「B を C に紹介した」リファーラル
   → introductions: from_member_id = A, to_member_id = C
   → 紹介された人物 B は note / suggested_to_label 等で追跡（B が章メンバーなら member_id も可）
```

**ポイント:** C が「B に会いたい」と知ったのは Religo の提案。**リファーラルを「出した」のは A**。C は依頼者・受け手（to）であり、**from を C にしない**のが BNI のつなぎ手尊重に沿う（§6 補足）。

#### 0.8.2 三者の役割（用語）

| 役割 | 例 | 提案・記録での扱い |
|------|-----|-------------------|
| **つなぎ手（connector）** | A | `suggested_from_member_id` = A。紹介実行後の **`introductions.from_member_id`**。 |
| **紹介される側（contact）** | B（A の顧客・知人・または章メンバー） | `suggested_to_label` または将来 `suggested_contact_member_id`。根拠に A の記録を引用。 |
| **依頼者（requester）** | C（提案を見た owner） | 多くの場合 **`introductions.to_member_id`** = C。横断提案の「自分向け」候補の受益者。 |

#### 0.8.3 提案 UI の方向性（Phase F 以降 — 未実装）

| 操作 | 説明 | MVP |
|------|------|-----|
| **紹介をお願いする** | A 向けの依頼メモ・121 下書き・（将来）A への通知。C は **from≠自分** を理解したうえでフォロー | 未実装（195 以降） |
| **紹介登録（register-introduction）** | つなぎ手経由候補では **既定 from=A, to=C**、note に B。C が代理記録する場合も **功績は A** | 現行は owner 中心の手動編集。既定マッピングは Phase F |
| **A が自分で登録** | A の owner で from=A, to=C | SPEC-009 既存画面 |

**禁止:** 提案を見ただけで **C を from にした introductions を自動作成**すること（§0 人の確認必須）。

#### 0.8.6 二つの経路 — 自己履歴 vs 他者ネットワーク（リファーラルが増える発想）

提案の **根拠コーパス** によって、Religo が促すフローが分かれる。**他者ネットワーク由来の候補は、必ずつなぎ手（A）を介した仲介・紹介**に乗せる。そうすると **リファーラルが連鎖的に生まれる**（C↔A の 121・依頼・A の introductions が増え、記録がまた次の提案の材料になる）。

| 経路 | 候補の出どころ | Religo が促すフロー | `introductions` の典型 |
|------|----------------|---------------------|-------------------------|
| **① 自己履歴** | **自分（C）の** 121 notes・自分が関与した定例会・自分の introductions | 議事録に **自分が紹介すると書いた**紹介先の取りこぼし防止。自分が **from** として紹介する流れ（MVP 190–192 の中心） | `from` = C（owner）、`to` = 紹介先メンバー等。議事録の direction に従う |
| **② 他者ネットワーク** | **他メンバー A の** 121・定例会（§0.7 同意済み）に登場する **A のネットワークの B** | **必ず仲介フロー。** C は B に直接いかない。 **C と A をつなぐ（121 等）→ A に B の紹介を依頼 → A が B を C に紹介** | `from` = **A**、`to` = **C**。B は contact |

**② の順序（製品がガイドするステップ）:**

```text
  他者ネットワーク由来の候補（A の B）

    Step 1  つなぐ
            C はまず A と関係を持つ（121 を組む・紹介依頼の文脈を作る）
            ※ Religo: 「A さんと 1 to 1」「紹介依頼メモ」等

    Step 2  依頼
            A に「B さんを紹介してください」

    Step 3  紹介
            A が B を C に紹介（BNI リファーラル）

    Step 4  記録
            introductions（from=A, to=C）＋ 121 / 議事録が蓄積 → 次の提案へ
```

**リファーラルがどんどん生まれる理由（フライホイール）:**

- ② を踏むたび **A のリファーラル実績**が 1 件増える（A にとって価値）。
- C↔A の **121・依頼の記録**が増え、章のコーパスが厚くなる。
- 他のメンバーが横断提案を受けると、また別の **つなぎ手 D** 経由の ② が起きる。
- ① と ② が混在しても、UI は **`corpus_source`**（`self` / `member_network`）で **② には「A 経由で紹介をお願い」だけ**を出し、B への直接アクションを出さない。

**提案メタ（Phase F 案）:** `corpus_source`, `connector_member_id`（A）, `contact_label`（B）を suggestion または run に保持し、② を機械的に判別する。

#### 0.8.7 Givers Gain（ギバーズゲイン）との整合

BNI の理念 **Givers Gain**（与えるものは与えられる）は、**先に与える・関係を築いてから紹介する**文化である。本機能族の設計は **これに反しない** — むしろ **「誰が与えるか」を A（つなぎ手）に残し、C が他者のネットワークを横取りしない**ことで理念に沿う。

| 懸念になりうる点 | Religo の線引き | Givers Gain との関係 |
|------------------|-----------------|----------------------|
| 他者の 121 から「候補を抜く」 | §0.7 **同意済み**コーパスのみ。根拠は明示 | **奪取ではなく、共有された記録から「誰に紹介をお願いすべきか」を見つける」** |
| C が B に直接営業する | §0.8 **非目的**。B への直接導線を出さない | **与えるのは A**。C は「受け取る側（to）」の典型 |
| C だけが得をする | `introductions.from = A`。A のリファーラル実績 | **与えた人（A）に功績が帰る** |
| 関係なく紹介を迫る | ② は **Step 1: C↔A をつなぐ（121）** を先に促す | **関係構築（コアバリュー）→ 依頼 → 紹介**の順 |
| 自分ばかり利用する | 横断共有 ON の人は **他者の提案の材料にもなる**（§0.5） | 与える側（A）・記録を共有する側にも回る **対称** |
| ① 自己履歴は「取る」？ | 自分の議事録で **自分が紹介すると書いた約束**の履行支援 | **自分が与える側（from=C）** の取りこぼし防止 |

**Religo が増やすのは「リファーラルの件数」そのものではなく、BNI 型の与え方のきっかけ**である。

- **② 他者ネットワーク:** C は **A に紹介をお願いする** → 実際に紹介するのは **A（ギバー）**。
- **① 自己履歴:** C は **自分が議事録でコミットした紹介**を実行しやすくする。
- **自動 introductions 禁止:** 人が関係の上で与える — **先にシステムが与えない**（§0 人の確認必須）。

**ギバーズゲインと矛盾する設計（採用しない）:** C を `from` にして A のネットワークの B を横取りする UI、B への直接コンタクトボタン、同意なしの横断スクレイピング、紹介の自動量産。

**運用上の注意（製品外だが SSOT 上の期待）:** Religo は **「紹介をねだる」ツール**ではなく **「誰にどう関係を持ち、誰に紹介をお願いするか」の準備**。121 で価値を先に与えてから依頼するのはメンバーの行動であり、Step 1 がそれを促す。

## 1. 製品上の位置づけ

Religo の **リファーラル提案**は、記録から **リファーラルのきっかけ**（誰に誰の紹介をお願いすべきか — §0.8）を示し、人が確認・依頼・紹介したうえで **`introductions`（SPEC-009）** に接続する機能族である。**入口と run の FK は接触単位**（121 行 / 定例会）を維持しつつ、**生成入力は §0 の関係コンテキストへ拡張**する。

| 入口（UI） | Spec | MVP 入力 | 理念上の入力 |
|------------|------|----------|--------------|
| **121 一覧** | SPEC-015 | 当該 `notes` | 当該 notes ＋ §0.1 の相手関連記録 |
| **定例会** | SPEC-016 | 当回 `body_markdown` | 当回 ＋ 関連回・参加者文脈 |

**正（確定した紹介の SSOT）** は常に `introductions`。提案テーブルは **由来・根拠・採否履歴** を保持する。

## 2. 共通用語

| 用語 | 定義 |
|------|------|
| **リファーラル提案** | AI またはルールが記録文脈から抽出した紹介候補 1 件。`pending` / `accepted` / `dismissed` / `deferred`。 |
| **主役（subject）** | 提案の中心となるメンバー（§0）。121 の相手・MP 登壇者等。 |
| **横断コーパス** | workspace 内の他者 121・定例会議事録の検索対象集合（§0.1 B）。 |
| **関係コンテキスト Pack** | 主役についての記録＋横断コーパス（要約可）のスナップショット。Phase F で `context_digest` 等を run に保存。 |
| **会うべき人候補** | 主役と章内メンバーをつなぐ提案。 |
| **つなぎ手経由リファーラル** | **他者ネットワーク由来**の候補。C↔A をつないでから A に B 紹介を依頼。実績 **A→C**（§0.8.6 ②）。 |
| **自己履歴由来** | **自分の議事録**からの候補。自分が from として紹介する流れ（§0.8.6 ①）。 |
| **corpus_source** | `self` \| `member_network` — 提案が ① か ② か（Phase F）。UI フロー分岐のキー。 |
| **つなぎ手（connector）** | 紹介を実行するメンバー A。`suggested_from_member_id` / `introductions.from_member_id`。 |
| **依頼者（requester）** | 提案を見て A に紹介をお願いする owner C。`introductions.to_member_id` の典型。 |
| **コミュニティ・フライホイール** | **同意済み記録**の蓄積に応じて横断提案品質が向上する効果（§0.5）。 |
| **横断共有の同意** | 他者向けに自分のネットワーキング記録を材料にするかの利用者選択（§0.7）。 |
| **提案実行（run）** | ある時点の入力（MVP: 単一議事録 / 理念: コンテキスト Pack）で提案一覧を**生成した 1 回**。再生成のたびに **新 run**。 |
| **本文指紋（digest）** | MVP: 単一 Markdown の SHA-256（`notes_digest` / `body_digest`）。Phase F: **コンテキスト Pack 全体**の digest を追加検討。 |
| **採用（accept）** | 提案を承認し編集可能に保存。任意で `introductions` 作成＋リンク。 |
| **つなぎ履歴** | 確定 `introductions` ＋ どの run / どの接触から生まれたかのメタデータ。 |

## 3. 共通 UX 原則

| 原則 | 説明 |
|------|------|
| **人の確認必須** | 自動で `introductions` を作らない。 |
| **run 履歴** | 同一接触の議事録が更新されても、過去 run は残す。 |
| **digest 重複抑制** | 同一 digest で再生成ボタン → **新 run を作らず**最新 run を返す（既定）。 |
| **stale 表示** | 最新 run の digest ≠ 現在 DB 本文 digest → 「議事録更新あり・提案未再生成」バッジ / Alert。 |
| **物理削除なし** | run / suggestion 行は監査・振り返りのため削除しない。 |
| **P1 対象** | **外部リファーラル**候補のみ。内部（TYFCB）は P2 以降。 |
| **AI** | ユーザー BYO key（`user_ai_credentials`）。OFF 時は「AI 設定が必要」。 |

## 6. `introductions` への共通マッピング（採用時）

| 提案 | introductions |
|------|----------------|
| `suggested_from_member_id` | `from_member_id` |
| `suggested_to_member_id` | `to_member_id` |
| `owner_member_id` | ログインユーザーの owner（**記録者**。BNI 実績の from/to とは別 — 依頼者 C が代理記録してもよい） |
| `summary` + `rationale` | `note`（結合テンプレは実装 Phase） |
| `suggested_to_label`（contact B） | `note` 内に **紹介された人物**として追記（B が非メンバー時） |
| 接触日 | `introduced_at` 初期値（121 日 / 定例会 `held_on` — 編集可） |
| — | `referral_kind = external`（固定） |

**つなぎ手経由（§0.8）の既定:** `from_member_id` = **A（connector）**, `to_member_id` = **C（requester・依頼者）**。contact **B** はラベルまたはメモ。**C を from にしない**（A がリファーラルを出した実績のため）。

**定例会（SPEC-016）追加:** `meeting_id` を **必ず**セット（どの定例会由来か）。

ソースごとにパス prefix が異なるが、操作は同型:

| 操作 | パターン |
|------|----------|
