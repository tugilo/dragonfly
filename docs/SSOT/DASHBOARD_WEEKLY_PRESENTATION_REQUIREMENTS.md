# Dashboard — ウィークリープレゼン原稿 表示 要件（SSOT）

**Spec ID:** SPEC-004（[SSOT_REGISTRY.md](../02_specifications/SSOT_REGISTRY.md) 登録）  
**ステータス:** active（**DASHBOARD-WEEKLY-P1** で主要実装：表示・API・DB カラム）  
**最終更新:** 2026-04-07 JST（実装反映）  

**目的:** Religo 管理画面の **Dashboard（`/`）** に、BNI チャプターにおける **ウィークリープレゼン（例: 25秒）の原稿** を表示し、例会前に **ログイン直後・ホームで確認・コピー**できるようにするための要件を一本化する。

---

## 0. 背景と製品上の位置づけ

### 0.1 用語

| 用語 | 意味 |
|------|------|
| **ウィークリープレゼン** | チャプター例会などで行う **短時間の自己紹介・事業紹介**（所要時間はチャプター運用に依存。多くはおおよそ **25秒前後**）。 |
| **原稿** | その場で読み上げる **確定文案**（カテゴリー名の名乗り・締めの一言など、名簿・運営ルールと揃えた表現を含む）。 |
| **Owner** | Dashboard の文脈での **操作主体**（[ADMIN_GLOBAL_OWNER_SELECTION.md](ADMIN_GLOBAL_OWNER_SELECTION.md) の `owner_member_id`）。 |

### 0.2 Dashboard との整合

Dashboard の役割は [DASHBOARD_DATA_SSOT.md §0](DASHBOARD_DATA_SSOT.md) に従う。**一覧の代替ではなく**、ログイン直後に **現状把握と次アクション**の入口とする。

本機能は **KPI / Tasks / Activity / Leads** のいずれにも置き換えない。**補助ブロック**として、**「例会直前に見たい定型テキスト」**をホームに載せる。

**優先度の目安:** 情報設計上は **Shortcuts の近傍**、または **左列の折りたたみ可能なカード**（下記 §4）が自然。

## 1. ゴール（ユーザー価値）

1. **確認:** 例会前に別ファイルを探さず、Religo を開いただけで **自分の確定原稿**を読める。
2. **一貫性:** 名簿・カテゴリー表示・チャプター表記と **文言を揃えたい**という運用に対し、**システム上の正**を1か所に寄せられる（少なくとも「表示の正」）。
3. **再利用:** 必要なら **クリップボードにコピー**して、外部ツールに貼れる（実装は Phase で選択）。

---

## 2. スコープ

### 2.1 含む

- Dashboard 上に **ウィークリープレゼン原稿**を表示する UI 領域の **要件**（レイアウトの候補・空状態・権限・データの所在）。
- **誰の原稿を表示するか**のルール（Owner / ログインユーザーと `members` の対応）。
- 既存 SSOT（Dashboard・Owner・データモデル）への **参照関係**。

### 2.2 含まない（本書の段階では固定しない）

- モック HTML のピクセル一致（必要なら別 Phase で [MOCK_UI_VERIFICATION.md](MOCK_UI_VERIFICATION.md) に従い **FIT_AND_GAP** を更新）。
- **編集 UI の有無**（初回は **表示のみ**でもよい。編集は Members 詳細や専用画面に分離する案あり → §7）。
- 音声読み上げ・タイマー・文字数カウントの自動演出（将来拡張）。

## 3. 機能要件

### 3.1 表示対象

- 表示する原稿は **現在の Dashboard Owner に対応するメンバー**のものとする（[ADMIN_GLOBAL_OWNER_SELECTION.md](ADMIN_GLOBAL_OWNER_SELECTION.md) と同じ `owner_member_id` 文脈）。
- Owner が未設定・解決不能のときは **本ブロックを非表示**にするか、**設定を促す短い文言**にする（既存 Dashboard の Owner 未設定パターンと揃える）。

### 3.2 コンテンツの単位

最低限、次を満たすこと。

| 項目 | 要件 |
|------|------|
| **本文** | 複数行のテキスト（改行を保持して表示）。 |
| **バリアント** | **標準稿**と**別稿**など **複数バージョン**を保持したい場合、**切替**または**折りたたみ**で両方見られること（データモデルは §5）。 |
| **メタ情報（任意）** | 最終更新日・文字数目安・参照元（例: 社内ライブドキュメントへのリンク）は **あればよい**。 |

**参考（コンテンツの例）:** オフラインでは `docs/strategy/networking/BNI_Tsugihiro_Atsushi_Intro_Living_Document.md` の **§2 ウィークリープレゼン**のように、標準稿・別稿を分けて管理している。**製品データの正**は DB またはアプリ内設定に置く想定（§5）。

### 3.3 操作

| 操作 | 必須 / 任意 |
|------|-------------|
| 原稿の **閲覧** | 必須 |
| **コピー**（全文または選択範囲） | 任意（Phase1 では「全文コピー」ボタンで十分な場合あり） |
| **編集** | 任意（初回リリースでは不要可） |

### 3.4 権限・プライバシー

- 原稿は **本人（そのメンバーに紐づくユーザー）**および、ポリシー上許される範囲の **管理者**が閲覧できる想定。  
- **他メンバーの原稿を Dashboard で勝手に見せない**（Owner が他者のときの表示は、**その Owner の視点で業務上必要な情報のみ**に限定する既存方針と整合させる）。

## 4. UI / UX 要件

### 4.1 配置

- **Dashboard 専用**（`/admin` 相当のホーム）。他画面への依存は「詳細へ遷移」リンク程度に留める。
- 既存レイアウト（[DASHBOARD_FIT_AND_GAP.md](DASHBOARD_FIT_AND_GAP.md)）を大きく壊さない。**左列**（Tasks / Shortcuts / Activity）の **Shortcuts の下**、または **Activity の上**に **カード 1 枚**が候補。

### 4.2 コンポーネント挙動

- **長文**になりうるため、**デフォルトは折りたたみ**（先頭 N 行＋「すべて表示」）または **スクロール可能な固定高**のいずれかを採用する。
- モバイル幅でも **読める行長・余白**（[ADMIN_UI_THEME_SSOT.md](ADMIN_UI_THEME_SSOT.md) に準拠）。

### 4.3 文言

- カードタイトル例: **ウィークリープレゼン（原稿）** または **25秒プレゼン原稿**（プロダクト内表記は Phase 確定時に統一）。
- 空状態: Owner 未設定時は Dashboard 既存の Owner 案内と **重複しない**よう簡潔に。

## 5. データ・API（実装方針の候補）

実装 Phase 前に **どちらか（またはハイブリッド）**を選ぶ。

| 案 | 内容 | 長所 | 注意 |
|----|------|------|------|
| **A. `members` 拡張** | `weekly_presentation_body`（text）や JSON（標準/別稿）を保持 | 一覧・検索・API と一貫 | マイグレーション・バリデーション |
| **B. 別テーブル** | `member_weekly_scripts` 等に履歴・複数稿 | 更新履歴・監査に強い | 結合・UI 複雑度 |
| **C. 設定 / ファイル** | 初回のみ静的・または workspace 単位テンプレ | 速いたたき台 | ユーザーごとの差分が弱い |

**API:** Dashboard 既存の `GET /api/dashboard/*` に **統合**するか、**`GET /api/members/{id}/weekly-presentation`** のように **読み取り専用**を分けるかは実装時に決定。  
**原則:** 表示は **GET のみ**で足りる。編集を載せる場合は **別エンドポイント**と CSRF・権限チェックを明記する Phase を切る。

## 6. 非機能要件

| 区分 | 要件 |
|------|------|
| **パフォーマンス** | Dashboard 初回ロードの体感を悪化させない（原稿取得は **stats/tasks と並列**可能にする）。 |
| **キャッシュ** | 原稿の更新頻度は低い想定。HTTP キャッシュやクライアントキャッシュは **任意**。 |
| **アクセシビリティ** | コピーボタンに **ラベル**、折りたたみに **適切な aria**（実装時）。 |

---

## 7. 未決事項（要プロダクト判断）

1. **編集場所:** Members プロフィール編集に含めるか、Dashboard から **モーダル編集**するか。
2. **ワークスペース共通テンプレ:** チャプター全体の「型」だけ DB にあり、会員は差分のみ、など **二層構造**にするか。
3. **リッチテキスト:** プレーンテキストのみでよいか（太字・箇条書き）。
4. **公開範囲:** 管理者・メンターが **他メンバーの原稿を閲覧**するユースケースを許可するか（許可するなら権限ロールの定義が必要）。
