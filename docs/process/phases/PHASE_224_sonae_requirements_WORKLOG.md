## 判断ログ

### 1. SSOT として配置

SONAE は将来実装の起点になる要件定義であり、単なる商談メモではなく Religo 拡張モジュールの仕様として扱う必要がある。そのため `docs/SSOT/SONAE_REQUIREMENTS.md` を作成し、`SPEC-017` として `SSOT_REGISTRY.md` に登録する方針にした。

### 2. Religo 連携と単体利用の両立

Religo 利用チャプターと SONAE 単体利用チャプターを別プロダクトとして分けると、通知・回答・発報履歴が二重実装になる。そこで SONAE コアは共通化し、`source_system` と `external_id` で Religo 側の organization / chapter / member 相当と接続する設計にした。

### 3. DragonFly MVP と横展開の線引き

20万円 MVP では、災害通知・回答・集計・訓練に集中する。メール、SMS、自動再送、課金、複雑な権限は後回しにしつつ、将来追加できるよう `member_contacts` や通知対象テーブルの形は拡張可能にした。

### 4. 気象庁連携の取得単位

気象庁情報はチャプターごとに取得せず、システム全体で一度取得して `AlertEvent` に正規化する。その後、各チャプターの対象地域・閾値に照らして発報判定する構造にした。これにより他チャプター追加時も取得ジョブが増殖しない。

### 5. LINE 公式アカウントの分離

LINE 公式アカウントはチャプターごとに用意してもらう前提とし、`line_accounts` を `chapters` に紐付けた。通知は対象チャプターの公式 LINE から送信するため、横展開時もブランドと運用主体を分離できる。

### 6. 訓練の重視

参考資料から、平時の訓練が本番時の回答率に直結する点を重視した。SONAE の MVP では手動訓練発報・訓練履歴・回答率集計を本番発報と同等の主要機能として扱う。
