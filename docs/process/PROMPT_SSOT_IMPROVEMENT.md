# SSOT 改善プロンプト（tugilo / dragonfly 流）

**用途:** `docs/SSOT/*.md` や仕様ドラフトを、**実装フェーズで迷いが出ない粒度**まで引き上げるときに、Cursor または人間レビュアへ渡す **そのまま貼れる指示文**。  
**起点:** 仕様の索引は [SSOT_REGISTRY.md](../02_specifications/SSOT_REGISTRY.md)。本ファイルは **作業手順用メタ文書**（Registry の Spec 行には載せない）。

---

## このプロンプトで得られるもの

| 観点 | 内容 |
|------|------|
| 単なる感想レビューではない | 実装可能レベルまで **本文の追記案** を出させる |
| 誰が使っても同じ軸 | Cursor / 人間レビュー / 設計ドキュメントのたたき台のいずれにも使える |
| tugilo 思想との整合 | **ブレない SSOT**、**UI に依存しないドメイン**、**後から壊れない境界** をレビュー観点に埋め込んである |

---

## 使い方（最短）

1. 下記 **「プロンプト本文」** から **「対象 SSOT」** 直前までをコピーする。
2. 最後に、改善したい SSOT の **ファイルパス** と **該当全文**（または主要セクション）を貼る。
3. （任意）**上級オプション**のブロックを末尾に追記する。
4. AI の出力に基づき SSOT を編集したら、**他 SSOT への波及**を確認し、必要なら [INDEX.md](../INDEX.md) ・ `SSOT_REGISTRY` を更新する。

## プロンプト本文（ここからコピー）

```
あなたはシステム設計レビューの専門家です。
以下の SSOT（Single Source of Truth）ドキュメントを、実装フェーズで迷いが出ないレベルまで改善してください。

このリポジトリは Religo（管理画面は Laravel API + React Admin）。仕様の一覧は docs/02_specifications/SSOT_REGISTRY.md を起点とする。改善提案が別 SSOT と矛盾する場合は、矛盾点を明示し、どちらを正とするか追記案を出すこと。

## 目的

- 実装者（Cursor / エンジニア）が迷わず実装できる状態にする
- UI・ドメイン・API の責務を明確にする
- 将来拡張（認証強化・権限・マルチテナント / 複数 workspace）にも耐えられる表現にする

## レビュー観点

### 1. 不明確・曖昧な表現の排除

- 「適切に」「いい感じに」などの曖昧語をすべて具体化する
- 特に次を必ず具体化する:
  - 未設定時の挙動
  - エラー時の挙動（再試行、表示、ログ、フォールバックの有無）
  - 初期表示の状態（ロード中・空・部分表示の可否）

### 2. マジックナンバー・暗黙仕様の排除

- `1` や `null` fallback、先頭レコード採用など、根拠が本文にない暗黙仕様が残っていないか確認する
- 残っている場合は、次のいずれかで処理する:
  - 仕様として明示的に定義する（条件・優先順位・永続化の有無）
  - 禁止とし、代わりに正規の解決策（例: owner の必須化、設定画面）を SSOT に書く

### 3. 状態管理の明確化

次を定義する（該当しない画面 SSOT では「対象外」と明記）:

- `owner_member_id` 等の **取得元**（どの API・どの SSOT 節が正か）
- フロントの **保持方法**（Context / react-admin store / ローカル state）
- **更新タイミング**（ログイン・オーナー変更・workspace 変更時など）
- **再取得（refetch）のトリガー**（どのユーザー操作・ルート遷移で無効化するか）

### 4. UI とドメインの分離

- UI コンポーネント名や配置に依存した仕様になっていないか確認する
- 「特定のヘッダーやレイアウトが無くても」ドメインルールが成立するか

### 5. 実装指針の具体性

- dataProvider（または同等の API クライアント）の **一覧取得・1件取得・更新・エラー時**の振る舞いが読み取れるか
- API 呼び出しの **条件**（パラメータ必須・スキップ条件）が明確か
- **初期ロード**（並列/直列、ブロッキング範囲）が定義されているか

### 6. 未設定状態の完全定義（重要）

次を SSOT に追加または改善する:

- owner（または必須コンテキスト）未設定時の **UI**
- その状態で **API を呼ぶか／呼ばないか**
- **表示メッセージ**（文言またはメッセージキー方針）
- **ユーザー導線**（設定へ誘導、read-only 表示など）

### 7. 将来拡張との整合性

- 認証・権限が厳しくなっても本文が破綻しないか
- 「誰がどのデータを見られるか」と矛盾する記述がないか

## 出力形式

次の 3 部で出力すること。

### 1. 改善ポイント一覧

各項目: **問題点** / **なぜ問題か** / **改善内容**

### 2. 追記・修正すべき SSOT 本文

- そのままコピペできる **差分形式**（例: 削除線 + 追加文）または **追加セクションの全文**
- 見出しレベルは既存 SSOT に合わせる

### 3. 実装リスク（あれば）

- 実装時にハマりやすいポイント
- 先回りしたい対策（テスト観点・フラグ・ログなど）

## 対象 SSOT

（以下にファイルパスを書き、続けて対象ドキュメントの全文または対象セクションを貼る）
```

---

## 上級オプション（任意・プロンプト末尾に追記）

レビューを一段深くしたいとき、**「対象 SSOT」ブロックの直前**に次を追加する。

```
## 追加指示

- この SSOT のまま実装した場合に起きうる **バグまたは仕様ギャップを 3 つ**想定して指摘する
- それぞれに **修正案**（SSOT のどこをどう直すか）を提示する
```

---

## メモ（運用）

- 出力は **あくまで提案**。採用した改定は **git で追える単位**でコミットする。
- 既存 Phase の REPORT と食い違う場合は、SSOT 側を正とするか Phase を更新するかを人間が決める。
- 新規 SSOT ファイルを増やしたら **SSOT_REGISTRY に Spec ID を足す**（運用ルール）。
- **グローバル Owner** など横断仕様を練るときは、[ADMIN_GLOBAL_OWNER_SELECTION.md](../SSOT/ADMIN_GLOBAL_OWNER_SELECTION.md) を対象に本プロンプトをかけると、実装 Phase のブレを早めに潰せる。
- 同じ SSOT を **より実装粒度まで詰めたい**ときは、専用の [PROMPT_ADMIN_GLOBAL_OWNER_SSOT_IMPROVEMENT.md](PROMPT_ADMIN_GLOBAL_OWNER_SSOT_IMPROVEMENT.md)（観点8・出力4部・Phase 順）を使う。
