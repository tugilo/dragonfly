## 目的

この文書は、tugilo が今後 BNI をどう使うかを整理する **運用方針** である。

既存の Living Document は、25秒プレゼン・BO・121・事例・思想・実験ログを集約する「素材集」として使う。一方で本書は、次の判断に使う。

- BNIで何を伝えるか
- 紹介後にどの提供レーンへ流すか
- 小中さんとカテゴリが被らないようにどう棲み分けるか
- 受託開発・診断・PoC・クラウドパッケージをどう位置づけるか

## 結論

BNI は単なる紹介獲得の場ではなく、tugilo にとって **困りごとの言葉・入口商品・商品導線を検証するPDCAラボ** として使う。

ただし、BNI上で前面に出すのは **AIや製品名ではなく症状** にする。

> 「AIを使えます」ではなく、
> 「Excel限界」「二重入力」「属人化」「改善が後回し」と言っている人を紹介してもらう。

紹介後は、相手の困りごとに応じて次の2レーンへ振り分ける。

| レーン | 位置づけ | 主な入口 |
|--------|----------|----------|
| **Track C: Custom** | 診断 → 1本化PoC → 伴走開発 | Excel限界・二重入力・属人化・複数工程 |
| **Track P: Product** | 月額クラウド・定型パッケージ | 予約・受付・日報など単一症状 |

Track C は tugilo の本業の芯として維持する。Track P は、BNIで紹介しやすいフロント商品として検討するが、**小中さんのAI情報発信カテゴリと被らないこと** を最優先条件にする。

## DragonFlyパワーチーム構想

tugiloがBNIを活用する目的は、紹介を受けることだけではない。

もう1つの重要な目的は、DragonFlyメンバー同士で強みを組み合わせ、**1人では受けきれない案件をチームで回せる状態** を作ること。

### 考え方

従来:

```text
案件
  ↓
tugilo
  ↓
個人対応
```

今後:

```text
案件
  ↓
tugilo
  ↓
最適メンバー選定
  ↓
チーム対応
```

### 理想状態

- tugilo = 全てをやる人ではない
- tugilo = 課題整理・設計・接続役

### 想定例

| 課題 | 関わる人 |
|------|----------|
| LP + 予約導線 + LINE | Webデザイナー + tugilo |
| AI教育 + 業務改善 | 小中さん + tugilo |
| 採用 + 業務フロー改善 | 採用支援 + tugilo |
| 現場運用 + システム | 現場専門家 + tugilo |

### BNIで確認すること

121では、紹介候補だけでなく **一緒に案件を回せる可能性** も確認する。

- どんな案件なら一緒に組めるか
- 得意領域
- 苦手領域
- 理想案件
- 紹介したい業種

### 目指す姿

BNIを単なる **営業ネットワーク** ではなく、**案件が回る実行ネットワーク** として育てる。

## パワーチーム候補（育成中）

今後の121・紹介・共同案件で育てる候補者を管理する。

| メンバー | 強み | 想定連携 | 状態 |
|-----------|------|----------|------|
| 小中さん | AI教育・情報発信 | AI教育 + 現場改善 | 構築中 |
| 深澤さん | LP・導線設計 | LP + 予約 + システム | 構築中 |
| （空欄） | 採用支援 | 採用 + 業務改善 | 候補 |
| （空欄） | 保険 | 福利厚生 + 経営改善 | 候補 |

## 今月の重点パワーチーム候補

毎月3〜5人だけ設定する。

全員を同時に深めようとしない。

| メンバー | 今月目的 | 次アクション | 状態 |
|-----------|----------|--------------|------|
| 小中さん | AI + 現場改善導線整理 | 121実施 | 構築中 |
| 深澤さん | LP + 導線連携 | 共同事例相談 | 構築中 |
| ○○さん | 採用連携確認 | BO実施 | 候補 |

目的は人数ではなく、**今月どこに時間を使うか** を決めること。

パワーチームは人数を増やすことが目的ではない。

重要なのは、次の条件が揃うこと。

- 相互紹介が起きる
- 得意／不得意が明確
- 案件を安心して任せられる
- 価値観が近い

### パワーチーム成熟度

| 状態 | 定義 |
|------|------|
| 候補 | 初回121前後 |
| 構築中 | 得意領域把握済み |
| 実験中 | 相互紹介・小案件実施 |
| 稼働中 | 実案件実施済み |
| コア | 継続的に案件が回る |

目的は人数ではなく、信頼ベースで徐々に育てること。

## BNIでの基本原則

### 1. 症状で覚えてもらう

25秒・BO・121では、カテゴリ名や機能説明ではなく、紹介者が日常会話で拾える言葉を使う。

| 症状 | メンバーに覚えてもらう言葉 |
|------|----------------------------|
| Excel限界 | 「Excelでやってるけど、正直もう限界なんだよね」 |
| 二重入力 | 「見積も現場も請求も、また同じことを入れている」 |
| 属人化 | 「この業務、あの人じゃないと分からない」 |
| 改善後回し | 「売上は伸びているのに、改善が後回し」 |
| 予約混在 | 「電話・LINE・紙の予約が混ざって取りこぼしている」 |

### 2. AIを主語にしない

AIは tugilo の差別化要素ではあるが、BNI上では主語にしない。

| 避ける言い方 | 使う言い方 |
|--------------|------------|
| AIで何でも効率化します | 業務の流れを整理して、1回の入力で回る状態にします |
| AIツールの使い方を教えます | 現場の流れを見て、仕組みとして直します |
| AI予約システムです | 予約・受付の取りこぼしを減らす仕組みです |

### 3. BNIでは売り込まず、入口を統一する

BNIでの入口は、商品名ではなく **困りごと** に統一する。

```text
症状を聞く
  ↓
90分診断 or Track P
  ↓
1本化PoC
  ↓
伴走開発
```

## 小中さんとのカテゴリ棲み分け

小中さんのカテゴリは **使って学ぶAIツールの情報発信**。tugilo は **AI業務改善システム構築**。

同じIT・同じSE・同じAI周辺に見えるため、BNI上では明確に役割を分ける。

| 軸 | 小中さん | tugilo |
|----|----------|--------|
| 覚えてもらう言葉 | AIを知る・学ぶ | 現場が回る・時間が生まれる |
| 主な価値 | AI情報・ツール活用・研修 | 業務分解・仕組み化・伴走開発 |
| 入口 | 月額情報発信 | 症状型リクエスト・診断・PoC |
| AIの位置づけ | 主役 | 手段・部品 |
| 紹介されたい人 | AIツールを学びたい人 | Excel・転記・属人化で困っている会社 |

### 振り分けルール

| 相手の困りごと | 紹介先 |
|----------------|--------|
| ChatGPTやClaudeの使い方を知りたい | 小中さん |
| AIツール研修を受けたい | 小中さん |
| Excel・手入力・属人化で業務が回らない | tugilo |
| 予約・受付・日報の運用が散らばっている | tugilo（必要に応じて軍司さん・小中さんと連携） |
| AIツール選定と現場システムの両方が必要 | 小中さん × tugilo の共同参画 |

### tugilo側のNG

- 月額AI情報発信を始めない
- AI研修・AIツール教室を単独商品にしない
- Track P を「AI SaaS」として売らない
- 25秒で「AIで業務改善」だけを言わない
- ThreeBiz内でAI最新動向の発信役を取りに行かない

## 提供レーン

### Track C: Custom

tugilo の中心。個別診断から入り、1箇所だけをPoCで直し、必要に応じて伴走開発へ進む。

| 段階 | 内容 | 役割 |
|------|------|------|
| 90分診断 | 入口→記録→確認→締めを整理 | どこが壊れているかを見つける |
| 1本化PoC | 2〜4週間で1箇所だけ繋ぐ | 結果を見せる |
| 伴走開発 | PoCから横展開 | 本商品 |

向いている紹介:

- 建設・製造・FC・多店舗・受発注がある会社
- 売上は伸びているが現場が追いついていない会社
- 士業・コンサルの顧問先で、数字以前の現場情報が整っていない会社

### Track P: Product

BNIで紹介しやすい月額クラウド・定型パッケージ候補。小中さんの「月額情報発信」と同じく、低い入口・継続接触の役割を持つ。ただし中身は情報ではなく **業務の仕組み** に限定する。

候補:

| 候補 | 向く相手 | 注意 |
|------|----------|------|
| 予約管理 | 個人サロン・クリニック・予約型サービス | 「AI予約」と言わない。予約・受付の一本化として売る |
| LINE現場日報 | 建設・現場系 | DragonFly内の現場系に刺さりやすい |
| 受付〜案件管理 | 解体・修理・問い合わせ型事業 | Track Cへの橋にしやすい |

Track P は、いきなり複数作らない。まず1製品に絞り、BNIで紹介しやすいか、サポート負荷が重すぎないかを検証する。

## 週次運用

BNI参加週は、次のサイクルを回す。

1. 今週試す症状を1つ決める
2. 25秒で同じ症状を話す
3. BO・121でも同じ言葉を使う
4. 相手の反応を1行メモする
5. Living Document のローテ表・ブラッシュアップメモへ反映する

反応の判断基準:

| 反応 | 判断 |
|------|------|
| 「うちもそう」と言われる | 継続 |
| 具体的な業務名が出る | 継続 |
| AIツールの話に流れる | 小中さん領域に寄っている。言い換える |
| 「へえ」で終わる | 症状か相手属性を見直す |

## BNI週間運用テンプレート

### 月曜（定例会前）

決める項目:

- 今週試す症状
- 今週強化する業種
- 話したい人
- 121候補
- 今週のパワーチーム候補

### 定例会当日

確認:

- 25秒で症状を使ったか
- BOで同じ言葉を使ったか
- 気になった人
- 新ビジター
- 一緒に組めそうな人

### 定例会後

記録:

- 刺さった言葉
- 相手の反応
- 紹介可能性
- パワーチーム候補更新
- Living Document更新

## 121で確認すること

BNIメンバーとの121では、自分の説明だけで終わらせず、次を確認する。

- 「私を紹介しやすくするために、足りない情報は何ですか？」
- 「顧問先や知人で、Excel・予約・日報・二重入力で困っている人はいますか？」
- 「AIツールの相談なら小中さん、現場の仕組み化なら次廣、という分け方で伝わりますか？」
- 「予約管理のような月額商品があると紹介しやすいですか？ それとも診断からの方が安心ですか？」
- 「どんな案件なら一緒に組めますか？」
- 「得意領域・苦手領域・理想案件は何ですか？」
- 「一緒に紹介したい業種や、チームで提案できそうな相手はいますか？」

### パワーチーム候補判断項目

121後に以下をメモする。

- 紹介しやすさ
- 一緒に仕事したいか
- 得意分野
- 苦手分野
- 信頼感
- 相性
- 将来的な共同案件可能性

## BNI到達イメージ（半年〜1年）

### 半年後

- 症状型25秒が定着
- 反応ログが蓄積
- パワーチーム候補5〜10名
- 相互紹介が自然発生
- 小規模共同案件実施

### 1年後

- DragonFly内で継続案件が回る
- tugilo単独ではなくチーム提案が増える
- 「困ったら次廣」と認識される
- 「AIの人」ではなく「仕組み化する人」と認識される
- BNI経由の案件導線が安定

### 判断基準

良い状態:

- 自然に紹介が発生する
- 一緒にやる人が増える
- 案件が継続する
- 自分が全部やっていない

悪い状態:

- 自分だけで抱えている
- AI説明ばかりしている
- 紹介が単発
- 毎回ゼロから説明している
